在那些无法通过进一步降低延迟来实现目标的系统中，异步处理是一种提升系统并发性和隐藏延迟的强大技术。通过以异步方式执行 I/O 操作并推迟非关键工作，你可以显著提升用户感知的响应能力，因为终端用户不会感知到延迟。虽然分区、缓存和其他延迟优化技术可以降低绝对延迟，但异步处理提供了一种互补策略——它通过允许系统即便在 [某些操作需要耗时](https://thenewstack.io/hpe-agentic-ai-ops-burnout/) 的情况下依然保持响应来隐藏延迟。

## 异步与同步处理

*异步处理* 使任务能够独立执行并在重叠的时间段内运行，这与同步处理不同，同步处理中的任务是按顺序运行的。换句话说，在同步系统中，一个任务必须执行完毕，下一个任务才能开始。例如，假设一个单线程同步服务器正在读取和处理消息。它首先从套接字读取消息，这可能需要线程阻塞直到消息完全到达。然后服务器处理该消息，最后在开始处理下一条消息之前发送响应。如果请求处理耗时较长或发生阻塞，系统就会同步等待。

异步处理消除了这一限制，允许多个任务同时进行。例如，使用异步处理的服务器可以通过使用 I/O 多路复用（一种操作系统 (OS) 接口）来轮询各种连接的状态，从而并发处理多个独立请求。服务器随后可以对事件（如套接字变为可读或可写）做出反应，以处理请求。同样，异步服务器可以发起网络响应发送，然后无需等待响应即可处理其他任务。

异步处理类似于 [并发编程](https://thenewstack.io/python-threadpool-vs-multiprocessing/)。然而，异步处理与并发编程的区别在于它具有显式接口。例如，使用线程的并发编程允许服务器在处理请求时保持同步，同时通过线程间的上下文切换保留并发性。服务器执行 `send()` 和 `recv()` 系统调用，如果套接字没有数据可读或套接字不可写，这些调用会阻塞。当服务器阻塞时，操作系统会切换到另一个线程进行并发执行。相比之下，在异步处理中，服务器使用 I/O 多路复用接口轮询套接字状态。I/O 多路复用接口告知服务器哪些套接字可读，服务器可以在不阻塞线程的情况下从这些套接字读取数据。同样，当服务器发送响应时，它使用异步接口发送响应，但随后可以立即继续工作而不必阻塞，让操作系统在后台发送响应。

图 1 展示了同步和异步处理之间的区别（这些区别也总结在题为“异步、并发和并行处理的区别”的侧边栏中）。在此示例中，我们有两个任务 A 和 B，必须全部运行才能完成工作。假设后端系统需要与外部系统 A 和 B 通信以完成其收到的请求。在同步处理中，每个任务必须在下一个任务开始前完成。我们运行任务 A 直到它完成（包括它提交的 I/O），然后运行任务 B。所需总时间是所有任务时间之和。如果后端服务需要执行所有这些任务，用户必须等待这段总时间才能得到响应。

![一个比较同步和异步处理的时间轴图。](https://cdn.thenewstack.io/media/2026/07/88dc0a53-image1.png)

**图 1** 同步处理与异步处理。同步处理（顶部）从一个任务顺序执行到下一个任务。在此示例中，我们有任务 A 和 B，其中 A 提交了 I/O。I/O 同步执行，然后任务 B 才能执行。相反，异步处理（底部）可以与任务 B 并行执行 I/O。也就是说，任务 A 运行，提交 I/O，并立即开始执行任务 B。当任务 A 和任务 B 的 I/O 完成时，我们就结束了，完成工作的速度比同步处理更快。

然而，在异步处理中，我们可以与任务 B 同时执行任务 A 的 I/O，并且当两者都完成时，工作即结束。如果 I/O 同时运行，用户的等待时间会短得多，即使每个任务本身耗时相同。这在后端服务调用数据库或其他服务时效果很好，因为这些任务可以独立运行而不阻塞彼此。但有一个问题：如果你的 I/O 无法并行运行，使用异步处理将无济于事，反而可能使运行变慢，因为管理异步任务增加了系统的额外负担。

| **异步、并发和并行处理的区别** *并发处理* 意味着通过在同一计算单元上进行多路复用同时执行任务，而 *并行处理* 意味着在不同的计算单元上同时执行任务。虽然这种区别看起来很微妙，但要点在于并发处理是关于以一种有利于执行多项任务的方式来构建应用程序，尽管它们可能是在同一个计算单元上按顺序运行的。另一方面，*并行处理* 是关于在不同的计算单元上执行不同的任务，从而减少执行时间。虽然异步处理与并发和并行处理相关，但它本质上是关于构建你的代码来处理那些可能需要时间完成的任务。换句话说，异步处理可以实现并发和并行，但它不能保证两者。例如，你可能编写了通过在任务之间切换而在单个 CPU 核心上并发运行的异步代码，或者你可能拥有在多个核心上并行运行的异步代码。 |

异步处理也是 *隐藏* 延迟的关键技术。尽管你竭尽全力减少延迟，但某些操作仍需要很长时间才能完成，因此在不让所有人等待其完成的情况下执行操作至关重要。例如，后端系统通常与第三方服务、数据库服务器和消息队列等外部系统交互，每次交互都会增加一些延迟。在同步处理中，你构建的系统往往无法利用现有的固有并行性，并且会导致闲置时间，即你在等待系统完成工作。相反，异步处理允许你通过异步启动操作并在其完成时做出反应来最小化等待时间。

在同步处理中，你将代码构建为一系列相互依赖的操作。例如，同步服务器的请求处理函数可能如下所示。

### 代码清单 1 同步系统的简单示例

```

fn process_requests(socket: &amp;Socket) {
  loop {
    process_request(socket);
  }
}

fn process_request(socket: &amp;Socket) {
  let msg = socket.recv();
  let request = parse_message(msg);
  let resp = match request {
    Request::GetUserInfo(id) => get_user_info(id);    
  };
  let resp = format_response(resp);
  socket.send(resp);
}

```

在高层面上，我们有 `process_requests` 函数，它处理来自套接字的任何传入请求。在 `process_request` 函数中，每个步骤在开始下一个步骤之前都必须运行完成。我们从套接字读取消息，解析消息以确定请求是什么，处理请求，最后通过套接字发送响应。更重要的是，在发送响应之前，我们不会开始另一个 `process_request`，我们也不允许来自多个套接字的请求被处理。

虽然协程和 Future 等并发原语可以实现并行执行，但它们不足以实现高效的异步处理，特别是在处理 I/O 时。如果服务器处理数千个并发连接，你必须以不同的方式构建应用程序。事件循环是跨多个连接高效进行 I/O 多路复用的基础。

## 事件循环

事件循环是所有输入和输出操作的中央协调器——它是异步系统的核心。传统的同步程序一次处理一个连接——就像一个工人按顺序处理任务——而事件循环作为调度程序，同时管理数千个 I/O 操作。这种架构模式（有时称为 I/O 循环或 I/O 调度程序）是异步处理高效处理并发操作的方式。事件循环不是为每个连接分配单独的资源，而是通过跟踪各种 I/O 源（网络连接、文件操作、计时器等）的状态并在它们准备好时进行处理，从而对它们进行多路复用。

> “事件循环是所有输入和输出操作的中央协调器——它是异步系统的核心。”

事件循环遵循一种简单而强大的模式：

1. 轮询事件。
2. 处理事件。
3. 运行计划任务。
4. 重复。

事件循环通过使用操作系统特定的 I/O 多路复用接口来轮询事件，例如 Linux 上的 `io_uring` 和 `epoll`、macOS 上的 `kqueue` 以及 Windows 上的 IOCP，以处理传入的数据、过期的计时器或 I/O 完成等事件。这些接口允许你注册对事件源的关注，并在事件发生时获得通知。例如，应用程序不是直接从套接字读取数据，而是表达对套接字变为可读状态的兴趣。当数据从网络到达套接字时，操作系统通过 I/O 多路复用接口通知应用程序该套接字现在可读。事件循环通过轮询发现这一点，并调用应用程序的事件处理逻辑来处理从套接字新到达的数据。

让我们在 Rust 中实现一个基本的事件循环，以更好地理解其结构：

```

struct EventLoop {
    // 保存已注册的事件源，如套接字、文件、计时器
    sources: Vec&lt;EventSource>,    
}

impl EventLoop {
    fn run(&amp;mut self) {
        loop {
            // 创建一个新集合来存储事件
            let mut events = Events::new();

            // 使用超时轮询新事件
            self.poll(&amp;mut events, Duration::from_millis(100));

            // 处理找到的每个事件
            for event in events.iter() {
                self.process_event(&amp;event);
            }

            // 运行任何计划的任务
            self.run_scheduled_tasks();
        }
    }
}

```

`EventLoop::run()` 方法演示了事件驱动编程的核心功能：持续轮询和处理事件。`poll()` 方法对事件源上的事件使用操作系统特定的 I/O 多路复用接口（如 `io_uring`）。如示例代码所示，我们还为事件轮询指定了超时。需要超时是因为事件循环中的 I/O 轮询通常是唯一会阻塞线程直到事件发生的同步代码。如果系统处于空闲状态且没有发生事件，轮询可能会阻塞，这种阻塞可以减少在没有任务可做时浪费的 CPU 周期。然而，为了确保事件循环不会永远阻塞，超时确保了我们能从 `poll()` 返回。这允许事件循环执行非条件于事件的工作，例如执行后台工作。但在某些情况下，你可能会使用忙轮询来避免某些对延迟敏感的事件循环的睡眠/唤醒周期延迟。

`process_event` 函数负责处理轮询期间发现的任何事件。例如，如果应用程序注册了对来自网络的数据到达的兴趣（如套接字变为可读），`process_event` 函数会从套接字读取数据并转发数据供应用程序处理。一个简单的 `process_event` 函数可能如下所示：

```

struct EventLoop {
    // 保存已注册的事件源，如套接字、文件、计时器
    sources: Vec&lt;EventSource>,    
}

impl EventLoop {
    fn run(&amp;mut self) {
        loop {
            // 创建一个新集合来存储事件
            let mut events = Events::new();

            // 使用超时轮询新事件
            self.poll(&amp;mut events, Duration::from_millis(100));

            // 处理找到的每个事件
            for event in events.iter() {
                self.process_event(&amp;event);
            }

            // 运行任何计划的任务
            self.run_scheduled_tasks();
        }
    }
}

```

正如你所见，每个事件都由一个 `Event` 枚举表示，该枚举具有不同事件的变体。事件处理逻辑特定于事件循环的结构。例如，如果事件循环使用回调进行事件处理，它会调用它们，将工作委托给应用程序。然后应用程序可以在回调中执行工作或将工作提交给另一个线程进行处理。

2. 处理请求
3. 发送响应

**图 2** 事件循环将工作分解为在事件发生时执行的单个任务。在此示例中，事件循环处理三个不同的任务——接受连接、处理请求和发送响应——作为处理来自网络的请求的一部分。当发生事件（例如套接字变为可读）时，每个任务都会运行。

当 I/O 多路复用器通知事件循环有传入连接时，第一个任务运行。应用程序通过接受连接并注册对已接受套接字何时变得可读的兴趣来对该事件做出反应。当数据从网络到达时，操作系统通知事件循环该套接字是可读的。应用程序通过读取套接字并处理传入的请求来对此做出反应。最后，应用程序注册对套接字变为可写状态的兴趣。当操作系统有足够的缓冲区内存用于传出响应时，它会通知应用程序，应用程序将响应写入套接字。

> “虽然事件循环是异步处理的底层基础设施，但你还需要一些并发原语来指定单个任务之间的依赖关系。”

如果你将事件循环与清单 1 中的同步服务器进行对比，你会看到这些方法之间的两个主要区别：

* *非阻塞操作*——事件循环不会阻塞线程，而是注册对事件（如套接字变为可读）的兴趣，并推迟从套接字读取，直到该条件为真，同时处理其他事件。
* *资源效率*——运行事件循环的单个线程可以处理数千个并发连接，因为它不需要等待 I/O 操作完成。相反，I/O 多路复用操作系统接口允许事件循环同时轮询多个事件源（如套接字）的状态，从而执行基于事件的处理。

虽然事件循环是异步处理的底层基础设施，但你还需要一些并发原语（如回调或 Future）来指定单个任务之间的依赖关系。

## 挑战

虽然异步处理可以显著提高应用程序性能，但它也存在一些需要考虑的重要陷阱：

* *复杂性*——异步代码通常比同步代码更复杂。你需要仔细 [管理任务依赖关系](https://thenewstack.io/unlocking-the-power-of-automatic-dependency-management/)、处理多个操作中的错误并处理竞争条件。
* *资源管理*——同时运行许多任务会消耗大量的内存和系统资源。你需要实施适当的限流和资源管理。
* *可调试性*——当异步代码出现问题时，更难追踪问题，因为执行顺序并不总是显而易见的，并且堆栈跟踪可能无法说明全部情况。
* *错误处理*——由于多个操作独立运行，错误处理变得更加复杂。你需要决定一个任务中的失败应如何影响其他正在运行的任务。
