本文强调代码本质上是一种跨越时间的沟通。开发者应将代码和Pull Request视为叙事，通过原子化提交、清晰的commit信息和意图表达，为未来的维护者和协作队友提供关键上下文，在AI编程时代，这种结构化的沟通尤为重要。

> 译自：[Code is a message to the future](https://thenewstack.io/code-message-to-future/)
> 
> 作者：Hernan Garchtrom

工程师无时无刻不在交流。Slack消息、设计文档、RFC线程、代码评审意见：这份工作很大程度上在于与他人分享意图，正如它在于解决技术问题一样。

但有一个沟通渠道并没有总被这样对待：代码本身。围绕它的所有一切都必须成为该渠道的一部分。

代码不仅是一组给机器的指令。它是发给下一位必须阅读、扩展或调试它的工程师的消息。它是发给评审中队友的消息。它是发给六个月后的你自己——那时所有原始背景信息都已消失——的消息。一条提交信息（commit message）就像一条必须独立存在的Slack消息。没有线程，无法追问。通常在撰写多年后才会被阅读。

> “（代码）是发给六个月后的你自己——那时所有原始背景信息都已消失——的消息。”

一旦你将*代码视为沟通*，你应该问自己的问题就会改变。这个提交（commit）本身是否易于理解？有人能按顺序评审这个PR吗？这段注释是在解释代码存在的原因，还是仅仅重复了代码已经表达的内容？

你本地的工作流是你自己的事。尽情实验、重写、丢弃。混乱是发现好点子的一部分。但一旦准备合并，规则就变了。那时，你产出的不仅仅是一个差异（diff）。它成为了代码库的一部分。以后会有人用它来理解为什么一个文件被修改、bug是什么时候引入的，或者这个PR试图解决什么问题。那是放慢脚步并自问的时刻：这讲述了一个其他人可以跟随的故事吗？

## PR 是一个故事

这里有一个有用的思维模型：把Pull Request（PR）想象成一本书。每个提交就是一章。每个提交内的差异就是散文。没有人会通过随机跳到某页来读书。你是按顺序阅读的，故事之所以通顺，是因为作者对顺序有明确的意图。一个按顺序阅读你提交的评审员，应该能够跟随你的逻辑推理而无需猜测，也不需要同时在大脑中记住整个差异来理解其中任何一部分。

> “没有人会通过随机跳到某页来读书。你是按顺序阅读的，故事之所以通顺，是因为作者是有意为之。”

以下是实践中的样子：

```

Add search endpoint to API
Add basic relevance ranking
Extract ranking logic into standalone module
Add unit tests for ranking
Add integration tests for search endpoint

```

按顺序阅读那五个提交，你就已经知道发生了什么。该功能是逐步构建的，排序逻辑被提取到了独立的模块中，并且行为在单元和集成测试层面都得到了覆盖。无需PR描述来重构。提交本身就讲述了这一切。

要使这一点奏效，每个提交都需要分量十足。重命名加行为修改不是一件事。那是两个独立的修改，通常应该作为两个独立的提交。这些章节的标题与章节本身一样重要。提交信息需要独立承载上下文。`Fix ABC-123` 只指向别处的上下文，而[该上下文](https://thenewstack.io/context-is-ai-codings-real-bottleneck-in-2026/)在别人需要时可能已经不可用了。

短小、以动词开头的消息效果更好：

* Extract validator
* Refactor: use validator in form
* Fix: [handle edge case](https://thenewstack.io/handling-edge-cases-and-exceptions-in-python/) on submit

按顺序阅读它们，你就已经知道这个故事了。

一旦评审开始，在对话活跃时避免重写提交。改为添加新的提交。评审员将他们的评论锚定在特定的行上。如果你强制推送（force-push）并重写历史，那些锚点就会失效，对话也将变得难以追踪。原始故事保持不变，反馈在之上得到解决，任何后来阅读的人都能准确看到PR是如何演变的。

## 对面的读者

当以沟通为核心编写代码时，评审员的视角会发生转变。他们不再试图从一系列修改中重构意图，而是在跟随叙事。这增加了作者的工作量，但减少了其他所有人的摩擦，特别是那些没有参与探索过程的评审员。在团队环境中，这种权衡是值得的。一个清晰的PR更容易评审。随着时间的推移，这很重要。花在重构意图上的时间越少，花在实际修改上的时间就越多。

评审员不是唯一的读者。有人可能会在几年后回到代码库，想知道某一行为什么存在。良好的提交历史为他们提供了一条回到上下文的路径：引入它的提交、围绕它的PR，以及背后的权衡。没有这条线索，很容易“修复”一些本来就是有意为之的东西。

> “代码已经展示了正在发生什么。它通常无法展示的是为什么要存在，以及它在解决什么限制。”

同样的直觉也体现在代码注释中。代码已经展示了正在发生什么。它通常无法展示的是它为什么要存在，它在解决什么限制，或者它编码了什么假设。一条写着 `// increments the counter` 的注释毫无价值。一条写着 `// must run before the subscription is set up, or the callback fires before state is ready` 的注释则是承重型的文档。这种文档能节省数小时的调试时间，并可能从一开始就防止bug的引入。

## AI生成的代码使这一点更重要，而非更少

AI代理可以快速生成大量代码，但速度无法取代上下文。[清理成本比大多数团队预期的更远离速度优势。](https://webflow.com/blog/cleanup-cost-ai-generated-code) 如果任其发展，结果将是一个巨大的提交，消息写着“实现功能”，却没有任何关于其背后推理的痕迹。

要求AI代理遵循与人类工程师相同的规范会改变这一点。原子化提交使代理的推理过程可读：你可以看到它先构建了什么，在哪里[重构了代码](https://thenewstack.io/whats-missing-with-ai-generated-code-refactoring/)，它测试了什么，以及执行这些操作的顺序。这不仅对理解代码有用，也是你在代理犯错进入生产环境之前捕捉其思维漏洞的方法。一个直接从“添加端点”跳到“添加集成测试”且中间毫无过渡的提交就是一个信号。清晰的提交历史具有可审计性，这是单一的差异转储永远无法做到的。

代码仍然是一条消息。只是现在，作者可能不是人类。

## 结构是消息的一部分

如果你一直在仔细阅读，你可能已经注意到本文的每个标题在阅读之前就准确地告诉了你里面有什么。你不需要阅读全文就知道自己身处何处。这就是优秀提交信息的作用。这就是结构良好的PR所做的事。结构本身就是一种沟通，当它做得好时，读者永远不需要猜测。
