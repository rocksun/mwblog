性能问题是 PHP 团队非常熟悉的领域。慢端点、响应时间不稳定以及不断上升的基础设施成本，规律性地出现在复盘会、事后分析和团队聊天频道中。然而，团队很少讨论如何投入时间和资源来持续解决性能问题。

认知与行动之间的差距指向了比语言运行时限制或工具缺失更深层次的问题；它反映了团队如何规划工作、衡量成功以及定义进展。这演变成了一个关于[性能改进为何消失](https://thenewstack.io/the-case-of-the-vanishing-performance-improvements/)的问题：当计划转化为路线图时，改进为何不见了？团队如何在不停止新功能开发的情况下，继续优先处理这些任务？

## 为什么性能惯例很难在路线图规划中幸存

产品路线图往往倾向于奖励可见的进展。新功能易于界定范围，易于展示，也容易与业务价值挂钩。而性能改进则是“无声”的。当它们成功时，什么都不会崩溃，用户也不会察觉到剧烈的变化，这使得在面临权衡时，其价值更难传达。

> “性能改进是无声的。当它们成功时，什么都不会崩溃，用户也不会察觉到剧烈的变化，这使得价值传达变得更加困难。”

来自 [Perforce 2026 PHP 现状报告](http://www.zend.com/resources/php-landscape-report)的调查数据支持了这一动态。性能和调试（以及[人才招聘担忧](https://thenewstack.io/php-web-skills-hiring-age/)位列前三）是 PHP 团队最常提到的挑战，但当被问及计划在何处投入开发时间时，提高性能的优先级却排在其他任务之后。与此同时，新功能交付年复一年地主导着规划讨论。

这种优先级模式创建了一个能在短期内满足产品目标的待办列表，但却稳步提高了变更成本。性能工作被推迟，优化滑向无限期的“以后”，而架构约束在每次发布中都变得更加难以解开。

## “维护”并不自动等同于优化

造成痛点与行动脱节的一个可能原因是，许多组织认为优化会在维护周期中自然发生。但维护时间通常被响应式工作消耗：事故响应、漏洞修复以及在压力下维持生产系统运行。

虽然这些任务至关重要，但它们几乎没有为刻意的性能改进留出空间。优化工作最终散落在重构、工单清理或广义的“质量”工作中。虽然这可能会产生增量收益，但很少能带来可预测的或复合的收益。结果是，性能变成了大家都认为重要，但很少有人能可靠地安排进日程的工作。

请记住，如果性能问题的第一个信号来自客户，那你就已经太晚了。尽管如此，61% 的[受访团队](http://www.zend.com/resources/php-landscape-report)仍依赖用户报告来识别和解决生产问题。但并非一切都无法挽回——虽然优先考虑性能改进的最佳时间可能是几个月前，但第二好的时间就是现在。

## 如何在不牺牲新功能的前提下优先考虑性能改进

能够有效管理性能的组织依赖于早期信号（趋势、阈值和对可接受行为的共享定义），并在问题演变为代价高昂的故障之前采取行动。换句话说，解决方案不是一次性的“优化修复”或一劳永逸的方案，而是流程、测量和所有权的一种转变。

以下是你可以采取的一些具体步骤，用于提高 PHP 性能并在不牺牲交付的前提下减少问题：

* **建立基准并定义“足够好”。** 如果没有被识别或命名，你的团队就无法优先处理某些事情。基准将性能从一个模糊的概念变成了一个可跟踪的指标。
* **在交付工作流中加入性能预算。** 通过为性能改进制定预算，你可以将处理性能问题从“以后再说”转变为一项准入要求。此外，你可以将预算节省与性能改进挂钩，以获得支持。
* **让监控和[可观测性](https://thenewstack.io/observability-every-engineers-job-not-just-ops-problem/)成为不可谈判的要求。** 调查中仅有 35% 的用户报告在关键应用中使用了应用性能监控（APM）工具。实施可观测性工具（包括 [OpenTelemetry](https://opentelemetry.io/) 或 [ZendHQ](https://www.zend.com/products/zendhq) 等选项）是从被动调试转向主动规划的有效途径。
* **为每个 Sprint 预留性能改进容量。** 在几乎所有的竞赛中，持续性都胜过英雄主义，小胜会随着时间推移产生复合效应。将固定比例的 Sprint 容量（哪怕是 10%）设为策略的一部分。
* **借助值得信赖的第三方力量来加速和保持连续性。** 性能工作通常需要专门的技能、持续的关注和跨团队协作。外部合作伙伴（如 [Zend Professional Services](http://www.zend.com/services/)）可以提供计划、客观分析以及许多团队在路线图压力下难以维持的连续性。

> “在几乎所有的竞赛中，持续性都胜过英雄主义，小胜会随着时间推移产生复合效应。”

性能挑战已经显而易见。而那些被忽视的优化将如何迅速影响未来的每一个决策，则需要更长的时间才能被察觉：更慢的发布、更高的基础设施账单、风险更大的升级以及抵制变更的系统。那些对性能工作进行规划、测量并配备人员的团队，减少了这种长期阻力。随着时间的推移，他们构建的应用能够更可预测地扩展，并随着业务需求的发展更轻松地适应。
