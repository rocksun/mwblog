Anthropic 推出的动态工作流允许 Claude 编写编排脚本并并行运行多个子智能体。实测显示其在复杂任务构建中表现出色，虽成本略高，但极大提升了处理大规模、长时间任务的能力。

> 译自：[Claude Code's biggest upgrade yet ran 5 agents at once — here's what happened](https://thenewstack.io/claude-code-dynamic-workflows-test/)
> 
> 作者：Jessica Wachtel

Anthropic 于 5 月 28 日发布了 [Claude Opus 4.8](https://thenewstack.io/claude-opus-48-release/)，并随之带来了 Claude Code 中的动态工作流。这一功能处于可完全测试的研究预览阶段，让 Claude 可以像一个开发团队一样运作，而不是仅靠单个 AI 单打独斗。动态工作流使 Claude 能够编写自己的编排脚本，在单次会话中启动数百个并行子智能体，并验证输出结果。引用 [Anthropic](https://claude.com/blog/introducing-dynamic-workflows-in-claude-code) 的话：“你通常需要花几个季度规划的工作，现在几天就能完成。”

动态工作流与标准单智能体在架构上的不同之处在于编排指令的位置。在常规子智能体会话中，Claude 会逐轮决定下一步做什么，并且每一个中间结果都会被添加到其上下文窗口中。而通过动态工作流，Claude 编写一个脚本来自行处理编排，因此上下文窗口只会收到最终答案。

根据 Anthropic 的 [文档](http://code.claude.com/docs/en/workflows)，将编排指令移动到单独的脚本中，而不是放在 Claude 的上下文窗口中，使得大规模并行运行成为可能。在 Anthropic 从事 Claude Code 开发的 [Thariq Shihipar](https://www.linkedin.com/in/thariqshihipar/) 在 [X](https://x.com/trq212/status/2061907538741006796) 上发布了这一最高评价：“工作流是自技能和子智能体功能以来，Claude Code 能力方面最大的升级。”

> “在动态工作流出现之前，你只有一个需要监督的 AI 杂工。现在，你拥有了一个总承包商，他带来了一整套团队，负责运行检查员，并直接交给你一份完成的工作。”
> ——知名 AI 教育家及爱好者 Ole Lehmann

知名 AI 教育家和爱好者 [Ole Lehmann](https://x.com/itsolelehmann/status/2060420998168809820) 在 [X](https://x.com/itsolelehmann/status/2060420998168809820) 的一篇文章中认为，动态工作流的发布“比 Opus 4.8 的发布重要得多”。他用通俗易懂的术语描述了动态工作流：在动态工作流之前，你只有一个需要监督的 AI 杂工。现在，你拥有了一个总承包商，他带来了一整套团队，负责运行检查员，并直接交给你一份完成的工作。就是这么简单。

这些都是相当强力的主张。但它真的有效吗？让我们一探究竟。我对比测试了动态工作流和单智能体工作流，以确定它们是一项有价值的功能还是营销噱头。

## 测试

动态工作流需要 Claude Code 2.1.154 或更高版本，以及 Max、Team 或 Enterprise 计划。我拥有 Max 计划。在开始之前，我已从 2.1.131 升级到 2.1.159。Opus 4.8 是 2.1.159 版本的默认模型。

我带着相同的目标进行了两次测试：构建一个名为 codebase-health 的 CLI 工具，用于分析任何本地代码库，并生成涵盖代码复杂度、文档覆盖率、依赖审计和测试覆盖率映射的 Markdown 健康报告。两个会话都在完全空的文件夹中启动。

我将在本文末尾粘贴这两个提示词，供任何有兴趣重现我测试的人参考。

## 动态工作流结果

在 6 分 59 秒内，Claude 规划了工作，搭建了一个共享契约，以便所有智能体都能在同一接口上进行构建，并同时调度了五个并行智能体。

每个智能体负责一个组件并独立工作（如提示词中所述）：

* **智能体 1** 构建了复杂度分析器，使用 Python 的 AST 模块来分析 Python 文件中的嵌套深度，并为 JavaScript 和其他语言提供启发式备选方案。阈值可配置：函数 50 行，嵌套 4 层，文件 400 行。
* **智能体 2** 构建了文档覆盖率分析器，使用 ast.get_docstring，按设计跳过私有方法和双下划线方法，并对非 Python 文件使用 JavaScript 注释启发式方法。
* **智能体 3** 构建了依赖审计工具以解析 requirements.txt 和 package.json，默认禁用过时检查的网络调用，并可通过可选标志启用。
* **智能体 4** 按惯例构建了测试覆盖率映射，查找 test_ 前缀、_test 后缀和 tests/ 目录，明确指出这是文件映射，而不是行覆盖率。
* **智能体 5** 将所有内容串联到一个带有 argparse 的 CLI 中，具备各分析器的错误隔离、总体健康评分以及自述文件。

结果：62 个通过的测试，两个可工作的入口点，可配置的标志，以及针对其自身代码库生成的用于自我验证的 SAMPLE_REPORT.md。当我询问它使用了多少 Token 时，五个子智能体报告合计使用了 109,237 个 Token。编排线程成本未测量，因为 `/cost` 在动态工作流会话中不可用。加上编排开销，总成本估计在 3-5 美元之间。

### 但它真的有效吗？

在新的终端窗口中，我针对 HTTPie 代码库（一个开源 Python CLI 工具）运行了该工具。我在几秒钟内就得到了完整的健康报告。

总体健康评分：60/100（一般）

* 代码复杂度评分为 93/100，发现了 35 个问题。
* 文档评分为 17/100，有 926 个未记录的公共函数。
* 依赖项评分为 100/100，但这仅仅是因为该工具无法解析 setup.cfg，而这正是 HTTPie 管理其依赖项的方式。
* 测试覆盖率评分为 28/100，72 个源文件中只有 20 个有匹配的测试文件。

虽然在大多数领域都很彻底，但反思依赖项的情况很重要。因为它无法解析 `setup.cfg` 而给出一个满分是一个局限性。这个工具是在不到 10 分钟内创建的。我不期望完美，但有更好的方法来记录这一点。Null、错误、未定义或类似的任何消息都足够了。

我（再次）意识到我正在测试 Claude 一次性构建 CLI 工具的能力。这是一个不可能的要求。大约一年前，我会对这个结果感到非常惊讶。但现在是 2026 年，AI 公司已经深深嵌入到我们的工作流中，使用 AI 现在已是入场券。

话虽如此，这第一次尝试，就像我从 Claude 的工具中看到的许多其他首次结果一样，接近目标但尚未完全达到。我们仍然需要现场有一位专家。

但是动态工作流与我们大多数人更熟悉的单智能体工作流相比如何呢？那是另一回事了。

## 单智能体结果

单智能体构建了另一个工具。它选择了 JavaScript 而不是 Python。提示词中均未包含语言偏好，我将其包含在这里只是作为一个细节，而不是缺陷。

结果：一个可工作的 JavaScript CLI 工具，带有 src/ 文件夹、test/ 文件夹和 bin/cli.js 入口点。与动态工作流会话不同，`/cost` 在这里起作用，并提供了完整的细分，显示 Opus 4.8 处理了大部分工作，而少量的 Haiku 4.5 用于更简单的任务。`/cost` 确认会话成本为 2.25 美元，耗时 10 分 42 秒，增加了 1,789 行代码。

### 但它真的有效吗？

让单智能体工具运行需要额外的步骤。当我尝试用 pip3 安装它时（就像我安装动态工作流版本的方式），它失败了。单智能体没有创建 pyproject.toml 或 setup.py。它构建了一个 Node.js 项目，带有 package.json、src/ 文件夹、test/ 文件夹和 bin/cli.js 入口点。运行它需要 node bin/cli.js，而不是 pip install。

对 HTTPie 运行相同的工具得出的总体评分为 30/100（差）。文档覆盖率为 21%。测试覆盖率为 29%。这两个数字与动态工作流的发现大致相符。两个完全独立的工具，用不同的语言构建，具有不同的架构，对同一个代码库得出了相似的结论。

单智能体版本总体评分较低，部分原因是权重决策。它将文档权重设为 30%，测试覆盖率权重设为 30%，这拖累了总分。此外，由于无法运行依赖审计，单智能体工具返回了 `_No package.json or requirements.txt found._`。这是处理此问题的更好方法。

## 对比

两个工具都有效。两个都发现了 HTTPie 中相同的实际问题。两个都错过了依赖审计，因为它们都无法解析 setup.cfg。两个都对阈值、边缘情况和错误处理做出了明智的独立工程决策。

动态工作流构建的结果更精致。它包括了共享契约架构、62 个通过的测试、可配置的标志以及内置的自我验证。但基于我们在这里构建的内容，对于这种（非常小）规模的任务，它仅略好一点。

现在让我们根据估算费用进行判断。我使用这个 7 分钟构建的大致成本来估算 24 小时运行动态工作流的成本。以 Opus 4.8 为例，输入 Token 每百万 5 美元，输出 Token 每百万 25 美元，假设粗略的 70/30 输入输出比例，子智能体成本约为 225 至 375 美元。加上编排开销，全天 24 小时并行运行的总成本可能为 400 至 600 美元。

> 单智能体的成本更低，但只有当工作完成时，便宜才有意义。而在长期运行的任务中，它很可能无法完成。

单智能体运行 24 小时的估算成本为 300 美元。单智能体更便宜，但只有当工作完成时，便宜才有意义。而在长期运行的任务中，它很可能无法完成。上下文窗口限制将迫使它重新开始或丢失对早期工作的跟踪。

考虑到所有细节，我很高兴地报告，动态工作流确实不负众望。

## 提示词

***动态工作流提示词：***

*创建一个工作流来构建一个名为 codebase-health 的 CLI 工具，用于分析任何本地代码库并生成 Markdown 健康报告。启动并行智能体来同时构建每个组件：*

***智能体 1***: 代码复杂度分析 — 查找长函数、深度嵌套逻辑和过大的文件

***智能体 2***: 文档覆盖率 — 识别没有文档字符串或注释的函数和类

***智能体 3***: 依赖审计 — 读取 package.json 或 requirements.txt 并标记过时或未使用的包

***智能体 4***: 测试覆盖率映射 — 识别哪些文件有对应的测试文件，哪些没有

*每个智能体都应独立构建其组件，解释其构建内容和决策方式，并包含测试。最后一个智能体应将所有四个组件串联成一个可工作的 CLI 工具。*

***单智能体提示词：***

*构建一个名为 codebase-health 的 CLI 工具，用于分析任何本地代码库并生成 Markdown 健康报告，涵盖四个方面：*

1. ***代码复杂度*** — 查找长函数、深度嵌套逻辑和过大的文件
2. ***文档覆盖率*** — 识别没有文档字符串或注释的函数和类
3. ***依赖审计*** — 读取 package.json 或 requirements.txt 并标记过时或未使用的包
4. ***测试覆盖率映射*** — 识别哪些文件有对应的测试文件，哪些没有

*该工具应可安装并以 `codebase-health /path/to/project` 运行。包含测试和 README。*
