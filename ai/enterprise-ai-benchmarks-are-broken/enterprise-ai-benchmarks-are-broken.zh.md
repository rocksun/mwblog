每一家企业软件供应商都声称其 AI 代理已[达到生产就绪水平](https://thenewstack.io/scaling-ai-agents-in-the-enterprise-the-hard-problems-and-how-to-solve-them/)，但如何衡量这一指标仍悬而未决。

由 Nutanix 联合创始人 Dheeraj Pandey 领导的企业 AI 平台 [DevRev](https://devrev.ai/) 认为，虽然存在 TAU-Bench 和 Agent’s Last Exam 等基准测试，但它们都无法真正测试员工实际执行的工作，在这些工作中，跨大型上下文窗口工作的能力往往比单纯的推理能力更重要。

本周，DevRev 发布了其 [Enterprise AI Agent Benchmark](https://devrev.ai/enterprise-bench-methodology) 的第一个版本，这是一个开放式测试，公开了数据集、评估工具、评判标准、[结果](https://benchmark-dashboard-five.vercel.app/analysis?uuid=8f38e704-8491-4099-9aa9-fc7cac32d20c) 以及原始追踪记录 [available](https://benchmark-dashboard-five.vercel.app/headline)，因此任何组织都可以将其应用于自己的系统。

此次发布仅涵盖了四级框架中最低的两个层级（L1 和 L2），DevRev 寄望于社区能帮助构建更高级别的层级。

DevRev 是在 Laude Institute 的评估工具 Terminal Bench 的基础上构建了该基准测试，该研究所的研究人员隶属于加州大学伯克利分校和斯坦福大学。它还邀请了加州大学伯克利分校教授兼 [Bespoke Labs](https://bespokelabs.ai/) 联合创始人 Alexandros Dimakis 对方法论进行了审查，Bespoke Labs 为前沿实验室投资的多个基准测试构建了任务。

Pandey 和 DevRev 首席技术官 Ahmed Bashir 告诉 *The New Stack*，构思这个新测试的初衷源于一种不满。在参加了一场又一场会议，看着每个展位都在推销同样的东西后，Bashir 不断质问谁在进行核实。

> “人们可以说他们拥有行业领先的 AI，而我问，‘基于什么指标？’……没有方法。这是即兴的。”

“人们可以说他们拥有行业领先的 AI，而我问，‘基于什么指标？’”Bashir 说。“没有方法。它是即兴的。它建立在这是一个营销概念的想法之上。没有基准。在数百个甚至数千个与 AI 相关的基准测试中，是否有一个真正触及了企业所寻找的本质？”

## 数据才是难点

DevRev 团队认为，前沿实验室所看重的基准测试奖励的是奇异的推理能力，而企业工作很少是那样的。

“我们不会通过创建更复杂的任务来更接近企业需求，”Bashir 说。“我们需要做的是将适度的任务复杂性与数据组织方面极大的复杂性结合起来，而数据组织关乎信息的来源、信息的形态以及控制信息访问的权限。”

> “我们不会通过创建更复杂的任务来更接近企业需求。”

大多数企业任务并不像化学问题那么难。它们是适度的任务，但因为运行这些任务的数据分散在各个系统中、形态不一致、且被权限锁定（数据孤岛从未真正消失），从而极其耗时。任务复杂性是一个已经被足够解决的问题，但 Bashir 说，数据复杂性作为衡量对象是“相当新颖的”：“数据本身可以通过各种方式进行组织、塑造和访问控制，从而使其访问变得复杂。”

为了将其转化为基准测试，DevRev 构建了所谓的“尺度不变地面实况”（scale-invariant ground truth）。该数据集模拟了一家四种规模（1x、4x、16x 和 64x）的中型软件公司。对于所有四种规模，每个任务的正确答案保持不变。额外的数据是噪声，运行正常的代理绝不应该提取这些数据。在 1x 时，大约 40% 的工单与给定问题相关。而在 16x 时，只有 2.5% 相关。

团队认为，这种设计针对的是大多数系统所依赖的捷径。当上下文窗口很小时，代理擅长在草堆里找针，但在生产规模下，这一点就会失效。

该基准测试从三个维度对代理进行评分：精度（是否通过可验证的来源和可审计的路径获得了正确答案？）、效率（成本是随着问题还是数据规模变动？）以及安全性（权限边界是否得到尊重，且每个操作是否可追溯？）。

一个独立的 LLM 裁判会根据已发布的标准核实结果，并且每次运行都必须提交其追踪记录以及评分。

## 四个层级，借鉴自自动驾驶

L1–L4 框架描述了递进的自主性，就像自动驾驶的级别一样。

Bashir 将 L1 任务描述为跨“最多两到三个来源”的检索和合成，界限清晰且为单轮或少量轮次。他说，L2 任务“通常涉及多步思考，但仍然是单一领域的”，其中计划非常明确，但代理尚未采取行动。L3 提升到跨结构化和非结构化数据混合的跨领域问题解决，而 L4 是完全自主，Pandey 称之为“你必须更改代码才能让 L4 工作”的层级。

Pandey 将这种进展概括为搜索、答案和行动的“三位一体”。他说，企业搜索“很大程度上已经被商品化了”，这就是为什么大多数供应商已经在 L1 阶段竞争。L2 是“我的订单在哪里，我的取消在哪里，我的退款在哪里”。

行动是买家真正想要的部分，从 L3 开始，但 DevRev 认为大多数市场仍然停留在搜索阶段。

L3 和 L4、更大的数据集以及语音场景都在路线图中，但 Bashir 表示公司暂缓了它们，以先吸引第三方提交。“我们希望看到提交内容有一个渐进的过程，这样就不只是我们的提交，也有第三方的提交，”Bashir 说。

## 相同的模型，不同的结果

为了展示该基准测试的实际应用，DevRev 让其自己的代理 Computer（一个在共享、感知权限的内存层之上跨企业系统工作的 AI 代理）与 [Claude Code](https://thenewstack.io/claude-code-vs-cursor-vs-codex-vs-antigravity-2026/) 进行了正面交锋。两者都在相同的 Opus 模型家族上运行，以相同的方式进行评判，并通过供应商 API 和 [Model Context Protocol](https://thenewstack.io/model-context-protocol-a-primer-for-the-developers/) (MCP) 服务器的忠实协议副本执行相同的 L1–L2 任务。Computer 综合排名第一。

在 L1–L2 的结果中，DevRev 的 Computer 在测试的每个数据规模上准确率都高出 22 到 35 个百分点，并且它以少得多的 token 达到了这些答案：每个正确答案消耗 268,000 个 token，而 Claude Code 为 902,000 个，即减少了 3.4 倍。将数据集从 1x 扩展到 64x，Computer 的 token 消耗量增加了 11%，而 Claude Code 的消耗量增加了 55%。

由于两个代理都使用了相同的 Opus 级模型，DevRev 将差距归因于数据检索架构，而不是模型本身。该基准测试将企业 AI 系统分为两类：成本随数据量增加的系统，以及成本与问题复杂性挂钩的系统。DevRev 的观点是，第一种在接触真实企业数据时无法存活。

## 供应商自己的量尺

由供应商自己评分的基准测试值得怀疑，但 DevRev 确实引入了第三方专家来验证它。

该基准测试的核心变量是规模化后的噪声数据，这奖励了选择性检索，并惩罚了将所有内容加载到上下文中的做法。并非巧合的是，这正是 DevRev 架构旨在解决的问题。比较目标 Claude Code 是一个被强制用于替代通用企业代理的编程代理，这并非 Anthropic 专门为此工作推销的产品，尽管 Anthropic 的 [Cowork](https://thenewstack.io/claude-cowork-cloud-mobile/) 是建立在相同的基础之上的。

Bashir 说，该公司将其任务在内部之外进行了审查，选择“将这些任务提交给 Laude Institute 以及 Bespoke Labs 进行审查，因此它已经过独立验证。”

发布数据集、工具以及其自己的追踪记录可以让任何人重新运行测试或对分数提出异议，该公司也邀请竞争对手提交到公开排行榜。这提高了信任的基线，但这并不能改变 DevRev 选择衡量哪个维度这一事实。

DevRev 认为这是第一个开放的、供应商中立的企业基准测试。但类似的努力也存在。Salesforce 在 2025 年推出的 CRMArena-Pro 模拟了 Salesforce 组织，并对代理的准确性、成本和安全性进行了评分。Sierra 的 tau-bench 测试了企业级领域中的工具-代理-用户交互和政策遵从性。两者也同样来自供应商。

DevRev 基准测试的不同之处在于数据规模维度以及将包括追踪记录在内的一切开源的决定，而不是企业代理基准测试的理念。与此同时，作为模板的编程基准测试 SWE-bench 已经趋于饱和，前沿模型聚集在顶部，并且[污染警告不断堆积](https://openai.com/index/separating-signal-from-noise-coding-evaluations/)。

## 是内存，而不是模型

在该基准测试中，Anthropic 的 Opus 4.6 和 Opus 4.8 表现几乎相同。“只要有数据，两个模型都可以完成任务，”Bashir 说。这些模型在企业级任务上已经足够好，以至于更新的模型几乎不会提高分数，这意味着现在的差异化来自其他地方：评估工具。

> “当你拥有更多数据来源时，在任何给定时间对所有来源有深刻理解的能力就会下降……你开始忘记你记得的事情。”

Bashir 指出了 Claude Code 的一个小版本更新，该版本专门针对 MCP 进行了提示词改进。“实际上，它在模型版本发布之外也在变得更好，”他说。他认为，前沿实验室正在优化代理处理数据的方式，而不只是优化它们的推理方式。

这很重要，因为在 Bashir 的描述中，MCP “是一种与源数据的无接触关系”，实际上是无状态的。“当你拥有更多数据来源时，在任何给定时间对所有来源有深刻理解的能力就会下降，”他说。“如果你有三四个来源，你就会开始挣扎。你开始忘记你记得的事情。”
