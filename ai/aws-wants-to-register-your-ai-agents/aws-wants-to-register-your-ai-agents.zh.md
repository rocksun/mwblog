AWS 周四推出了 AWS Agent Registry，这是一项旨在帮助企业在团队之间发现、共享和重用 AI 智能体（Agents）、工具和技能的新服务。

这个新的注册表是 [AWS AgentCore](https://thenewstack.io/aws-unveils-bedrock-agentcore-to-scale-ai-agents-from-prototype-to-production/) 的一部分，后者是该公司用于构建和部署智能体的、与模型和框架无关的解决方案。尽管它会为在 AWS 平台上运行的智能体提供一些额外功能，但该服务的目的是索引来自任何供应商的智能体。

AWS 在今天的公告中指出：“没有任何组织的智能体格局完全存在于单一供应商之内。智能体构建在 AWS 服务、其他云平台和本地环境中。一个仅覆盖部分技术栈的注册表会使其余部分变得不可见，而不可见的智能体无法被发现、治理或重用。”

根据 AI 开发平台 [OutSystems](https://www.outsystems.com/1/state-ai-development/) 最近的一项调查，几乎每家企业公司现在都至少在探索如何使用 AI 智能体，但只有约三分之一的企业拥有集中的 AI 治理方法。

由于构建 AI 智能体变得越来越容易——员工可以轻松使用现成的工具而无需征得 IT 部门的许可——管理所有这些智能体成了一项重大挑战。注册表虽然不能解决所有问题，但至少应该能让控制这种“智能体扩张（agent sprawl）”变得更容易。

AWS 团队在公告中辩称：“如果没有中央注册表，开发人员会向外寻找第三方工具，或者重复邻近团队已经交付的工作。你会失去对已构建内容、所有者以及是否获准使用的可见性。”

有了注册表，开发人员可以先搜索注册表，查看是否已有可用的功能，然后在决定自行构建之前重用现有的功能。

## 智能体时代的目录

AWS Agent Registry 将存储每个智能体、工具、MCP 服务器、智能体技能以及围绕这些智能体的其他附属资产的元数据，包括它使用的协议、暴露的能力以及实际调用它们的方法。

要注册智能体，开发人员可以通过 AWS 控制台、SDK 或 API 提供元数据。然而，最简单的方法是将服务指向 MCP 或 Agent-2-Agent (A2A) 端点，注册表将自动收集所有这些详细信息。

由于注册表本身提供了 API 和 MCP 服务器，因此像 Claude Code 或 AWS 自家的 [Kiro IDE](https://thenewstack.io/kiro-is-awss-specs-centric-answer-to-windsurf-and-cursor/)（其针对 Windsurf 和 Cursor 的竞品）等客户端可以轻松查询注册表以获取可用工具和其他能力。

## 内置治理

鉴于这是一个企业级产品，它还带有自己的治理功能。AWS 写道：“如果没有治理，任何人都可以注册任何东西。你会失去对可发现内容的控制，无法强制执行标准，无法跟踪所有权，也无法管理智能体从开发到退役的全生命周期。”

因此，为了管理谁可以发布到注册表，管理员可以为谁可以发布以及谁可以发现这些智能体定义权限。还有一个审批流水线可以挂载到现有的审批工作流中，当然，还有一种方法可以在智能体达到生命周期终点并“退役到云端的伟大数据中心”时将其移除。

随着时间的推移，AWS 计划让注册表能够自动发现在组织内部 AWS 环境中构建的任何智能体，并提供更多关于这些智能体如何使用的运营数据。

## 竞争对手

当然，AWS 并不是第一个进入智能体注册表市场的。微软在 2024 年 11 月的 Ignite 大会上宣布了 Agent 365，将其定位为在整个企业中发现、治理和监控 AI 智能体的集中控制平面。配合将零信任身份管理扩展到 AI 智能体的 Entra Agent ID，微软的产品在检测“影子 AI”方面比 AWS 走得更远。

Google Cloud 在 Vertex AI Agent Builder 中也有一个工具治理层，而 Google 旗下的 Apigee 现在也集成了 Google 所称的 Agent Registry。

在开源方面，Solo.io 将其 [agentregistry 项目](https://aregistry.ai) 贡献给了 CNCF，提供了一个更具供应商中立性的目录，具有语义搜索、审批工作流以及对 MCP 和 A2A 的支持。

谈到智能体、MCP 和技能注册表，这个领域的参与者甚至更加广泛，Chainguard、Kore、JFrog、Kong 等许多公司都在提供这一主题的不同变体。
