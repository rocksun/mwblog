Spark 4.2 版本发布，通过引入原生向量搜索、受控指标、Python互操作性优化及增强的流处理能力，显著提升了平台在AI生产环境中的地位，使开发者能在一个平台上完成更多数据处理与AI任务，减少对外部工具的依赖。

> 译自：[Spark 4.2 has a feature that could retire your vector database](https://thenewstack.io/spark-4-2-ai-workloads/)
> 
> 作者：Amanda Caswell

Apache Spark 4.2 于上周发布，标志着 Spark 在企业数据处理核心领域十余年地位的进一步扩展。

随着针对 AI 工作负载的新功能（包括受控指标、向量检索原语、实时处理、改进的 Python 支持和原生地理空间分析）的推出，Spark 4.2 在近期 AI 和流处理功能的基础上进行了构建，反映了当今许多工程团队使用该平台的方式。该版本在 Spark 作为数据处理引擎的传统角色之上，[增加了支持生产级 AI 应用所需的更多功能](https://thenewstack.io/meta-iris-ai-chip/)。

此次发布引入了多项功能，使开发人员无需离开平台即可完成更多工作，对于已经在使用 Spark 的团队来说，这意味着需要管理的系统更少了。

## 受控指标防止冲突

一个团队对业务指标的定义并不总是与其他团队相同。随着时间的推移，这些差异可能导致报告冲突，并使人们不确定该信任哪个数字。

当 AI 应用开始消耗分析师和商业智能工具使用的相同企业数据时，这个问题会变得更加棘手。如果不同团队对同一指标的定义不同，AI 系统对同一个问题可能会产生不一致的结果。

Spark 4.2 引入了受控指标视图（governed metric views）来解决这个问题。组织可以定义一次业务指标，并在多个应用程序中重用该定义。指标视图使维度和度量成为 Spark 可以理解的一等对象，因此无论谁或什么在查询它，引擎都能保留预期的聚合语义。

> 组织可以定义一次业务指标，并在多个应用程序中重用该定义。

## 原生向量搜索

其中一项更重要的补充是原生向量搜索，它减少了在 Spark 和单独的向量数据库之间移动数据的需求。

Spark 4.2 引入了向量距离和相似度函数、向量归一化、向量聚合，以及 NEAREST BY（一种用于 Top-K 相似度搜索的新 SQL 操作符）。通过将向量搜索引入 Spark，开发人员可以将更多的检索管道保留在同一平台上。

> 通过将向量搜索引入 Spark，开发人员可以将更多的检索管道保留在同一平台上。

## Python 互操作性变得更容易

Spark 4.2 使在 Spark 和基于 Arrow 的工具之间移动数据变得更容易。通过对 Arrow C Data Interface 和 PyCapsule 协议的支持，Spark DataFrames 可以直接传递给 Polars 和 DuckDB 等工具，而无需复制或序列化底层数据（前提是双方都支持这些标准）。

Python 还获得了一些其他更新。PySpark 得到了扩展；Arrow 优化的 UDF 执行现在成为默认设置，并且 Python 数据源现在包含内置的时间和内存分析功能，以帮助开发人员排查自定义连接器的问题。

## Spark Connect 使引擎可被代理（Agents）调用

Spark Connect 通过基于 gRPC 和 Arrow 的协议将客户端与 Spark 服务器分离，在 4.2 版本中进行了多项更新。其核心理念是：客户端构建逻辑计划，服务器负责分析、优化和执行，结果以 Arrow 批次的形式返回。客户端不需要完整的 Spark 运行时或共置的 JVM。

此更新包括对该项目客户端-服务器接口 Spark Connect 的几项更改。[AI 应用可以将处理请求发送到远程 Spark 集群](https://thenewstack.io/agentic-ai-token-costs/)，而工作则继续在 Spark 内部运行。该版本改进了 RDD API 兼容性、错误处理和状态报告。

## 流处理赋能实时 AI

流处理在 Spark 4.2 中获得多项更新，包括自动 CDC 和实时模式。许多 AI 应用依赖于持续更新的数据，而不是定期的批处理作业。自动 CDC 为 Spark 声明式管道带来了顶级的数据变更捕获（change data capture）功能，处理合并逻辑以在源数据发生变化时保持目标表处于最新状态——这在过去需要手写容易出错的代码。新的 CHANGES SQL 子句允许团队通过单一的 SQL 接口检索数据变更。

Spark 4.2 还增加了内置的 GEOMETRY 和 GEOGRAPHY 类型以及用于位置感知分析的 ST_* 函数，而无需外部空间扩展。对于从事位置数据（物流、房地产、物联网）工作的团队来说，这消除了另一个将数据移出 Spark 的理由。

## 更宏大的图景

Spark 4.2 将更多的 AI 和数据栈引入到平台本身。[曾经依赖单独工具的功能](https://thenewstack.io/future-proof-ai-infrastructure/)现在可以直接在 Spark 中处理。

对于目前使用 Spark 进行 ETL 然后将数据移交给其他系统进行检索、治理或实时处理的团队来说，这个版本开始模糊了这条界限。随着越来越多的 AI 应用直接在运营数据上运行，Spark 正在成为服务层的一部分，而不仅仅是为服务层准备数据。

> Spark 正在成为服务层的一部分，而不仅仅是为服务层准备数据。
