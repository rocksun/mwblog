Tenet Security揭露了名为“Agentjacking”的攻击手段。黑客利用公开的Sentry DSN密钥注入伪造错误报告，诱导Claude Code、Cursor等AI编程助手执行恶意代码，直接获取开发者权限及机密信息，且因流程“合规”而难以被传统安防检测。

> 译自：[A public Sentry key is all it takes to hijack Claude Code, Cursor, and Codex](https://thenewstack.io/agentjacking-sentry-mcp-attack/)
> 
> 作者：Janakiram MSV

6月17日，AI代理安全初创公司Tenet Security的威胁实验室团队记录了一种他们称之为 [agentjacking](https://tenetsecurity.ai/blog/agentjacking-coding-agents-with-fake-sentry-errors/) 的攻击方式。

整个攻击过程依赖于一个针对 [Sentry](https://sentry.io) 中未解决错误的常规修复请求，Sentry 是成千上万个团队接入应用程序的错误监控服务。而仅仅一条虚假的错误报告，就能将一个AI编程代理变成开发者机器上的代码执行引擎。过程中既不需要接触恶意软件，也不会窃取密码。

> 而仅仅一条虚假的错误报告，就能将一个AI编程代理变成开发者机器上的代码执行引擎。过程中既不需要接触恶意软件，也不会窃取密码。

你可以把这种攻击想象成滑入建筑作业系统中的一张伪造维修单。值班的承包商信任该系统，将这张单据视为指令并执行，而从不会质疑是谁提交的。 [模型上下文协议 (Model Context Protocol)](https://thenewstack.io/why-the-model-context-protocol-won/) 以同样的方式将编程代理与外部服务连接起来。代理将这些服务返回的任何内容都视为值得采取行动的指导。

## 为什么代理无法区分数据与指令

要理解为什么这种攻击有效，我们需要看看 Sentry DSN 的设计初衷。DSN（数据源名称）是一种仅写凭据。Sentry 文档中指出将其嵌入前端 JavaScript 是安全的，这样应用程序就可以报告错误，而不会暴露项目的其余部分。根据设计，它是公开的，并且摄入端点除了 DSN 本身之外不需要任何其他信息。

这种安排在人类阅读错误报告时是安全的。当AI代理阅读同样的报告时，这一设计决策就成了攻击者的突破口。任何找到 DSN 的人都可以发布一个精心构造的事件，Sentry 会将其像真正的崩溃一样处理。通过检查网站的 JavaScript、运行 Censys 查询或搜索 GitHub 代码，就可以获取 DSN。

> 这种安排在人类阅读错误报告时是安全的。当AI代理阅读同样的报告时，这一设计决策就成了攻击者的突破口。

代理无法将它读取的数据与行动指令区分开来。种植在代理会读取的地方（即使是人类永远不会看的地方，例如错误日志）的命令可能直接运行。这是模型本身的局限性，而不是通过配置更改可以修复的问题。

公开的 DSN 和代理对 MCP 输出的信任是一种致命的组合，因为两者单独看是无害的，但结合起来却非常危险。凭据允许攻击者写入代理读取的数据中，而代理提供了基于这些数据采取行动的权限。

## 攻击是如何展开的

攻击链的展开方式如下：每一步单独看都是普通的，没有任何环节看起来像是一个漏洞利用。

![](https://cdn.thenewstack.io/media/2026/06/7e47d006-agentjacking_attack_flow1-1024x937.png)

### 1. 寻找 DSN

攻击者首先寻找目标的 DSN。Sentry 的文档指出公开暴露它是安全的，所以它存在于无数生产网站的 JavaScript 中。通过 Censys 查询或 GitHub 代码搜索可以找到它。

### 2. 发布精心构造的事件

获取 DSN 后，攻击者向 Sentry 的摄入端点发送一个错误事件。除了 DSN，不需要任何身份验证。攻击者控制整个有效载荷，从消息和标签到上下文键和堆栈跟踪。Sentry 返回 HTTP 200 并将该事件与真正的崩溃一起归档。

### 3. 将命令伪装成解决方案

精心构造的事件在消息和上下文字段中携带 markdown。当 Sentry MCP 服务器将事件交给代理时，markdown 被渲染为标题、代码块和一个伪造的解决方案部分。该部分符合 Sentry 自己的模板，其中包含攻击者希望运行的 `npx` 命令。

### 4. 引导代理

开发者要求代理修复未解决的 Sentry 问题。成千上万的团队每天都会发出这样的请求。代理通过 MCP 拉取注入的事件，并将伪造的解决方案视为可信指导。从那里开始，它被引导向建议的命令，而不是源代码。

### 5. 运行命令

代理以开发者的权限在开发者自己的机器上运行该命令。在 Tenet 的测试中，包来自公共 npm 注册表，并自称为安全扫描，这使得演示处于负责任披露的范围内。

### 6. 获取机密

一旦运行，该包确认它可以读取环境变量、云配置文件和凭据存储，然后向 Tenet 控制的服务器发送信号，确认暴露是真实的。AWS 密钥、GitHub 令牌和 git 凭据都可以从这一个立足点中获取。

## Tenet 在大规模验证中证明了什么

想象一下，一个开发者在周五下午一次性清理 Sentry 问题积压，这是攻击中每一步都依赖的例行公事。Tenet 围绕这一时刻进行了验证，并衡量了其影响范围。

Tenet 报告称，通过被动侦察发现了 2,388 个具有可注入 DSN 的组织，其中 71 个位列 [Tranco 前 100 万繁忙站点列表](https://tranco-list.eu/)。同样的情况也存在于成千上万个从未测试过的项目中。

> Claude Code、Cursor 和 Codex 都对注入的错误采取了行动，该团队在不同组织中记录了 100 多次确认的执行。

Tenet 研究员 [Ron Bobrov](https://il.linkedin.com/in/ron-bobrov) 报告称，在受控验证波次中成功率达到 85%。Claude Code、Cursor 和 Codex 都对注入的错误采取了行动，该团队在不同组织中记录了 100 多次确认的执行。Tenet 出售他们认为必要的代理运行时防御系统。这些数字最好被视为他们自己的受控测试结果，而不是独立的测量值。

Tenet 确认在一家价值 2500 亿美元的财富 100 强科技公司内部开发人员的机器上执行了代码，这是全球最大的科技公司之一。同样的活动也波及到了在沙箱化 CI 流水线中运行的代理、托管 Windows 机器上的 WSL 内部，以及企业 VPN 后面，无论是在 macOS 还是 Windows 上。

一个运行 Claude Code 的被捕获环境中持有一个实时 AWS 秘密访问密钥。它还持有其他连接代理的标识符，因此一个立足点打开的不仅仅是一台机器。据 Tenet 称，构建版本是当前的，于 2026 年 6 月初捕获，而非来自过时的实验室设置。

在企业内部，危险在于代理已经能够触及的所有内容。一个单一的注入错误将 CI/CD 凭据、私有仓库 URL 和云基础设施令牌暴露给了攻击者。这些正是平台团队每天都在保护的凭据。

## 为什么没有安全控制能发现它

使得 agentjacking 难以阻止的原因是每一步都是被授权的。攻击者从不触碰受害者的基础设施，开发者从未批准任何代码，代理完全按照被要求的方式行事。Tenet 将此称为授权意图链，这就是为什么 EDR、WAF、IAM、VPN 和防火墙没有记录任何值得标记的内容。

> 使得 agentjacking 难以阻止的原因是每一步都是被授权的…… EDR、WAF、IAM、VPN 和防火墙没有记录任何值得标记的内容。

提示词层的防御也没有帮助代理。研究人员表示，即使系统提示词和技能指示它们忽略不受信任的数据，代理仍然运行了有效载荷。这指出了当前模型处理工具输出方式的局限性，而不是团队可以关闭的设置。

## Sentry、模型供应商和运行时

一旦攻击奏效，仍然需要有人阻止它，这比听起来要难。在注入的错误和损害之间存在三层。两层是发出数据的平台和代理操作的模型供应商。第三层是代理周围的运行时，每个动作都在那里决定。

| 防御层 | 对 agentjacking 的影响 | 原因 |
| --- | --- | --- |
| Sentry 平台 | 有限 | 内容过滤器拦截了已知的有效载荷字符串，但摄入端点根据设计保持开放 |
| 模型供应商 | 最多部分有效 | 尽管系统提示词和技能指示忽略不受信任的数据，但代理仍运行了有效载荷 |
| 代理运行时 | 最直接 | 运行时在执行前查看每个动作，并且可以拦截源自外部数据的命令 |
| 网络和端点控制 | 无 | EDR、WAF、IAM 和 VPN 只能看到授权动作，因此没有任何内容触发警报 |

Sentry 对披露的回应是保持该攻击开放的原因。Tenet 于 6 月 3 日披露了该问题，Sentry 同一天予以承认。Sentry 拒绝从源头修复它。

该公司称此类攻击“在技术上无法防御”，并指出了模型供应商运行的中间件。它确实针对 Tenet 概念验证中的特定字符串发布了一个全局内容过滤器，这阻止了那一个有效载荷，但没有关闭路径。

## **谁拥有修复权**

这使得运行时处于聚光灯下，即代理周围决定每个行动的层。Sentry 将其开放端点视为一项功能，而模型不会可靠地拒绝该指令。因此，修复不能仅靠其中任何一方实现。

现在的争论焦点是关于谁拥有修复权，它的解决方式对团队来说比概念验证本身更重要。

## 超越 Sentry

总之，Tenet 证明的弱点并不单单属于 Sentry。任何将受外部影响的数据返回给代理的 MCP 集成都具有相同的暴露风险。随着越来越多的工具通过 MCP 连接，攻击者可以通过受信任的遥测触及的表面将会扩大。安全团队一年来一直警告的 [提示词注入 (prompt injection)](https://thenewstack.io/mcp-a-practical-security-blueprint-for-developers/)，现在已经有了一条从公开可用凭据到代码执行的清晰路径。

如果团队在没有检查这些服务返回内容的情况下，继续将代理连接到外部服务，注入的数据将不断找到执行路径。Tenet 开源了一套名为 [agent-jackstop](https://github.com/tenet-security/agent-jackstop) 的插入式配置，用于加强 Cursor 和 Claude Code 以抵御此类注入。它们为团队提供了一个具体的起点，同时更大的问题还在争论中。

一个在几秒钟内清理 Sentry 积压问题的代理，也会运行任何受信任工具交给它的东西。这使得代理周围的运行时成为软件供应链安全中的下一个真正边界。企业在信任第三方库之前已经会对它们进行审查。像对待第三方库一样对待每一个 MCP 集成，才能防止 agentjacking 将团队自己的遥测工具变成针对他们自己的武器。
