本月，AI 编程领域的话题发生了一些转变。争论的焦点不再是 AI 代理（Agents）能否编写生产代码，也不再是哪个模型最强，而是谁（或什么）应该负责向它们下达指令（prompting）。

“设计能够指引你代理的循环（design loops that prompt your agents）”这一概念引发了一周的热议，Matt Van Horn 对此进行了[精彩的总结](https://x.com/mvanhorn/status/2063865685558903149)。剥离掉噪音，其核心信号很明确：第三个[代理开发（agentic development）](https://thenewstack.io/5-key-trends-shaping-agentic-development-in-2026/)时代正在成型，它引发了关于工程师工作内容、软件交付成本以及底层基础设施等重大问题。

对于那些在 Kubernetes 上构建云原生应用的团队来说，这三个问题的答案即将变得至关重要。

## 三个时代，一个方向

代理开发让开发人员的抽象层级一次提升了一个台阶。

第一个时代是提示词驱动（prompt-driven）。开发人员处于循环内部，键入指令，读取输出，键入修正。工作吞吐量受限于单个开发者的注意力。

第二个时代是规范驱动（spec-driven），这也是目前大多数采用该技术的团队所处阶段。开发人员在前期投入大量精力：详细的规范说明、上下文文档、仓库中编码的约定。代理根据规范执行，人类负责审核完成的工作。工作单元从“提示词”演变为“任务”。

第三个时代将“循环”本身作为工作单元。一个循环是一个小型程序，它向代理发送提示，评估响应，判断目标是否达成；如果未达成，则根据所学信息再次发送提示。它按计划运行，而非依赖人类的注意力。循环可以调度其他循环。

开发人员[不再编写代码](https://thenewstack.io/netlify-agent-experience-engineers/)，且越来越多地不再编写任务。他们转而编写生成、评估和重试工作的系统。

> “开发人员不再编写代码，且越来越多地不再编写任务。他们转而编写系统，负责生成、评估和重试工作。”

怀疑论者称此为“营销包装更好的定时任务（cron job）”，这一类比在揭示其差异时很有用。定时任务执行的是固定的脚本，而循环在主体内包含一个决策者：一个能够读取工作状态并选择下一步操作的模型。工程上的挑战在于如何围绕该决策进行封装，使循环能够趋向于“正确”而非漫无目的地游走。

## 开发者的角色演变

在提示词驱动的世界里，开发者的杠杆作用源于引导技巧；在规范驱动的世界里，源于意图的清晰度。

在循环驱动的世界里，杠杆作用源于代理周围系统的质量。核心工程问题变为：这个循环在宣布成功前检查了什么？它失败时收到了什么反馈？它何时停止？

这些问题将角色一分为二。应用程序开发人员成为了意图和循环的编写者：他们决定构建什么并对目标进行编码。平台工程师则成为了“完成标准（what done means）”的拥有者。

循环运行的检查、运行的环境、运作的预算以及它附加在输出上的证据，都必须在组织内的每个循环中保持一致，而不是由每个循环各自即兴发挥。

这是一个有着新主题的熟悉模式。十年前，平台团队将 CI/CD 从每个团队拼凑的东西转变为了一条“铺设好的道路（paved road）”。代理循环的验证层也是同样的转变，只是它位于内部循环中，出现在任何 PR 存在之前，且运行在循环所产生的任何并行度下。

## 经济学：循环要么收敛，要么耗尽预算

人们容易认为代理使代码生成变得廉价，所以这些都不重要。事实并非如此。

代理让生成变快了，且比开发人员工时更便宜，但 Token 消耗是一个真实且不断增长的项目。一个运行数小时的代理是一项支出决策，而循环在设计上就是为了无限期运行的。那些在交互式会话中尚能承受的预算，在循环面前未必能撑得住。

首要的应对措施是护栏：迭代上限、无进展检测和支出上限。这些是必要的，但它们只能限制低效循环带来的损害，并不能使其变得高效。

循环效率取决于两个维度，且二者相互倍增。

首先是**反馈质量**，它决定了循环需要多少次迭代。一个得到模糊失败信号的循环只能猜测原因并重试；一个能从真实系统获得真实错误，且具备足够上下文来定位原因的循环，能修复实际问题并继续前进。反馈质量也限制了循环所能达到的准确度上限：循环只能收敛于其反馈所能观察到的目标。

其次是**循环闭合的位置**，这决定了每次迭代的成本。如果循环在 CI 中或 PR 之后闭合，每个周期都要支付流水线运行外加排队的时间，节奏是“分钟到小时”级。如果它在内部循环中闭合，针对代理可以直接访问的运行时，节奏则是“秒”级。

> “总循环成本 = 验证所需的迭代次数 × 每次迭代的成本。”

总循环成本是验证所需的迭代次数与每次迭代成本的乘积。这两个维度也相互影响：将真实反馈推送到 CI，每个周期会变慢，代理在期间只能迭代部分信号；将其拉入内部循环，则能同时压缩这两个参数。

## 云原生提高了“完成”的门槛

对于一个自包含的程序，这两个维度几乎是免费的。测试套件在几秒钟内本地运行并告知全部真相，因为真相是局部的。

在一个分布式系统中，真相并非局部。变更是否正确，取决于它与所调用的服务、所触及的数据存储和队列、以及它所运行的路由和策略层的交互行为。

代理能快速触及的反馈（如本地测试和桩代码）是局部的。而真正能说明真相的反馈通常存在于 CI 和共享的预发布环境（staging）中，那里的周期缓慢且存在竞争。云原生团队被迫在“收敛于错误目标的快速循环”和“以流水线速度迭代的真实循环”之间做出选择。

我曾写过关于传统环境选项为何在代理规模下力不从心的文章。这里重要的结论是：在云原生系统中，循环的反馈必须来自运行时，且该运行时必须能从内部循环中访问。云原生代理循环的架构，很大程度上取决于你给它提供的验证表面的架构。

## 云原生代理循环的四个层级

完整的系统包含四个层级，循环本身是其中最小的一部分。

**运行时层**。循环在每次迭代时都需要一个行为像生产环境但成本不高的环境。答案是共享集群上的轻量级瞬态环境（ephemeral environments）：仅部署变更涉及的服务，利用共享集群上的请求路由，引导循环的流量通过这些服务，而其余部分则通过实时的共享基准运行。

环境在几秒钟内实现，边际成本仅追踪变更的 Pod 而非整个堆栈，反馈则来自真实的依赖项和真实的数据路径。这就是将循环闭合点移至内部循环的关键。

**验证接口**。代理不会点击仪表盘，它们也不应该发明自己的“完成标准”。变更必须通过的检查属于平台团队定义、版本化并暴露给代理的声明式工作流，这是验证变更的授权方式。

由组织而非代理来决定什么叫“通过”。附加在变更上的证据来自人类可审计的过程，人类审查可以专注于意图和设计。

**反馈层**。这是收敛引擎。仅仅一个“通过”或“失败”的位（bit）只能告诉循环重试，而不能告知修改什么。运行时需要交回结构化的结果：哪个检查失败了、循环自身请求范围内的日志和链路追踪、边界处的行为差异。

反馈精确度的每一次提升都能消除猜测，而消除猜测就是消除成本。

**控制层**。预算、迭代上限、无进展检测，以及对每个循环运行和证明内容的持久化记录。这使得组织能够自信地运行多个循环，而不是在焦虑中运行一个循环。

当支出有界且收敛可衡量时，代理开发就成为了一项可以规划的产能，而不是一笔事后发现的账单。

“编写循环，而非提示词”是一个更大主张的显性表现：团队的杠杆作用现在存在于循环共享的验证系统中，而该系统由平台组织所拥有。

## 从哪里开始

向循环式开发的转变不会作为单一决策到来。它会表现为一个团队的实验快速收敛，而另一个团队耗尽了季度的预算却产生了一些无人信任的变更。区别在于四个层级，而非循环的巧妙程度。

对于云原生团队，运行时层是起点，因为其他所有层级都依赖于它。验证工作流的意义取决于其执行的环境，反馈的真实性取决于生成它的系统。
