本文介绍了AWS如何通过EKS节点监控代理实现大规模Kubernetes GPU节点的自动化修复。总结了在构建自愈系统时的核心经验，包括API契约维护、故障检测与诊断的分离、以及避免资源监控对分布式GPU训练任务产生性能干扰的策略。

> 译自：[Self-healing GPU nodes in Kubernetes: What we learned building the EKS node monitoring agent](https://thenewstack.io/self-healing-gpu-nodes/)
> 
> 作者：Sajjan Gundapuneedi

当你在 Amazon EKS 上运行大规模 Kubernetes 时，节点故障是常态。GPU 会从 PCIe 总线上掉线，容器运行时会卡死，网络接口会消失。在数以万计的集群中，“罕见”的硬件故障每天都会在舰队的某个角落发生多次。

多年来，每个人的应对方式都是一样的：运维人员醒来，读取仪表板，SSH 连接到节点，对其进行封锁（cordon）、排空（drain）、终止实例，然后等待替换。每一步都依赖人工，每一步都充满琐事。如果故障发生在周末凌晨 3 点，工作负载在有人察觉之前会处于降级状态运行数小时。

我们构建了 [EKS Node Monitoring Agent](https://github.com/aws/eks-node-monitoring-agent) 来帮助弥补这一差距，该代理已于今年 4 月开源。它能够检测节点故障并写入 Kubernetes NodeConditions，向 Karpenter 发出问题信号，随后 Karpenter 会在必要时自动替换该节点。该代理是一个更大系统的一部分。要了解它所处的位置，你需要理解谁在管理它所监控的节点。

> “在数以万计的集群中，‘罕见’的硬件故障每天都会在舰队的某个角落发生多次。”

[AWS 推出了 Amazon EKS Auto Mode](https://docs.aws.amazon.com/eks/latest/userguide/automode.html)，它全面自动化了 Kubernetes 集群基础设施：计算配置、扩缩容、网络、存储、操作系统补丁和安全加固，让团队专注于应用程序而非集群运维。它能动态选择最优的 EC2 实例（包括 P5、P6 和 G6 等 GPU 实例系列）、根据工作负载需求进行伸缩、整合未充分利用的节点并保持操作系统补丁更新。EKS Auto Mode 默认提供自动节点修复功能：故障检测、严重程度分类以及由 Karpenter 驱动的节点替换，这一切都是开箱即用的，无需安装插件、无需配置控制器、也无需编写修复策略。

这是我们构建自动节点修复系统的故事，包括塑造该系统的设计决策，以及在 GPU 规模化运营中获得的惨痛教训。

## 大规模构建自愈 Kubernetes 节点的六个教训

在数千个集群中运行此系统后，这些经验教训可以精简为一份清单。这些教训对我们来说并不独特。同样的模式出现在 NPD、NVSentinel、AKS Periscope、GKE 的自动修复以及任何构建自定义节点控制器的系统中。它们是应该作为检查清单的“民间智慧”。[开源仓库](https://github.com/aws/eks-node-monitoring-agent)以代码形式反映了每一个教训，从 API 中原因代码（reason-code）的稳定性保证，到解决了 GPU 工作负载干扰的抖动实现。

> “你的原因代码就是 API 合约。新增是功能，重命名是破坏性变更。”

1. **你的原因代码就是 API 合约。** 每个下游消费者（修复控制器、仪表板、客户自动化）都通过字面字符串匹配来识别它们。新增是功能，重命名是破坏性变更。严重程度的改变也是破坏性变更。像规划 API 版本一样规划它们。
2. **“缺失”和“未知”不是一回事。** “我们没有在监控”和“我们在监控但无法确定”需要下游自动化采取不同的响应。如果你禁用的监控器写入“未知”，某个控制器最终会对其采取行动。当你不在监控时，请不要发出任何信号。
3. **不要跨越所有权边界。** Kubelet 拥有工作负载驱动的条件。你的节点健康代理拥有硬件和基础设施故障。跨越该边界意味着你的修复系统正在与 Kubelet 的驱逐系统作斗争，其中一个必然会做出错误的决定。
4. **从源头测量延迟。** 检测 SLO 包含信号链中的每一跳：硬件事件到驱动程序日志，驱动程序日志到 journald，journald 到代理轮询，代理轮询到 NodeCondition 写入。最长的一跳决定了整体延迟。对于内核级信号，journald 的刷新节奏是瓶颈。对于通过 DCGM 的 GPU 遥测，基于推送的策略违规（DBE、XID、NVLink）是近乎即时的，但轮询式字段监控（NVSwitch 结构健康、时钟节流）有 5 分钟的最低阈值。了解每种检测使用哪条路径。
5. **检测和诊断是具有不同消费者的独立系统。** 检测服务于自动化（快速、持续、数据量小）。诊断服务于人类（按需、详细、数据量大）。混为一谈会降低两者的效果。
6. **根据规范测试遥测解读，而不是根据经验值。** 硬件遥测接口不是布尔型的。我们读取用于 GPU 结构健康的 DCGM 位域，并将非零值视为故障。当一次驱动程序更新将健康的返回值从零改为规范定义的非零掩码时，所有 GPU 节点同时被标记为不健康。安全断路器（按设计）发挥了作用，让我们有时间发布修复程序。教训是：如果你在解析来自 GPU 固件的封装枚举或位域，你的测试装置必须来自供应商文档，而不是之前硬件上碰巧返回的值。

## Kubernetes 中的节点健康检测：无人警示的陷阱

Kubernetes 生态系统中的每个节点健康代理都执行相同的翻译。Node Problem Detector (NPD)、NVSentinel、GKE 的自动修复、AKS 的 Linux 扩展以及 EKS 节点监控代理，都从机器获取嘈杂的低级信号，并将它们翻译成一组 Kubernetes 原语：NodeCondition、Event，有时是 CRD。这个翻译看起来很简单，但实际并非如此。

输出是一个 NodeCondition，它只包含类型、状态（True/False/Unknown）、原因代码和消息。这四个字段。但这个表面之下隐藏的决策决定了修复操作是有益还是有害。

**原因代码是公共 API。** 我们在这上面吃了苦头。在 1.6.2 版本中，我们将 NvidiaDeviceCountMismatch 从“警告”级别改为了“致命”。技术理由是充分的：一旦 GPU 从 PCIe 总线上掉线，除非重启或更换节点，否则它不会恢复。将其保留为“警告”意味着 GPU 工作负载会不断被调度到降级的节点上，浪费昂贵的加速器资源。所以我们发布了修复程序。结果下游自动化挂了。客户有基于旧严重程度的修复配置。过滤“警告”的仪表板停止显示故障。只对“致命”采取行动的自动化突然开始排空它以前从未触碰过的节点。从那时起，我们将每一个原因代码的增加视为功能开发，将每一次重命名或严重程度更改视为破坏性变更。

**“缺失”不得等同于“健康”。** 当我们在 v1.6.0 中发布按监控器配置功能时，我们必须做出选择。禁用的监控器需要产生某种输出（或不产生输出）。三个选项：写入 True（你的自动修复现在认为节点健康，因为你没在监控）、写入 Unknown（模棱两可，可能根据下游逻辑触发修复），或者完全省略条件。只有第三种是安全的。

回想起来这似乎很明显，但请考虑 NPD 通过完全不同的机制实现了相同的结果：通过构建标签在编译时禁用。NVSentinel 将其委托给操作员编写的 CEL 规则。上游 Kubernetes 规范定义了 Unknown 的含义，但如果你的修复自动化将 Unknown 视为可操作的，你将无缘无故地丢失节点。我们选择在监控器关闭时不发出任何信号，并将其记录为硬性合同。

**检测延迟由源头限制，而非代理。** 我们最初告诉客户，“我们在 30 秒内检测到内核恐慌”。这是错误的。我们代理的检测时间在 30 秒内。但内核恐慌出现在 journald 中，journald 的刷新节奏才是真正的瓶颈。如果 journald 需要 45 秒来写入那一行，我们 30 秒的声明就是不完整的。

对于 GPU 故障，情况更微妙，因为我们使用两条具有截然不同延迟特性的检测路径。关键故障（双位 ECC 错误、XID 错误、NVLink 故障、页面停用、热量和功率违规）通过 DCGM 的基于推送的策略违规通道。DCGM 一旦检测到违规就会通知我们的代理；没有轮询间隔。这些故障的检测近乎即时（实践中为亚秒级）。另一条路径使用 5 分钟的字段值窗口来监控 NVSwitch 结构健康、Fabric Manager 状态和时钟节流原因。该窗口是这些特定检测的基准，但不适用于触发自动修复的关键 GPU 故障。教训：面向客户的 SLO 必须包含真实源的延迟，同一子系统内的不同信号路径可能有截然不同的基准。

## 两种严重程度，一个开关：自动修复如何决定更换哪些节点

Kubelet 已经报告 DiskPressure、MemoryPressure 和 PIDPressure。NMA 用五个额外的条件补充了这些，涵盖了 Kubelet 不监控的领域：内核健康、容器运行时、网络、存储和加速硬件。每个检测都带有两种严重程度之一，严重程度是决定是否触发修复循环的开关。

**条件严重程度（Condition severity）** 指的是致命故障。它会将匹配的条件翻转为 False，并使节点符合自动修复资格。例如 GPU 设备计数不匹配、严重的 XID 和双位 ECC 错误、NVLink 和 NVSwitch 结构故障、缺失的 Fabric Manager 以及 Neuron DMA 和 HBM 不可纠正错误。在网络和运行时方面：VPC CNI 进程关闭、IPAMD 无法到达 API 服务器、由于 PID 耗尽导致的 fork 失败，以及工作负载卡死、在损坏的容器运行时后终止。这些是无法自行恢复的故障。在 GPU 节点上，单个降级的加速器可能会损坏训练检查点，或者每小时浪费数千美元的计算资源。

**事件严重程度（Event severity）** 是信息性的。它发布一个 Kubernetes 事件，NodeCondition 保持为 True，运维人员可以在不中断的情况下获得可见性。例如带宽上限、连接跟踪限制、Amazon Elastic Block Store (Amazon EBS) IOPS 节流、I/O 延迟、文件系统碎片、时钟漂移、活跃度和就绪性探测失败、kube-proxy 异常、GPU 热量和功率警告、PCIe 链路降级以及页面停用阈值。这些信号预示着在问题演变成致命故障之前的情况。

在任何方向上搞错严重程度的代价都很昂贵。过于激进，你就会终止健康的节点并毫无必要地置换工作负载。过于保守，降级的节点就会服务流量数小时，同时拥有坏内存条的 GPU 会损坏训练检查点。分类原则：如果故障是确定性的且归属于基础设施（硬件损坏、固件崩溃、物理链路断开），它就会触发替换。如果信号可能是应用程序引起的或短暂的，它将保持为信息性。你永远不想因为行为不端的 pod 占用了资源而终止健康的节点。

> “在任何方向上搞错严重程度的代价都很昂贵。过于激进，你就会终止健康的节点。过于保守，降级的节点就会服务流量数小时。”

DiskPressure、MemoryPressure 和 PIDPressure 是典型的例子。每个主要的自动修复系统（GKE、AKS、NPD）都独立得出了相同的答案：不要碰它们。这些是工作负载驱动的条件，而不是节点级故障。更换节点只是将行为不端的工作负载移动到一台新机器上，它会再次耗尽内存。正确的响应是 Kubelet 级的 pod 驱逐，而不是节点替换。如果你在构建节点健康系统，请尽早划定这条边界并公开记录它。

## 伤害了被保护对象的代理：健康监控带来的 GPU 工作负载干扰

最深刻的教训来自一位运行大规模分布式 GPU 训练的客户。他们的工作负载在数百个 GPU 节点上使用 NCCL 集体通信，通信组中的每个节点在任何节点继续之前都必须完成其步骤。一个慢节点会让每个节点都等待。

他们发现 NMA 本身导致了周期性的减速。代理的监控器都在独立的 goroutine 上运行，当它们的轮询间隔对齐时，数十个 goroutine 会同时唤醒并在一瞬间爆发式占用多个 CPU 核心。在通用的 Web 服务上，这可能是不可见的。但在分布式训练作业中，一个节点上微秒级的抖动可能会级联到整个 GPU 集群，导致可测量的吞吐量损失。

客户完全禁用了 NMA，并立即看到了改进。这对我们来说是可能的最坏结果：一个本应保护工作负载的健康代理如果反而干扰了它，那比没有代理更糟糕。

一旦理解了问题，修复就直截了当。我们在每个监控器的轮询间隔中添加了一个启动抖动（startup jitter）。每个 goroutine 将其第一次触发时间延迟一个随机偏移量（最高可达其基础间隔的 20%），错开唤醒时间，使它们在启动时不会对齐。我们缓存了每次轮询时命中 /proc 的系统调用。我们将共享同一间隔的处理程序合并到一个顺序工作队列中，减少了不需要自己线程的监控器的 goroutine 数量。结果是一个 CPU 配置文件平滑且可预测，而不是突发性的代理。

这个教训可以概括为：如果你的健康代理与工作负载运行在同一主机上，其资源消耗模式与其资源消耗总量同等重要。一个均匀分布使用 0.5% CPU 的进程是隐形的。一个以集中爆发方式使用 0.5% CPU 的进程可能会以训练时间损失而非 CPU 告警的形式，干扰对延迟敏感的分布式 GPU 工作负载。

这就是为什么按监控器配置如此重要。并非每个监控器都与每个工作负载相关。一个专用 GPU 训练集群，每个节点一个 pod 且没有 pod 流失，不需要 IPAMD 监控或环境扫描。我们发布了禁用单个监控器的功能，以便客户可以保持他们需要的健康覆盖范围，而无需支付他们不需要的覆盖范围带来的开销。

## 修复循环的工作原理

Karpenter 是配置和扩展 [EKS Auto Mode 节点](https://thenewstack.io/eks-auto-mode-kubernetes/) 的计算控制器。它已经拥有它启动的每个节点的生命周期，并将我们的 NodeConditions 用于修复是该所有权的自然延伸。没有单独的修复后端，没有 sidecar 控制器，没有 webhook 链。创建节点的系统就是替换节点的系统。

Karpenter 的 AWS 云提供商声明了修复策略：每个策略将条件类型与意味着“替换此节点”的状态配对。这些策略包括防止对短暂干扰做出反应的容忍窗口（toleration windows）：

* 加速硬件故障：10 分钟。这些是明确的（GPU 要么存在要么不存在），且任其运行成本高昂（在降级节点上的训练作业浪费了 GPU 小时）。
* 其他所有故障（内核、运行时、网络、存储、kubelet NotReady）：30 分钟。有足够的时间让短暂的网络抖动或临时的运行时故障自行解决。

流程如下：

1. 代理检测到致命故障并带有原因代码地将匹配条件翻转为 False。
2. Karpenter 的健康控制器看到此转换并开始计时。
3. 如果条件在窗口过期前清除，计时器会静默重置。节点从未被触碰。
4. 超过容忍窗口后，安全门会检查舰队健康状况。Karpenter 不会同时修复 NodePool 中超过 20% 的节点。如果相关事件（错误的 AMI 发布、控制平面抖动、区域性损坏）同时触发了多个节点的条件，系统将暂停。当 Amazon Application Recovery Controller 区域转换处于活动状态时，自动修复也会停止，因此不会将受损可用区之外的刻意流量移动误认为是发生了一群坏节点。
5. 在安全阈值内，Karpenter 会标记节点以阻止新的调度，优雅地排空正在运行的 pod（遵循 PodDisruptionBudgets），终止实例，并配置适合已置换工作负载的替换节点。

替换节点在启动时会有一个新的代理对其进行监控。过程中没有运维人员介入。在我们的测试中，从故障注入到替换节点运行工作负载的完整周期不到 12 分钟。检测在不到一秒内完成（关键 GPU 故障使用 DCGM 基于推送的策略通道，而非轮询）。然后是 10 分钟的容忍期，大约 90 秒用于启动替换节点并注册。

## 令我们惊讶的部分：检测和诊断不是同一个问题

自动修复处理的是常见情况：坏节点被替换，工作负载继续运行。但“为什么该节点会失败？”是另一个问题，我们最初试图在检测路径内回答这个问题。这是一个错误。

检测回答的是“此节点是否健康？”它以最小的开销持续运行，并且需要快速：产生结果需要 5 分钟的条件翻转意味着工作负载降级了 5 分钟。诊断回答的是“出了什么问题？”它需要收集详细的工件：完整的 journald 输出、containerd 状态、网络配置、dmesg、GPU 驱动程序日志。在我们的测试中，收集过程在大约 7 秒内完成，并生成一个压缩的日志包。将其嵌入到检测的关键路径中会减慢客户最关心的内容：系统的响应速度。

我们将它们构建为共享一个代理二进制文件的独立关注点。NodeDiagnostic CRD 允许你通过 kubectl 从任何节点请求完整的日志包，而无需 SSH。在 EKS Auto Mode 上，节点是按设计没有 shell 访问权限的 Amazon EC2 托管实例，这是 GPU 故障或其他节点级故障后进行调查的唯一方法。

体验只需要一个命令：

`kubectl ekslogs <node-name>`

插件创建一个 NodeDiagnostic 资源。目标节点上的代理通过 watch 检测到它，将系统状态收集到压缩的 tarball 中，并暂时存储它（可用 10 分钟）。然后插件通过 Kubelet 的节点日志查询 API（KEP-2258，在 Kubernetes 1.36 中已 GA）下载它。没有 SSH，没有安全组，没有密钥对。

这种分离意味着检测不会为了收集证据而减慢速度，诊断也不需要始终处于开启状态（节省节点资源），并且你可以诊断一个已被自动修复标记但尚未终止的节点。加速硬件故障的 10 分钟窗口为你提供了刚好足够的时间在节点消失之前[获取日志](https://thenewstack.io/observability-isnt-enough-its-time-to-federate-log-data/)。如果你对进一步改进感兴趣，请在 [EKS 公共路线图](https://github.com/aws/containers-roadmap/issues)上与我们联系。

## 如果你在运行 EKS，这意味着什么

在 [EKS Auto Mode](https://docs.aws.amazon.com/eks/latest/userguide/automode.html) 上，所有这些都是默认开启的。[Auto Mode 完全管理你的集群](https://thenewstack.io/how-amazon-eks-auto-mode-simplifies-kubernetes-cluster-management-part-1/)基础设施（计算、网络、存储、补丁和安全加固），因此你专注于应用程序，而不是集群运维。代理作为节点镜像中的 systemd 服务运行（而不是你管理的 DaemonSet），Karpenter 将其条件作为它已经拥有的计算生命周期的一部分来消费，而 `kubectl ekslogs` 为你提供了无需 SSH 的诊断访问权限。无需安装、配置或运维任何东西。对于 GPU 工作负载，这意味着你昂贵的加速器节点会在没有运维人员干预的情况下自动被监控、分类和替换。

在托管节点组或自管理的 Karpenter 上，你可以组装相同的循环：安装 EKS 插件形式的节点监控代理，并为每个节点组选择加入自动修复。架构是相同的，只是没有预先组装。

EKS 节点监控代理已在 [github.com/aws/eks-node-monitoring-agent](https://github.com/aws/eks-node-monitoring-agent) 以 Apache 2.0 协议开源。

我们在大规模运行时遇到的故障模式，以及由此产生的修复方案，都会回馈给任何使用它的人。如果你正在构建节点健康系统或正在运行我们的系统并遇到了边界情况，请加入我们一起构建！
