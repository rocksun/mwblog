文章对比了自托管AI智能体OpenClaw与谷歌云端托管的Gemini Spark。普通用户倾向于便捷的托管服务，但由于隐私和控制权，自托管在开发者中仍具持久生命力。

> 译自：[OpenClaw passed 300,000 GitHub stars. Then Google launched Spark.](https://thenewstack.io/gemini-spark-vs-openclaw/)
> 
> 作者：Janakiram MSV

OpenClaw 通过让全天候运行的智能体（agent）驻留在你触手可及的地方——搁板上的一台 Mac mini，耗电仅 7 瓦，在你熟睡时依然在运行——从而使其充满了个人专属感。Peter Steinberger 的开源项目在 4 月前就已经突破了 300,000 个 GitHub Star，成为 GitHub 上增长最快的仓库之一。它的全部魅力在于所有权：你的硬件、你的凭据、你的龙虾（lobster）。

在周二的 I/O 大会上，[谷歌推出了 Gemini Spark](https://thenewstack.io/google-ai-ultra-pricing/#:~:text=AI%20Ultra%20subscribers%20(at%20both%20the%20%24100%20and%20%24200%20tier)%2C%20will%20also%20be%20the%20first%20to%20get%20access%20to%20Gemini%20Spark%2C%20Google%E2%80%99s%20new%2024/7%20AI%20agent%2C%20and%20Gemini%20Omni%2C%20Google%E2%80%99s%20new%20multimodal%20model.)，它做出了截然相反的赌注。Spark 是一款基于 Gemini 3.5 Flash 构建的 24/7 全天候个人智能体，并连接到了谷歌的 Antigravity 智能体技术栈。它在 Google Cloud 的虚拟机后台运行。你永远看不到那台机器。谷歌计划让你能够直接向该智能体发送短信和电子邮件，因此即使你的笔记本电脑合上，它也能继续工作。

## 分歧在于智能体驻留在哪里，而非它能做什么

剥离品牌外衣，Spark 和 OpenClaw 做的工作大致相同。监控收件箱、起草状态更新、浏览网页、运行定期任务。尽管在实现的成熟度上有所不同，但两者都正在向 MCP（模型上下文协议）收敛以实现工具连接。两者都承诺提供一个能够真正做事、而不仅仅是回答问题的助手。

> 底层基质决定了谁掌握你的上下文、谁能看到你的凭据，以及谁可以在以后更改条款。

两者的区别在于底层基质。OpenClaw 运行在你购买的实体硬件上。而 Spark 运行在谷歌租给你且从未命名过的硬件上。这听起来像是一个部署细节。但实际上，这才是争论的核心。底层基质决定了谁掌握你的上下文、谁能看到你的凭据，以及谁可以在以后更改服务条款。

## 便利性通常会赢得这场斗争，谷歌深谙此道

自托管版本需要付出真正的劳动。购买 Mac mini、保持其处于唤醒状态、安装守护进程、配置 Tailscale，并在密钥过期时进行轮换。而回报则是控制权。取决于你如何接入模型和集成，你的凭据和工作流都可以保留在自己的掌控之中。但这种控制权并不等同于安全。一个配置错误且拥有 shell、浏览器和收件箱访问权限的本地智能体自身就是一种隐患，中国监管机构已经针对 OpenClaw 明确指出了这一风险。

Spark 则无需你付出任何努力。它已经内置在 Gmail、Docs 和 Sheets 中，无需手动连接，因为谷歌同时掌控着两端。这种开箱即用的触达能力是任何第三方智能体都无法复制的结构性优势。历史的规律已经相当明确。Dropbox 击败了家用 NAS。Gmail 击败了自建邮件服务器。对于普通用户来说，托管服务几乎总是能击败自托管，因为大多数人会为了省心而放弃控制权。

> OpenClaw 并没有输。它正在被归入那个规模更小、但粘性更强的另一半市场。

因此，个人智能体层分化为两派。一派是托管层，谷歌以及很快加入的 OpenAI 将拥有其运行环境和上下文；另一派是自托管层，适合那些希望将凭证保留在自己的硬件上、并愿意为此付出配置时间的开发者。OpenClaw 并没有输。它正在被归入那个规模更小、但用户粘性更强的另一半市场。

## 这里的隐私权交易与 Dropbox 当年的情况不同

在急于对这场竞争下结论之前，我们应该在这里放慢脚步思考一下。云存储之所以能赢，是因为你交出的东西是静态的。文件静静地存放在 Dropbox 文件夹里，没有人去阅读它们。而个人智能体则完全是另一回事。为了发挥作用，Spark 需要对你的 Gmail、Docs、Sheets、日历以及实时收件箱拥有广泛且长期的访问权限。它不仅存储你的上下文，还会阅读它并据此采取行动。

这就改变了交易的性质。向谷歌交付一个文件夹，与向谷歌交付一个能处理你的工作、人际关系和日历，甚至能代表你发送邮件的系统，这两者是完全不可同日而语的。真正令人担忧的并不是谷歌保留了你的数据，而是数据访问、保留以及用于训练下一个模型之间的模糊地带尚未得到解决。

自托管阵营如今虽然规模较小，但这并不是因为人们怀念运行自己服务器的旧时光。而是因为一种本能：如此私密的智能体应该只听命于你，并且运行在你可以随时拔掉电源的硬件上。这种本能无法普及到每个人。但也无需如此。它只需要吸引开发者和对隐私敏感的用户，这就足以构成一个稳固且持久的底线。

对于开发者而言，问题不在于哪个智能体更好，而在于你是否能安心地让谷歌掌握掌控你生活的智能体的钥匙。
