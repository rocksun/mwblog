本文探讨了编码智能体在长周期任务（超200步）中面临的性能衰减和错误累积问题。小米开源的MiMo Code通过优化终端交互框架，旨在提升长任务下的可靠性，并强调了基于实际交付成果的评估标准比单一排行榜更具参考价值。

> 译自：[Xiaomi's MiMo Code claims it beats Claude Code past 200 steps](https://thenewstack.io/coding-agent-endurance-gap/)
> 
> 作者：Janakiram MSV

一个能在午餐时间构建出原型应用的编码智能体，在生产环境重构任务中往往会在30步左右陷入停滞。它过早地锁定了一个假设，并不断修补错误的假设，导致微小的错误不断累积，最终导致整个运行过程崩溃。

这可能是一个小米 MiMo AI 团队即将成为历史的场景，因为该公司现在开源了 [MiMo Code](https://venturebeat.com/technology/xiaomis-new-open-source-agentic-ai-coding-harness-mimo-code-beats-claude-code-at-ultra-long-200-step-tasks)，这是一个终端原生的工具框架，小米声称其在超过200步的智能体任务中表现优于 Anthropic 的 Claude Code。该基准测试为自报数据，取自小米内部测试及对576名开发者的调查，因此它更像是一个参考标志，而非定论。

> “持久性差距定义了这种距离，即智能体在放弃任务前能够存活的步数。”

数字本身并不重要，重要的是小米选择竞争的维度。长周期可靠性——即在数百个相关联的步骤中保持任务的完整性——是编码智能体的新前线。该领域目前才刚刚学会如何衡量一个智能体在失去线索前能走多远。持久性差距定义了这种距离，即智能体在放弃任务前能够存活的步数。

## 长周期智能体在哪里会崩溃

让任何智能体从一个清晰的提示词开始构建一个小应用，它都能出色完成。将一个单一目标贯穿数百步的编辑、测试和修订过程，才是故障模式按时出现的地方。

长期运行中会出现三种反复出现的故障：

1. 假设过早固化，导致智能体不断修补错误的假设。
2. 第40步继承了第12步的错误，导致错误累积。
3. 运行到中途时，起初重要的上下文信息逐渐消失。

从业者几个月来一直在描述这种崩溃。一份来自 Ejentum 团队广为流传的 [报告](https://x.com/ejentum/status/2064330447752999088) 将故障点定在第30步左右，并指出假设锁定和错误累积是常见的罪魁祸首。试想一个没有检查点的长批处理作业，一旦崩溃，整个作业就必须从头开始，而不是从最后保存的状态恢复。

评估这种差距更清晰的方法是评价已交付的工作，而不是依赖演示。伯克利大学的一项新基准测试正是为了做到这一点。

## 深入伯克利的最后考试

加州大学伯克利分校 RDI 实验室的 Dawn Song 和博士后 Yiyou Sun 构建了 [Agents’ Last Exam](https://arxiv.org/abs/2606.05405)。该团队报告称，来自55个职业的超过250名行业专家参与了构建。该基准测试设计严格，旨在暴露智能体的不足之处，而非展示其优点。每一项任务都是专业人士已经交付的真实项目，并被转换为无需人工裁判的代码评分测试。智能体拥有图形界面和命令行权限。它以自己的方式解决任务，基准测试只对它留下的工件进行评分。

这一头条发现让任何兜售“智能体已准备好工作”的人感到尴尬。伯克利团队将 Codex 与 GPT-5.5 进行了配对，这是测试中最强大的配置。论文报告显示，即使在最简单的层级上，通过率也低于50%，而在最难的层级上则低于10%。大多数主流智能体（包括 Claude Code）在该难度下的通过率几乎为零。Song 总结得很直白：今天的智能体能处理专业任务中有意义的一小部分，而最困难的长周期任务仍然遥不可及。

伯克利的测试结果发布的那一周，正好赶上 Anthropic 将 [Fable 5](https://thenewstack.io/fable-5-developer-reactions/) 推向市场并伴随大量“已准备好工作”的营销。这次考试用数字而非承诺回应了那种营销。

考试的价值在于它评估的是最终成果而非演示。一个模型可以在编码排行榜上名列前茅，但在任务运行过长时却留下半成品，这正是基于代码评分的底线测试所要捕捉的差距。

## 工具框架层

目前编码智能体领域最活跃的工作集中在工具框架（harness）层，即负责维护状态、控制工作节奏并决定下一步执行路径的层级。三种方法展示了其发展方向。

Anthropic 报告称，其 [Claude Code 中的嵌套子智能体](https://x.com/_mauriciorubio/status/2064329181958443326) 的运行深度被限制在5层。前沿模型负责规划，而成本较低的子智能体执行工作并生成它们自己的辅助程序。中国人民大学的研究人员在 [Arbor](https://ruc-nlpir.github.io/Arbor/) 上采取了不同的路线。它将一个长寿命的协调器与短寿命的执行器以及一个持久化的假设树结合在一起，该假设树可以在中断后检查进度并恢复。小米的 MiMo Code 从开源侧解决了同样的问题，针对超过200步的运行调整了终端原生的工具框架。

这些方法并非可互换，且报告的证据在置信度上差异巨大。

| 方法 | 如何扩展运行 | 报告结果 | 需权衡的注意事项 |
| --- | --- | --- | --- |
| Claude Code 嵌套子智能体 | 前沿模型规划，廉价子智能体执行并生成自己的辅助程序 | 成本最优的多智能体运行，当时最受开发者关注的故事 | 深度限制在5层，长度带来的故障模式尚待证明 |
| Arbor, 中国人民大学 | 长寿命协调器，短寿命执行器，支持恢复的持久假设树 | MLE-Bench Lite 上 86.36% 的 Any-Medal，在六项任务中收益是 Codex 和 Claude Code 的 2.5 倍 | 自报数据：内部测试及 576 名开发者调查，无第三方运行验证 |
| 小米 MiMo Code | 针对200步以上任务调优的开源终端工具框架 | 声称在长周期编码任务中击败 Claude Code | 自报数据：内部测试及 576 名开发者调查，无第三方运行验证 |
| Kimi K2.7-Code, 月之暗面 | 为长周期智能体任务定位的模型，Cloudflare Workers AI 首日支持 | 首日基础设施支持信号显示了真实的市场需求 | 厂商定位，独立的长周期数据待定 |

每一个可信的答案都外化了状态，而 Arbor 最清晰地表达了这一点。它的设计将假设树（而非上下文窗口）视为必须长期存活的东西。要理解为什么这一点成立，我们需要看看多年前耐用的工作流引擎是如何解决同样问题的。任何系统无法保存检查点的内容，在故障后都必须重做。

## 为什么持久性差距很重要？

对于将智能体接入交付流水线的企业来说，持久性差距是一个采购问题，而不是一个好奇心问题。一个在生产环境中第30步失败的智能体不会主动公告。它会提交一个基于早期错误转向构建的看似合理的工件。成本将稍后作为返工、隐性缺陷和必须以最坏情况进行评估的审查显现出来。像伯克利这样的基准测试为买家提供了一个基于代码评分的底线，即智能体在需要人工介入前能走多远。

> “一个在生产环境中第30步失败的智能体不会主动公告。它会提交一个基于早期错误转向构建的看似合理的工件。”

> 团队应该询问候选工具框架如何在长周期运行中保持状态，它是否能从检查点恢复，以及其公布的数据将上限放在了哪里。

企业现在可以将持久性视为评估中的独立指标。这意味着要衡量智能体能完成任务的程度，而不是盲目相信编码排行榜的分数。团队应该询问候选工具框架如何在长周期运行中保持状态，它是否能从检查点恢复，以及其公布的数据将上限放在了哪里。一些供应商以模型基准测试作为宣传点，而对长周期行为保持沉默，回避了读者真正关心的问题。

忽视任何一方的结果都是错误的。小米的200步声明一旦经过独立的工具框架运行，可能会得到验证。伯克利的最高难度层级可能过分强调了那些团队根本不会交给无人值守智能体的任务。如果模型继续扩展其保持连贯性的跨度，部分差距会从模型端缩小，而工具框架也会获得一些喘息空间。但这一切都无法改变近期的现实，即智能体的最终输出在演示效果破裂之前就已经出现了问题。

## 下一步

对于跟踪工具领域的开发者来说，信号已经脱离了模型本身的头条。现在重点在于两个数字：智能体保持连贯性的时长，以及谁验证了它。MiMo Code、Arbor 和 Claude Code 的子智能体都是这一领域尚在起步阶段竞赛的早期参与者。

> “真正重要的排行榜是根据已交付工作进行评分的，而不是基于自报的胜绩……”

接下来需要观察的是，对“Agents’ Last Exam”的独立运行是否会确认或削弱目前堆积起来的持久性声明。另一个需要观察的是工具框架是否会在编码智能体买卖的地方变得更加稳健，这种转变已经在进行中。当基准测试出现第一个受到争议的排行榜时，买家首先查看的数字将是智能体在人工介入前能存活的步数。这个数字将供应商的承诺转变为了一次采购测试。

总之，真正重要的排行榜是根据已交付工作进行评分的，而不是基于自报的胜绩，这种严谨性将使每一个将智能体纳入交付流水线的团队受益。
