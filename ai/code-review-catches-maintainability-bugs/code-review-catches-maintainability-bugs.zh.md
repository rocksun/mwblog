软件代码审查流程是一种系统性的、由同行驱动的质量保证程序，旨在当开发人员提交拉取请求或合并请求时，对代码进行审查。

尽管糟糕的代码审查流程常因不可避免的延误，以及人们因琐碎细节造成瓶颈而受到诟病，但良好的代码审查可以及早发现Bug，培养指导关系，并被视为一种民主的分担责任的方式。

事实确实如此，但又不尽然。

资深软件工程师 [Mark Dominus](https://www.linkedin.com/in/mark-jason-dominus-93545b/) 告诉 *The New Stack*，尽管许多公司确实在进行代码审查，但他们并没有“明确表达审查的结果应该是什么”，这是一个问题。

“假设你是一名初级工程师，第一次被告知要进行代码审查。你的交付成果是什么？在我工作过的许多地方，交付成果都没有被明确说明，” Dominus 说。

在关于该主题的一篇 [Mastodon 帖子](https://mathstodon.xyz/@mjd/115096720350507897) 中，Dominus 写道：“任何依赖代码审查来寻找 Bug 的人都是生活在‘愚人天堂’中”，主要是因为通过检查代码来发现 Bug “通常是不可能的”。他强调了他的评论，称“代码审查的主要目的”是发现未来难以维护的代码。

## 通往糟糕代码审查结果的必然路径

他建议，在预期软件工程师轻松浏览 [变更集](https://en.wikipedia.org/wiki/Changeset) 并指出他们不喜欢的事物的情况下，这会导致糟糕的结果，即它鼓励人们试图强加自己关于应该如何做事的主观偏好，从而引发争论。

> “仅仅通过检查变更集来发现 Bug 是极其困难的，并且需要一定的运气。” – Mark Dominus。

“有时老板会给初级工程师多一点指导，例如：‘看看能不能发现任何 Bug，’这使初级工程师处于一个令人不快的境地，” Dominus 澄清道。“也许他们确实发现了一个 Bug，太好了！但如果没发现呢？[寻找 Bug](https://thenewstack.io/what-can-we-learn-from-historys-most-bizarre-software-bugs/)，仅仅通过检查变更集，是极其困难的，需要一定的运气。”

他建议采用一种不同的方法，即由软件团队的项目负责人主导代码审查流程，方式是说：“花两个小时看这个，写下任何你不理解的内容，如果你没有全部看完，记下你停在哪里。”

## 提供真正软件工程价值的途径

“现在，如果审查员没有时间查看所有内容，你学到了一些有价值的东西：变更集太大或太复杂，无法在两小时内理解。也许变更集应该拆分成两个或更多较小的提交，应该分别进行审查。无论审查员多么初级、缺乏经验或状态不佳，他们都能交付所要求的内容，而且他们交付的内容将具有真正的工程价值，” Dominus 解释道。

在这种情况下，这种真正的工程价值是一种“正向的负面结果”。团队现在已经识别出其他团队成员无法理解的代码，或者变更集对于当前的代码审查流程来说太笨重了……又或者两者皆有。

## 代码审查是一场戏，即将上映

高负载电子商务平台公司的 QA 工程师 [Mikhail Golikov](https://www.linkedin.com/in/michael-golikov-b19308263/) 在很大程度上同意这些观点，他告诉 *The New Stack*，如果一个团队依靠代码审查来查找 Bug，那么“代码审查就是一场戏”。

> “当一个团队将审查视为其 Bug 过滤器时，它在发布时会有一种温暖的感觉，但没有证据，然后在生产环境中才会发现问题，” – Mikhail Golikov。

“人类在[浏览差异文件](https://www.codecademy.com/article/what-is-diffing)时无法看到竞态条件或在高负载下变为负数的折扣。审查是为了那些你以后会讨厌维护的代码；测试则是为了那些现在就坏掉的代码，” Golikov 强调说。“审查员阅读干净的差异文件，看到合理的命名和整洁的函数，然后点击批准。这些都不能告诉你应用程序的关键部分是否出现故障，并向用户提供损坏的服务。”

同时也是 [Python 开源测试工具维护者](https://github.com/golikovichev) 的 Golikov 解释说，导致这些错误的 Bug 并不存在于开发人员可以阅读的代码中。相反，它们存在于代码在运行时所处的状态中。

## 不要带着温暖的感觉和没有证据的心态发布

“当一个团队将审查视为其 Bug 过滤器时，它在发布时会有一种温暖的感觉，但没有证据，然后在生产环境中才会发现问题，” Golikov 确认道。“审查是为了回答‘六个月后我会讨厌维护这段代码吗’以及‘它是否真的有效’（当针对真实输入运行测试时），而不是为了让一个人在下午 5 点浏览拉取请求。”

开发人员和供应商几乎一致认为，代码审查流程需要改变，特别是在人工智能时代，代理编码工具日益普及的今天。

[Hetz Ventures](https://www.hetz.vc/) 的执行合伙人 [Judah Taub](https://www.linkedin.com/in/judah-taub-3773247b/) 同意这里的叙述，并告诉 *The New Stack*，多年来，工程团队一直将代码审查视为抵御 Bug 的最后一道防线，但这从来都不是它们的真正强项。

> “工程师的角色继续远离实际代码，转向验证架构、意图和业务逻辑。未来，甚至这一点也可能被自动化，” Judah Taub。

“代码审查捕捉风格、架构、可读性和可维护性，” Taub 说。“测试捕捉 Bug。生产环境捕捉所有其他东西。最好的工程组织不会依赖另一个开发人员发现隐藏在数百行代码中的微妙边缘情况——他们构建系统，在拉取请求到达另一个人之前很久就自动验证正确性。代码审查是为了确保下一个工程师可以无缝地处理代码。”

## 代码审查的未来

随着 [AI 生成的代码](https://thenewstack.io/anthropic-launches-a-multi-agent-code-review-tool-for-claude-code/) 现在进入竞争，Taub 认为不可避免的新平衡将是人类直接审查的代码减少，而越来越多地审查 AI 对代码的审查。

“工程师的角色继续远离实际代码，转向验证架构、意图和业务逻辑。未来，甚至这一点也可能被自动化，” 他预测道。

我们当然可以从中提取关键趋势，并说软件工程团队如果目前还停留在九十年代，就确实需要走出来了。

云原生、平台工程、氛围编程以及向代理编码发展的共同革命（如果你愿意，还可以加上大数据、DevOps 和向企业开源标准化的趋势）已经改变了软件开发人员的工作方式，因此，他们审查刚刚发生的事情的方式也应该发生相应的改变。
