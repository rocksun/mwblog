Vespa AI 与 Voyage AI 推出非对称检索方案，通过共享向量空间模型，让企业能用顶级模型索引文档并配合本地微型模型免费处理查询，实现零成本查询嵌入与更强韧性。

> 译自：[Cut your AI search costs without sacrificing quality](https://thenewstack.io/cut-ai-search-costs/)
> 
> 作者：Bonnie Chase

## 驱动 AI 搜索账单的成本杀手

每一个运行 AI 驱动搜索的企业都面临着同样的隐形成本驱动因素：**查询嵌入 (query embeddings)**。文档只需嵌入一次，但查询则是为每个用户、每次搜索、每一秒钟**持续不断**地进行嵌入。在大规模应用中，这会迅速成为 AI 基础设施预算中最大的开支项目之一。

Vespa AI 和 Voyage AI 联手通过一种名为**非对称检索 (asymmetric retrieval)** 的技术解决了这个问题。你可以为文档使用市面上最好的嵌入模型（在索引时执行一次），然后使用微型的、[本地运行模型](https://thenewstack.io/how-to-run-deepseek-models-locally-on-a-windows-copilot-pc/)免费嵌入查询。Voyage AI 的 **voyage-4** 模型系列正是为此而生。这四款模型共享一个通用的向量空间，使得这种分离方案在无需重新索引或更改架构的情况下变得切实可行。

> “每一个运行 AI 驱动搜索的企业都面临着同样的隐形成本驱动因素：查询嵌入。”

**决策者核心结论：** 你的查询嵌入账单实际上降至为零，且搜索路径变得更具韧性，而这一切都无需更换现有的搜索基础设施。

## 问题所在：对称性成本高昂

传统方法对文档和查询使用相同的嵌入模型。这种方式虽然简单，但忽略了这两种操作在运作方式上的关键不对称性。

|  | **文档嵌入** | **查询嵌入** |
| --- | --- | --- |
| **频率** | 每个文档一次 | 每一个请求 |
| **延迟敏感度** | 无，没有用户在等待 | 在关键路径上，24/7 全天候运行 |
| **10K QPS 下的成本** | 分摊后可忽略不计 | 约 15,500 美元/月 |

在每秒 10,000 次查询（QPS）且查询长度约为 30 个 token 的情况下，你每月会产生大约 **7770 亿个 token**，这些全部通过外部 API 路由并产生实际成本。

## 解决方案：利用 Voyage AI + Vespa 实现非对称检索

Voyage AI 的 voyage-4 系列推出了四款模型（`voyage-4-large`、`voyage-4`、`voyage-4-lite` 和 `voyage-4-nano`），它们都在**共享向量空间**中生成嵌入。你可以用最强大的模型嵌入文档，用最小的模型进行查询，它们之间保持完全兼容。

Vespa 现在原生支持这种工作流，在其容器节点内本地运行 `voyage-4-nano`，无需 API 调用，没有速率限制，也没有额外成本。

## 工作原理

### 第 1 步：索引阶段：文档 → `voyage-4-large` (API)

使用 Voyage AI 的顶级模型为每个文档嵌入一次。结果具有最高的准确度，且没有延迟压力。成本在文档的生命周期内被完全分摊。

### 第 2 步：查询阶段：查询 → `voyage-4-nano` (本地)

使用运行在 Vespa 内部的微型模型嵌入每个用户查询。在 CPU 上运行时间仅为个位数毫秒。零外部 API 依赖。零成本。

[阅读完整技术博客](https://blog.vespa.ai/asymmetric-retrieval-spend-on-docs-queries-for-free/)。

### 业务影响一览

| **指标** | **对称检索（传统）** | **非对称检索 (Vespa + Voyage AI)** |
| --- | --- | --- |
| 10K QPS 下的查询嵌入成本 | ❌ 约 15,500 美元/月 | ✅ 0 美元/月 |
| 查询嵌入延迟 | ❌ API 往返 (10–80ms) | ✅ CPU 上 <5ms (本地) |
| 检索质量 (对比 OpenAI v3 Large) | 基准线 | ✅ +14.05% NDCG@10 |
| 关键路径上的 API 依赖 | ❌ 是，服务中断会影响搜索 | ✅ 否，完全自包含 |
| 升级查询模型需重新索引 | ❌ 需要 | ✅ 不需要 |
| 多层级文档质量 | ❌ 不支持 | ✅ 支持 |

## 为什么运维韧性至关重要

从查询路径中消除外部 API 不仅仅是一项成本优化，更是一个可靠性决策。

> “从查询路径中消除外部 API 不仅仅是一项成本优化，更是一个可靠性决策。”

| **风险** | **传统架构** | **非对称架构** |
| --- | --- | --- |
| **API 服务中断** | 搜索服务宕机 | 无影响，完全本地化 |
| **速率限制** | 流量高峰时请求被丢弃/延迟 | 无速率限制 |
| **扩展性** | 需要数天协商更高的 API 配额 | 数分钟内即可添加 Vespa 容器节点 |

通过非对称检索，查询路径是完全自包含的。无论第三方 API 状态如何，搜索都能正常工作。

## 进阶：旨在实现最高准确度的两阶段排序

Vespa 将此架构与两阶段排序策略相结合，在大规模场景下同时提供速度和精度。

Vespa 以两种形式存储文档向量：紧凑的二进制嵌入（内存占用减少 16 倍）用于快速的第一阶段检索；以及全精度 bfloat16（存储在磁盘上）用于准确的第二阶段重排序。结果是既拥有二进制级别的搜索速度，又具备全精度的准确性。

### 第一阶段：全量索引扫描

对二进制向量进行汉明距离计算。每秒约进行 10 亿次距离计算。在毫秒内从整个语料库中检索出前 2,000 个候选文档。

### 第二阶段：精准重排序

仅对前排候选文档进行 Bfloat16 点积计算。全精度向量从磁盘分页载入，用于对前 2,000 个结果进行计算。准确且计算量受限。

[二进制量化](https://thenewstack.io/why-vector-size-matters/)还减少了存储需求：一个 2,048 维的向量从 4,096 字节缩减至 **256 字节**，减少了 16 倍，而对最终排序质量的影响微乎其微。

## 专为企业规模设计

Vespa 将无状态容器节点（运行嵌入的地方）与内容集群（存储数据的地方）分离，因此查询嵌入能力和文档存储可以独立扩展。多租户部署可以在同一个索引中混合文档嵌入层级，为高端客户使用 `voyage-4-large`，为成本敏感层级使用 `voyage-4-lite`，同时所有租户共享同一个本地查询模型。

## 何时使用此架构

| **场景** | **建议** |
| --- | --- |
| 高 QPS (>1,000 次查询/秒) | ✅ 非常适合，节省成本随规模线性增长 |
| 海量文档语料库 | ✅ 非常适合，文档嵌入成本被分摊 |
| 延迟敏感型应用 | ✅ 非常适合，本地推理消除了网络往返 |
| 多租户平台 | ✅ 非常适合，可按层级控制质量/成本 |
| 低业务量 (<100 QPS)，对延迟容忍度高 | 在此规模下，[单模型可能更简单](https://thenewstack.io/ai-unified-data-foundation/) |
| 追求极致质量，不考虑成本 | 仍然可以选择对两者都使用对称的 voyage-4-large |

## 来自两家 AI 搜索领导者的联合方案

[**Vespa AI**](https://vespa.ai/) 提供了业界领先的开源搜索和推荐平台，为 Spotify、Yahoo 和 Perplexity 的 AI 应用提供支持。

[**Voyage AI**](https://www.voyageai.com/) 提供最前沿的嵌入模型。在撰写本博文时，`voyage-4-Large` 在涵盖 29 个检索数据集的 RTEB 基准测试中排名第一，性能优于 Gemini Embedding 001 (+3.87%)、Cohere Embed v4 (+8.20%) 以及 OpenAI v3 Large (+14.05%)。
