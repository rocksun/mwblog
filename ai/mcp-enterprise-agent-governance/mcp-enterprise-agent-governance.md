文章指出MCP协议虽在个人项目中有过度设计之嫌，但在企业级Agent应用中，它通过提供结构化的权限管理与审计凭证，解决了直接API调用难以满足的安全治理需求，是实现大规模AI部署的关键。

> 译自：[The MCP debate has a context problem](https://thenewstack.io/mcp-enterprise-agent-governance/)
> 
> 作者：Amanda Rueda

每隔几个月，就会有一项新技术被过早地“写下讣告”。在许多开发者圈子里，MCP 的讣告已经写好了。人们认为它带来了过多的开销和复杂性。对于某些小规模的用例，怀疑论者确实有道理，但当你的项目从个人项目扩展到企业试图构建的那种代理（Agentic）系统时，他们的论点就站不住脚了。

## 怀疑论者为何有道理

当你是一名独立用户且信任是隐式存在时，使用 MCP 服务器确实会增加不必要的复杂性。对于连接本地 AI 助手、构建个人自动化或在自己的虚拟机上运行隔离脚本而言，直接进行 API 调用是更简单且通常更好的选择。在那种语境下，开发者的观点没有错。

## 论点为何站不住脚

企业级代理系统并非单一开发者、单一信任边界的架构。代理跨越团队、系统和组织运行，通常代表那些甚至不知道代理参与其中的用户行事。在这种环境中，直接 API 调用开始引发无法通过简单的“允许列表”（allowlist）可靠解决的问题。

> “每隔几个月，就会有一项新技术被过早地“写下讣告”。在许多开发者圈子里，MCP 的讣告已经写好了。”

凭证委派（Credential delegation）是首批摩擦点之一。构建一个代表企业客户运行代理的平台，会迅速引发关于代理身份、权限继承以及它是否以服务账户身份运行等问题。虽然没有普适的正确答案，但选择错误却会带来后果。

拥有过度访问权限的代理会造成 [审计与合规风险](https://thenewstack.io/agentic-cicd-audit-compliance-gap/)。虽然访问权限不足会导致代理悄无声息地失败，但它会随着时间推移侵蚀平台信任。受监管行业的企业买家在进行正式安全审查之前，就会提出这一担忧。

在受监管的环境中，“AI 做的”并不是一个充分的审计条目。团队需要回答：代理做了什么、代表谁做的，以及在什么授权下做的。这些信息需要通过设计嵌入到系统中，而不是在事故发生后从日志中重建。直接的 API 集成无法提供这一点，但实现良好的 MCP 设置却可以。它在工具调用层面呈现必要的上下文，捕获每次操作是由哪个服务器、哪些凭证以及在什么范围内授权的。

## 允许列表并非治理

当有人问代理是否可以访问它不该触碰的东西时，直觉反应是使用“允许列表”。定义许可工具，在执行前进行核对，完成任务，然后结束。这种方法让人感觉像是治理，因为它类似于一种策略。然而，策略的强度取决于执行它的系统。在实践中，允许列表会因配置错误、意料之外的代理行为以及最初编写列表时未考虑到的边界情况而失效。

> “在受监管的环境中，‘AI 做的’并不是一个充分的审计条目。”

更持久的选择是“结构化最小权限”。当 [代理的 MCP 连接](https://thenewstack.io/mcp-security-risks-multiply-with-each-new-agent-connection/) 仅暴露其特定任务所需的工具时，代理就无法超出该范围。对于受监管环境中的安全团队来说，“它在结构上无法超出其范围”与“它遵守规则”是意义迥异的保证。

## 仍需改进之处

MCP 在两个领域仍存在摩擦。

> “对于受监管环境中的安全团队来说，‘它在结构上无法超出其范围’与“它遵守规则”是意义迥异的保证。”

基于能力的服务器配置仍然过于手动。当正确的配置需要深厚的基础设施知识时，管理员往往会跳过它，转而采用广泛连接加“允许列表”作为防线。在行业内能够让非专业人士轻松进行配置之前，其更广泛的普及将会停滞。

管理 MCP 连接的操作层需要对安全和平台团队开放，而不仅仅是针对最初构建集成的开发者。管理员应该能够查看环境中的每一个活动连接，了解每个连接暴露了什么，并识别是谁在使用它——所有这些都无需阅读配置文件或费力查找最初的实施者。

对于组织来说，在大规模部署 [代理](https://thenewstack.io/cloud-native-and-open-source-help-scale-agentic-ai-workflows/) 时，这些要求是必不可少的。

## 企业规模真正需要什么

暴露薄弱治理模型的条件总会通过事故显现出来。以企业规模运行的自主代理会比你更早发现漏洞。如果现在的诚实回答仍然是“我们有一个允许列表”，那么下一步就很明确了：审计你的代理可以访问什么，将其与它们实际所需进行对照，并在漏洞演变成事故之前，建立结构化的控制措施。
