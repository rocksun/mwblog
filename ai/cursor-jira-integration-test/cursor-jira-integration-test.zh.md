Cursor 上周推出了其 Jira 集成。该集成的宣传非常简单：在 Jira 中分配一个工单，Cursor 就会处理它。无需切换上下文，无需复制或粘贴。

> 欢迎来到全新的氛围编程时代，在这里，工单就是提示词。

欢迎来到全新的氛围编程时代，在这里，工单就是提示词。这听起来容易得令人怀疑。在实际操作中也这么容易吗？

我进行了四次测试，其中两个工单描述清晰，另外两个则含糊不清。我让 Cursor 修复两个 Bug 并编写两个新功能。我的目标是确定工单的质量是否会影响 Cursor 的输出结果。

## 集成

让我们来谈谈付费墙。Cursor/Jira 集成无法在免费工具中使用。当我尝试升级 my Jira 账户时，我发现它在注册时自动赠送了一个月的免费试用。无需信用卡。感谢 Jira。Cursor 则是另一回事。Cursor Teams（使用该集成所需的方案）没有免费试用。完成这次测试花了我 40 多美元（只要我记得在接下来的 30 天内取消 Cursor 订阅）。

结账之后，我准备开始集成。我惊讶地发现，在 Jira/Atlassian 市场中提供的 Cursor 集成目前还不怎么受欢迎（也许是暂时的？）。当我于东部时间 5 月 28 日星期三下午 7 点左右下载它时，仅有 548 次安装，且没有任何评论。我很好奇这是因为市场还不知道这个功能，还是大家对此不感兴趣。

在 Jira 中点击几下，再在 Cursor 中点击几下，我就全部设置好了。设置起来非常、非常简单。

## 执行

它有效吗？是的。效果出奇地好。我在测试过程中只遇到了一点小麻烦，不过我愿意承认这可能是用户操作失误。在写好工单后，我找不到“指派”给 Cursor 或向其发送评论的方法。于是我通过 Cursor 执行这些工单，输入的提示词是：“can you read and fix this ticket in my Jira account: ticket title”（你能读取并修复我 Jira 账户中的这个工单吗：工单标题）。

我在 HTTPie 开源代码库的两份独立克隆版本中运行了四个工单。克隆 A 测试了编写清晰的 Bug 修复和新增功能需求工单。克隆 B 测试了与克隆 A 类似的任务，但使用的是编写含糊的工单。我会在本文底部发布所有工单的副本以供参考。

### 克隆 A（清晰编写的工单）

**Bug 修复**

*当使用恰好一个自定义请求头发送 POST 请求时，详细输出中缺少 Content-Type: application/json（[问题 1856](https://github.com/httpie/cli/issues/1640)）。* 该工单为 Cursor 提供了受影响的文件、重现步骤以及明确的验收标准。Cursor 修复了该 Bug，添加了回归测试，并在 Jira 上发表了详细评论，引用了上游 GitHub 的问题编号。我从未在工单中提及此事。Cursor 是通过自行交叉引用代码库和问题历史记录找到该关联的。

Cursor 在工单上发表了评论（评论显示为来自我，而不是 Cursor），然后关闭了工单。它对所有工单都执行了同样的操作，在此说明一下，后续章节就不再赘述了。

**功能需求**

*添加一个 –no-elapsed-time 标志，以在不影响请求头或请求体的情况下，从 HTTPie 输出中抑制耗时行。* Cursor 实现了该标志，更新了帮助文本，并在 `tests/test_meta.py` 中添加了测试。它逐一明确勾选了 Jira 工单中的每一项验收标准。

### 克隆 B（模糊编写的工单）

**Bug 修复**

相同的 Bug。这一次，Cursor 修复它的速度比在克隆 A 上还要快。我有点惊讶，因为这个工单要糟糕得多。Cursor 在 Jira 工单上留下的评论引用了另一个 Bug 修复请求中那个编写良好的工单。Cursor 利用前一个工单的上下文填补了模糊工单中的空白。太酷了。结果棒极了，但这污染了测试，因此我需要寻找一个新 Bug 让它修复，然后才能充分测试模糊工单。

我测试的下一个是[问题 1642](https://github.com/httpie/cli/issues/1642)。*当使用 –download 检索服务器响应为 Content-Encoding: gzip 的文件时，HTTPie 会错误地报告“下载未完成”，因为它将压缩后的 Content-Length 与解压后的字节数进行了比较。* Cursor 正确地修复了它。它在 `downloads.py` 中找到了根本原因，在存在 Content-Encoding 时将 `total_size` 设置为 `None`，添加了两个回归测试，并再次在 Jira 上发布了详细评论，引用了上游 GitHub 问题编号。这一次项目中没有先前相关的工单可以依靠。Cursor 仅凭两句模糊的描述就诊断并修复了 Bug。哇。

**功能需求**

最后一个工单请求的是一个全新的功能，因此 Cursor 无法利用任何上下文来完成工作。该工单要求 Cursor “无需每次都手动重定向输出即可保存响应”。它没有提供任何验收标准、文件引用或实现指导。

Cursor 构建了一个完整的 `–save` 标志，并配备了用于自定义输出目录的伴生 `–save-dir` 标志、源自 `Content-Disposition` 和 URL 的自动生成文件名、防冲突的数字后缀、与 `–download` 和 `–output` 的兼容性检查，并在新的 `tests/test_save.py` 文件中通过了六项测试。这些都是独立的工程决策，而且它们起作用了。再次惊叹，哇。

10/10 分。无可挑剔。

## 这对我们其他人意味着什么？

软件开发格局正在不断改变。是的，我使用的是一个著名的仓库，而且需求也相当基础，但这依然显得非同寻常。我可以很放心地说，这个编程智能体（coding agent）可以阅读你的 Jira 工单，修复它们所描述的问题，并通过评论进行汇报。

> 我毫不介意让 Cursor 通过 Jira 工单来“氛围编程”出一个原型或个人软件。我会在生产环境的软件中信任它吗？目前还不会。

模糊工单的测试结果是本次测试中最有趣的部分。我原本预计 Cursor 能够成功关闭记录清晰的工单。我也曾预计它在处理编写模糊的工单时会陷入死循环。但我错了。Cursor 在几乎没有任何信息的情况下处理了一个功能需求，并产出了一个完整的、经过测试的实现，而且做出了合理的架构决策。

我毫不介意让 Cursor 通过 Jira 工单来“氛围编程”出一个原型或个人软件。我会在生产环境的软件中信任它吗？目前还不会。我是在一个有据可查 Bug 的著名开源仓库上进行的测试。生产环境的代码更混乱、文档更少，而且修复错误的代价要高得多。

工单：

### **清晰的 Bug（GitHub 问题 #1640）**

**标题**：单个自定义请求头时详细输出中缺少 Content-Type 请求头

**描述：**

发送带有一个自定义请求头和 -v 标志的 POST 请求时，详细输出中缺少 Content-Type: application/json。添加第二个自定义请求头则能正确显示。

**重现步骤**：

1. `https post pie.dev/post -v ‘header1: xyz’ x=1` —— 缺少 Content-Type
2. `https post pie.dev/post -v ‘header1: xyz’ ‘header2: abc’ x=1` —— 出现 Content-Type

**预期结果**：无论请求头数量多少，Content-Type: application/json 始终显示在详细输出中。

**验收标准**：

* 修复不一致问题
* 添加回归测试

### **清晰的功能**

**标题**：添加从输出中抑制耗时的标志

**描述**：

HTTPie 目前在每个响应的末尾显示耗时。无法在不抑制其他有用信息的情况下抑制此信息。

**建议的解决方案**：

添加一个 –no-elapsed-time 标志，该标志仅从输出中抑制耗时行，同时保持所有其他响应数据完好无损。

**验收标准**：

* –no-elapsed-time 标志抑制了耗时行
* 所有其他响应头和响应体输出均不受影响
* 该标志适用于现有的输出标志（如 -v 和 -b）
* 帮助文本已更新
* 已添加测试

### **模糊的 Bug（与 #1640 相同的 Bug，描述糟糕）**

**标题**：Content-Type 请求头有时缺失

**描述**：

发送 POST 请求时，Content-Type 请求头并不总是出现在响应中。不确定具体是什么时候发生的，但这不一致。似乎是个 Bug。

### **模糊的 Bug 2（GitHub 问题 #1642）**

**标题**：在某些服务器上下载中断

**描述**：

有时当我下载一个文件时，即使文件下载得很好，它也显示下载未完成。不知道为什么，在某些特定的服务器上会发生，而在其他服务器上不会。

### **模糊的功能**

**标题**：添加一种自动保存响应的方法

**描述**：

HTTPie 应该有一个选项可以自动保存响应，而无需每次都手动重定向输出。这对于 API测试非常有用。
