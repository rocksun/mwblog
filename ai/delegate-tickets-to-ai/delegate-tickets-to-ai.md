本文探讨了如何通过构建基于背景信息、护栏和可观测性的自动化工作流，将软件交付任务从人工操作转向AI代理。作者详细介绍了从计划、评审到开发、预览及部署的五个阶段，旨在通过代理化交付提升工程效率并实现工作透明化。

> 译自：[How to delegate 40% of tickets to AI](https://thenewstack.io/delegate-tickets-to-ai/)
> 
> 作者：Zohar Einy

AI在编程方面已经超越了我们。

但它在几乎所有其他方面也表现得更好、更快：规划、质量保证（QA）、以及使用软件开发生命周期（SDLC）中的所有工具。

那么，为什么我们不让代理（agents）全权处理功能开发和Bug修复，从票据（ticket）到生产环境全程负责呢？

我们应该做的仅仅是决定**要构建什么**，然后让代理做它们最擅长的事情。

但它们做不到。

因为如今每个人的SDLC都是手动的。

审批、审查、交接。所有这些让我们感到安心的事情，大多是不必要的。

> “如今每个人的SDLC都是手动的。审批、审查、交接。对于代理来说，这一切看起来就像是在演戏，因为它们只想快速推进工作。”

对于代理来说，这一切看起来就像是在演戏，因为它们只想快速推进工作。

因此，我们需要重新设计软件交付流程，以**服务代理优先**。

这意味着必须具备三个要素：上下文（context）、护栏（guardrails）和可观测性（visibility）。

1. **上下文**，这样代理从工作的完整图景开始，而不仅仅是一个票据标题。
2. **护栏**，这样它知道什么是范围内的，什么应该避开，以及何时停下来询问。
3. **可观测性**，这样工程师可以看到发生了什么，而不必整天盯着代理日志。

让我们构建基础设施来支持这一点，从票据落地的那一刻到它交付的那一刻。

## 我们正在构建什么

在进入细节之前，这是整个流程的轮廓。

它分为五个阶段，在代理工作和人工关卡之间交替进行。

**第一阶段：规划** —— 将原始票据转化为代理真正可以处理的内容，包含丰富的上下文、详细的产品需求文档（PRD）和全面的技术规范。

**第二阶段：审查中** —— 记分卡（scorecard）验证工作是否安全且范围明确，足以交给代理。产品经理（PM）签署PRD，工程师签署技术规范。然后将其交给编码代理。

**第三阶段：开发中** —— [Cursor](https://www.cursor.com/) 或 [Claude Code agent](https://thenewstack.io/claude-code-agent-view/) 处理票据，并在代理工作时保持实体状态更新。

**第四阶段：预览** —— 一旦合并请求（PR）打开并在云环境中运行，工程师和PM可以进行预览。然后，他们可以将其移交给部署流程。

**第五阶段：部署** —— 对当前的事故和冻结窗口进行实时检查，生成发布说明，并进行部署。

![展示从规划到部署五个阶段的图表。](https://cdn.thenewstack.io/media/2026/06/ad06b48c-image1-1024x485.png)

本文的其余部分将逐步介绍每个阶段，首先从在任何工作开始前你需要准备的东西开始。

## 先决条件：你的上下文湖（Context Lake）

为了让代理发挥最佳工作水平，[它们需要最好的上下文](https://thenewstack.io/better-context-will-always-beat-a-better-model/)——一个你的工程系统的实时模型，每个工作流和代理都可以从中读取信息。

具体到本指南，你需要将三件事放入它们的上下文中：

* 服务，包括层级（T1/T2/T3）、所属团队、仓库链接
* 通过 Port 的原生集成同步的 GitHub 数据：仓库、打开的 PR、最近的提交、CODEOWNERS
* 来自 Jira、Linear 或 GitHub Issues 的工作项，链接到它们所涉及的服务

我们即将构建的工作流还可以根据需要通过 MCP 获取数据。

例如，如果代理想要编写准确的技术规范，它们可能想要阅读 Notion 或 Confluence 中过去的 ADR、运行手册或过去的规范。

为了改进 PRD，我们将连接到真实的客户问题，因此你可能还需要为 Zendesk 或 Intercom 连接 MCP。

## 第一阶段：规划

整个工作流是在创建票据时触发的。它可以在 Jira、Linear 中，或者直接在 Port 中。

![规划阶段的工作流图，在创建票据时触发。](https://cdn.thenewstack.io/media/2026/06/230c6998-image5-1024x310.png)

我们建议创建一个通用的蓝图，如 work_item，以便每一步都可以从这些字段读取和写入：

```

service          →   链接到目录中的服务实体
team             →    来自目录所有权
tier             →       T1 / T2 / T3
blast_radius     → 低 / 中 / 高（已计算）
open_prs         → 针对关联仓库的计数
active_incidents → 来自事故追踪器的布尔值
prd              → Markdown 格式，由工作流编写
tech_spec        → Markdown 格式，由工作流编写
stage            → incoming / enriched / prd_draft / prd_ready / spec_draft / spec_ready / ready

```

### 第 1 步：使用上下文湖中的详细信息丰富票据

工作流查询上下文湖以解析工作项中缺少的任何详细信息，例如其组件或标签。它还可以从目录中添加服务层级、所有者和仓库，从依赖关系图中计算爆炸半径，获取打开的 PR，并检查是否存在活跃事故。所有内容都会写入实体，票据进入下一个阶段。

### 第 2 步：根据 PRD 技能起草 PRD

Port AI 针对票据、已丰富实体以及知识库中类似的 PRD 运行 PRD 技能。草稿落入实体中；阶段变为 prd_draft；PM 收到 Slack 提醒。

你编写的技能定义了结构。

### 第 3 步：暂停以进行 PM 审查

如果初稿在结构上是完整的，但在证据上不够充分。PM 可以从实体打开 Port AI 聊天并质疑其中的差距：

*“在过去 6 个月里，有多少客户询问过这个问题？”*

*“这会影响哪些服务？最近有哪些被触及？”*

*“我们为什么不应该构建这个？”*

Port AI 随后会根据此对话更新 PRD。

一旦他们对 PRD 感到满意，PM 会将其标记为就绪（ready），然后它会继续前进。

![PM 审查暂停前各阶段的工作流图。](https://cdn.thenewstack.io/media/2026/06/ca8ec4c9-image6-1024x330.png)

### 第 4 步：根据技术规范技能起草技术规范

模式与之前相同，但输入不同。Port AI 针对 PRD、目录或 MCP 中的代码库数据以及知识库中的 ADR 运行技术规范技能。草稿落入实体，工程师收到提醒。

![工程师审查前各阶段的工作流图。](https://cdn.thenewstack.io/media/2026/06/ef632311-image3-1024x323.png)

### 第 5 步：暂停以进行工程师审查

工程师随后可以打开 Port 聊天并迭代技术规范：

*“过去 30 天这个区域有什么变化？谁在那里活跃？”*

*“是否有我们需要遵循的现有模式？有什么是我们正在重复的吗？”*

*“可能会出什么问题？最近哪些文件导致了事故？”*

根据聊天内容，技术规范会得到更新。

一旦工程师将其标记为就绪，它就会进入下一个阶段。

![工程师审查阶段 Port 界面的截图。](https://cdn.thenewstack.io/media/2026/06/39167af1-image4-1024x658.png)

## 第二阶段：审查中

在票据可以移交给代理之前，需要发生两件事：它必须清除自动化记分卡，并且工程师必须实际上从看板上将其捡起。

### 记分卡检查

记分卡是你的治理层（它也可以作为工作流中的一个条件）。它针对离开第一阶段的每个工作项提出两个问题：

1. 它的范围是否足以让代理完成？
2. 现在自动化它是否安全？

如果任何规则失败，实体会被标记为已阻塞，并附带原因。

如果一切通过，则移至就绪状态并继续。

一套合理的初始规则：

| **规则** | **失败条件** | **原因** |
| --- | --- | --- |
| 服务层级 | T1 | T1 服务，需要人工交付 |
| 爆炸半径 | 高或未评估 | 没有范围审查太冒险，无法自动化 |
| 活跃事故 | 服务上有任何打开的 | 服务不稳定时不进行更改 |
| 优先级 | P0 / P1 | 紧急工作直接交给工程师 |
| 验收标准 | 为空 | PRD 不完整，无法验证输出 |

### **看板仪表板**

代理工作流中很快就会出现的一个问题是[工作变得不可见](https://thenewstack.io/agentic-cicd-audit-compliance-gap/)。你委派了某事，代理运行，除非你盯着代理的日志，否则你不知道它在哪里。乘以十个并行代理，你就失去了对工作的追踪。

> “代理工作流中很快就会出现的一个问题是工作变得不可见。乘以十个并行代理，你就失去了对工作的追踪。”

解决方案是 Port 中的一个仪表板，将所有工作项放在一个页面上，按管道阶段分组。

我决定在一个自定义小部件中显示工作，该小部件根据工作项实体呈现一个看板，列的运行状态为：传入 → 规划中 → 审查中 → 准备委派 → 开发中 → 预览 → 已部署。

![Port 看板仪表板的截图。](https://cdn.thenewstack.io/media/2026/06/435c1f76-image2-1024x617.png)

## 第三阶段：开发中

此时，你的工作项或票据应该已经丰富了充足的上下文、PRD、技术规范，并获得了 PM 和工程师的批准。

现在是时候将其交给编码代理来实现。

### 让我们构建一个委派动作

在 Port 中构建一个名为“委派给 Claude Code/Cursor”的自助服务动作。它位于工作项实体上，并显示为看板卡片上的按钮。当你点击它时，该动作会根据工作项构建一个提示（prompt）并启动代理。

重点是编码代理从工作的完整图景开始：构建什么，如何构建，哪些文件在范围内，以及已经做出的架构决策。它在工作时仍然可以引用 Port，但最好预先拥有尽可能多的信息。

在代理运行时保持看板的诚实。还记得第二阶段的“十个并行代理，你就失去了剧情”的问题吗？这就是我们解决它的地方。

3 个 Port 自动化监听 GitHub 事件并实时更新实体：

1. PR 已打开 → 将 PR 链接写入实体，将阶段移至已委派（delegated）
2. PR 已合并 → 将阶段设置为完成，记录时间戳
3. CI 失败 → 标记实体并将其移至 needs_attention

工程师可以看到哪些票据有活跃的 PR，哪些卡在 CI 中，哪些已完成，而无需打开代理日志或 GitHub 通知。

如果你希望代理诊断并重试 CI 失败而不是仅仅标记它们，那是[自动修复 CI 文章](https://newsletter.port.io/p/how-to-build-self-healing-ci-pipelines)中涵盖的另一种模式。

一旦代理的 PR 已打开并准备好供人类查看，工作就会进入第四阶段。

## 第四阶段：预览

一旦代理的 PR 打开，工作就进入预览阶段。

我不会在这里深入讨论，但会给你一些你可以添加到 Port 的想法，每一个都有独立的用途。

选择能解决你当前瓶颈的内容。

### AI 作为法官

在人工审查之前，运行一个 Port AI 检查，将差异（diff）与技术规范进行比较，并标记任何偏差，例如对“禁止触摸”文件的修改或未达到的验收标准。

### 启动预览环境

添加一个自助服务动作或自动化，为 PR 分支配置云环境并在 PR 上发布链接。

### 查看你的 PR 队列

为每位工程师构建一个个人仪表板，显示分配给他们的每个 PR、检查它的预览环境以及它挂起的时间。

### 提醒审查者

在工作项实体上添加一个自动化或工作流，用 PR 链接和工作项上下文通过 Slack 提醒分配的审查者。

## 第五阶段：部署

当预览阶段的审查者对代理的工作感到满意时，触发此阶段。它可以通过状态更改或自助服务动作来实现。

在任何东西发布之前，工作流会针对当前状态运行实时检查。

这是大致的流程：

* 针对实时状态的关卡检查：
  + 爆炸半径内的任何服务是否有活跃事故？查询 PagerDuty。如果有，则暂停
  + 是否有打开的冻结窗口，或已经有其他部署在进行中？暂停
  + 爆炸半径高？在实体上需要明确的工程师批准
* 部署。工作通过 Port 中的自助服务动作进行部署
* 生成发布说明。Port AI 从实体中构建它们：构建了什么（来自 PRD），更改了什么（来自 PR diff），测试了什么。（可选，你可以将其推送到你的发布说明系统）

如果出现问题，回滚是一键式的（为你或代理）。这是实体上的一个动作，预加载了部署参考，因此你不会浪费任何时间。

如果你想更进一步，你甚至可以在部署开始之前，通过自动对变更量、爆炸半径和发布策略进行评分，来运行全面的发布风险评估。这是[发布风险评估](https://newsletter.port.io/p/how-to-build-a-release-risk-assessment-agent)指南中涵盖的另一种模式。

## 衡量工作流动的速度

在投资像这样的工作流时，你需要看看它运行得有多好。因此，让我们衡量交付周期：从票据创建到生产环境需要多长时间。

此工作流中的每个阶段都会写回工作项：阶段转换、谁采取了行动、时间，因此数据已经在票据中了。

所以你最终得到的是：

**每个服务的交付周期：** 平均票据从“准备好委派”到部署需要多长时间，按服务分解。告诉你哪些服务是真正自主的，哪些仍然在人类身上存在瓶颈。

**每个阶段花费的时间：** 工作实际停留的地方。如果所有东西都在“准备委派”处堆积，说明你的记分卡太严格了。如果它在“预览”处堆积，说明审查者超负荷了。

**委派率：** 实际上委派给代理的工作项百分比，与手动处理的相比。这是告诉你系统是否在使用的单一数字。

**人工干预点：** 哪些阶段需要有人介入。此工作流中的积压点表明代理可能需要更多的上下文、更好的护栏或不同的记分卡规则。

这些数字告诉你的事情是周期时间永远无法做到的：你的团队在交付过程中的瓶颈究竟在哪里，哪些步骤在节省时间，哪些在增加摩擦。

![Port 上 ATR 交付周期和 ROI 界面的截图。](https://cdn.thenewstack.io/media/2026/06/be3055c0-image7-1024x664.png)

这就是完整的工作流。我们从一张两行的票据开始，最终完成了受监控的生产部署。我们在每个需要的关卡上都有人类参与。

如果你构建任何代理化工作流，我希望你构建这一个，以了解代理工程的全部内容。
