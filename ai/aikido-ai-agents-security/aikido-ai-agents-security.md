AI编程智能体自主安装软件包带来严重的安全漏洞与“责任真空”。Aikido等安全公司推出防护工具，旨在通过实时监控和策略执行来应对日益扩大的软件供应链风险。

> 译自：["There is no accountability": AI coding agents are installing packages no one owns](https://thenewstack.io/aikido-ai-agents-security/)
> 
> 作者：Darryl K. Taft

“无人负责。”

这是 [Aikido Security](https://www.aikido.dev/) 的联合创始人、首席技术官兼首席执行官 [Willem Delbare](https://www.linkedin.com/in/willemdelbare/) 向 *The New Stack* 描述的情景：AI 智能体自主安装了某个软件包，但没有任何人决定应该由谁来为此负责。

随着企业内部各个部门（市场、销售、产品等）都在使用 AI，这让企业暴露在各种各样的攻击之下。

Delbare 表示，在目前的大多数公司中，还没有人做出决定，也没有人承担这一风险。这产生了一个安全漏洞，使得攻击有机可乘。

Delbare 表示，他的 Aikido Security 正试图填补这一空白。随着 Claude Code、GitHub Copilot 和 Cursor 等 AI 编程智能体越来越多地自主拉取软件包、添加依赖项并安装工具，安全团队正处于两眼一抹黑的状态。

上个月，Aikido 推出了 [Aikido Endpoint](https://www.aikido.dev/protect/device-protection)，该产品在每次安装前都会对软件包、插件、IDE 以及浏览器扩展进行检查，并在恶意软件被下载前自动对其进行阻断。

Endpoint 通过向安全团队提供实时监控和策略执行，使企业能够安全且规模化地采用 AI 原生软件开发工作流，同时让开发人员能够灵活、安全地使用软件包、MCP、扩展、AI 模型和 AI 智能体，以满足现代开发的需求。

今年 3 月，该公司还推出了 [Aikido Infinite](https://thenewstack.io/aikido-self-securing-software/)，这是一个持续的 AI 渗透测试平台，旨在使软件实现自我保护。它旨在通过在整个软件开发生命周期中提供持续的安全验证，来解决传统手动渗透测试的局限性。

Aikido 在这个市场上并不孤单。Socket 刚刚以 10 亿美元估值完成 6000 万美元 C 轮融资，专注于实时检测和阻断恶意开源软件包。该公司声称在 6 分钟内就识别出了广泛使用的 Axios JavaScript 软件包中的一个恶意依赖项，并帮助企业在将其推向生产环境之前进行了阻断。与此同时，Endor Labs 于 2026 年 3 月推出了 AURI，这是一个 Skills 插件、一个 MCP 服务端和命令行界面（CLI），旨在实时检测 Cursor 和 Claude Code 等编程助手中的实际漏洞。

[Chainguard](https://thenewstack.io/chainguard-500-million-builds/) 则采取了完全不同的角度，提供加固的极简容器镜像和精选的软件包仓库——在写下第一行代码之前就保障了基础设施层的安全。[今年早些时候，Snyk](https://thenewstack.io/security-firm-snyk-tackles-ai-codings-perfect-storm/) 的安全研究人员完成了他们所谓的针对 AI 智能体技能（skills）生态系统的首次全面审计，扫描了近 4000 个技能，并发现超过三分之一的技能至少包含一个安全缺陷——这突显了整个市场上攻击面正在以多么惊人的速度扩张。此外，[Arcjet](https://thenewstack.io/arcjet-wafs-guards-ai-agents-security/) 在智能体工作流内部提供运行时执行、提示词注入防护以及个人身份信息（PII）阻断。而 [Mobb Security](https://thenewstack.io/ai-coding-tools-create-more-bugs-than-they-fix/) 则针对 AI 智能体技能供应链漏洞进行防御。

Delbare 接受了 *The New Stack* 的采访，探讨了 Aikido 在该领域中的定位，以及为什么他认为“责任真空”是行业目前仍未解决的问题。

*以下采访内容经过了编辑，以确保清晰与简练。*

**关于允许 AI 编程智能体安装什么，其安全策略应该由谁负责？这是开发团队的决定、安全团队的决定，还是说目前在大多数公司中仍处于未定义状态？**

目前在大多数公司中，这仍处于未定义状态，这是一个真正的风险。当人类开发人员安装软件包时，至少存在隐性的责任制。当智能体自主行动时，除非有人刻意承担起责任，否则就无人负责。

我们的观点是，安全团队需要设定护栏（即策略、阈值和获得批准的生态系统），而开发人员应该能够在这些护栏内自由活动。智能体就在这个范围内运行。这并不是一种新模式，它只是应用了同样适用于人类开发人员的共同责任模型。

**大多数企业对智能体触发的安装没有任何可见性。Aikido 是否在客户环境中看到了这种情况——智能体悄悄拉取了没有任何人审查过的软件包？**

就在今天，我们与一位客户进行了讨论，他们希望确保即使是非开发人员的设备也能得到保护。产品、销售和市场营销等技术性较低的团队正在使用 AI 智能体来完成工作，但他们并没有意识到软件包和智能体技能正被安装到他们的本地环境中。安全团队基本上对这种风险没有任何控制力或可见性，也无法在安全事件发生后识别受影响的机器。

**AI 编程智能体能够自主拉取软件包并添加依赖项——Endpoint 是如何区分人类触发的安装和智能体触发的安装的？这种区分会改变其响应方式吗？**

它不作区分。安装恶意软件的风险与该行为是由人类还是智能体触发无关。

**Endpoint 目前监控哪些 AI 编程智能体（如 Copilot、Cursor、Claude Code、Devin 等）？当新的智能体发布时，监控覆盖又是如何进行的？**

Endpoint 监控横跨 Gemini、OpenAI、GitHub Copilot、xAI、MCP 服务端、Claude Code 和 skills.sh 的 AI 工具和模型。当新工具发布时，只需对该 Agent（探针/客户端）进行更新即可。

**AI 智能体技能市场被列为了覆盖区域——具体是哪些市场？这种覆盖的成熟度如何？**

目前，我们覆盖了 skills.sh 和 VS Code 插件市场（Marketplace）。

**你们的新闻稿中提到了“8 美元的 ChatGPT 订阅”降低了编写供应链恶意软件的门槛。AI 生成的供应链恶意软件与人类编写的相比有什么不同？它是更难检测、更具多态性，还是数量更多？**

AI 正在催生更为复杂的攻击。在 12 个月内，我们经历了从单一软件包沦陷，到自我复制的蠕虫，再到跨注册表链式传播的完整 CI/CD 流水线劫持。主要的变化在于准入门槛的降低和攻击速度的提升。过去有才华的黑客可能需要花费大量时间来探测漏洞，而现在这项工作可以分派给 AI 智能体来完成。

**Aikido Intel 引用了“每天 10 万个恶意软件包”的数据，它是否使用 AI 检测来跟上 AI 生成的恶意软件？其方法论是什么？**

是的，Aikido Intel 使用 AI 以及我们内部的研究团队来发现开源供应链中的漏洞。

Intel 的工作原理是审查所有公开可用的变更日志和发布说明，以确定是否已经进行了安全修复但尚未公开。我们使用两个大语言模型（LLM）来实现这一点。其中一个 LLM 用于过滤数据并移除所有不必要的上下文。接着，第二个 LLM 专注于漏洞分析。随后，人类安全工程师会审查 LLM 的发现，验证这些结果，并在确认漏洞后发布安全情报（Intel）。

与使用 LLM 直接扫描所有代码库、系统或生产环境以查找潜在安全问题相比，这种方法在识别漏洞方面极其有效，同时所需的计算资源要少得多。

**Endpoint 执行了 48 小时安装封锁。这是一种简单粗暴的手段——许多合法的软件包发布得非常快。在实践中，这会产生多少误报？“申请与审批”工作流又是如何处理开发人员的抵触情绪的？**

在实践中，开发人员有多大频次会*需要*今天早上发布的新版本，而不是上周的版本？几乎从来不需要。安装封锁针对的是一个时间窗口，绝大多数恶意软件包都会在此窗口内被捕获，而合法的开发工作几乎从未因此受阻。

Endpoint 会回退到一个已被允许的软件包版本。在这种情况下，即发布时间早于 48 小时安装封锁窗口的版本。但安装封锁是一个可配置的设置。团队可以根据每个生态系统的风险承受能力来匹配安装封锁时间。例如，由于许多攻击都针对 npm， 48 小时可能是合理的；而对于具有 GPG 签名要求的 Maven Central，你可能不需要安装封锁。

软件包或整组软件包可以被列入白名单，从而绕过安装封锁。而对于需要更快部署的安装，开发人员可以申请单次审批。
