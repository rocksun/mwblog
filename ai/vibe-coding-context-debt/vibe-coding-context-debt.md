本文指出，AI驱动的氛围编程导致了大量低质量代码，而更深层的问题是上下文债务。随着AI生成代码速度激增，系统架构复杂度和依赖关系已超出人类理解范围，企业需建立机器可读的上下文图谱来解决这一结构性危机。

> 译自：[Vibe slop is the symptom. Context debt is the disease.](https://thenewstack.io/vibe-coding-context-debt/)
> 
> 作者：Matt Burns

一些推动了[氛围编程](https://thenewstack.io/vibe-coding/)发展的工程师们现在认为这是一个问题。上个月，《华尔街日报》的 Christopher Mims [采访了](https://www.wsj.com/tech/ai/vibe-coding-slop-ai-tools-e6a99394) Armin Ronacher 和 Mario Zechner，他们是支撑 OpenClaw 的 Pi 引擎背后的工程师。这些工程师在推广这些代理工具方面发挥了重要作用，但他们的评估直言不讳：这些工具正在向世界灌输糟糕的、有时甚至是危险的代码。“最终我们会为此付出代价，”Zechner 告诉 Mims。

他们指的是“氛围垃圾”（vibe slop）。这是指由某人通过提示词（prompt）生成的、充满错误、效率低下且难以维护的软件。Ronacher 和 Zechner 还谈到了氛围编程的一个被忽视的方面。他们表示，草率的代码不仅更容易崩溃，还会消耗更多的计算资源、内存和带宽。他们警告说，一些通过氛围编程构建的初创公司甚至可能无法承担自己的计算成本。

我很大程度上同意 Ronacher 和 Zechner 的观点。但有人为这个问题提出了一个更尖锐的名称，而且是在一份产品发布文章中提出的。

## 真正的债务是上下文

6 月 2 日，AI 原生平台 [Postman](https://www.postman.com/) 发布了其所谓的“AI 工程师”（AI Engineer），首席执行官 [Abhinav Asthana](https://www.linkedin.com/in/abhinavasthana/) 在发布文章中[提出了这个观点](https://blog.postman.com/introducing-the-ai-engineer/)。

![](https://cdn.thenewstack.io/media/2026/06/e37a1831-1528853823461.jpg)

Abhinav Asthana，Postman 的首席执行官兼创始人

他认为，糟糕的代码是你所能看到的部分。因为显而易见，所以它受到了指责。

更棘手的问题是包裹在它周围的一切，即那些堆积速度快到无人能追踪的各种服务和 API。他将其称为“上下文债务”，我完全认可他的说法。

这个概念是这样的：在氛围编程平台下，系统会迅速从最小可行性产品（MVP）演变成充斥着无数 API、服务和数据库的产品，而这些组件之间的交互方式并没有人进行完整的设计或理解。

每一个新服务、每一个新平台、每一个由代理生成的变更都会堆叠在过去的工作之上，而在添加新工作时，这些往往会被遗忘。就像技术债务一样，上下文债务会复合增长。

与技术债务不同，你无法通过重构来摆脱它，因为债务不在代码本身，而在于代码的含义**以及它如何连接**。

> 与技术债务不同，你无法通过重构来摆脱它，因为债务不在代码本身，而在于代码的含义以及它如何连接。

大多数组织仍然用老派的方式处理这个问题，即依靠少数几位将整个地图记在脑子里的资深工程师。组织依赖于这些工程师的机构知识，而这个过程无法扩展。现在，当一群编程代理开始以机器速度构建和发布时，生产速度超过了团队理解已构建内容的能力。

这就是陷阱。在不修复上下文的情况下修复“垃圾”只会将故障推向上游。你开始将代码推送到一个无人再能理解的系统中。

Asthana 认为时间不多了。他预计在六到九个月内，上下文债务将超过大多数团队手动管理的能力，他说警告信号已经显现：处于早期阶段的初创公司无法保持其自身架构的一致性。在他看来，在这个氛围编程的时代，规模最小、行动最快的公司正率先撞墙。

## 代理工程需要记忆

Simon Willison 指出[代理工程](https://simonwillison.net/guides/agentic-engineering-patterns/what-is-agentic-engineering/)是一个有用的框架。代理在循环中运行，编写和执行代码，而真正的人类技能在于目标定义、工具准备和验证。人类成为定义、审查和集成的架构师。

这种框架定义了人应该如何与代理协作完成任务。上下文债务则是关于在一个组织中，数千个此类任务之间会发生什么的问题。代理可以验证其代码是否有效。但它很难验证它刚刚设计的 API 是否与另一个团队已经使用的 API 重复，或者它悄悄更改的契约是否破坏了无人记录的内容。

Postman 的解决方案是其所谓的“上下文图谱”（Context Graph）：这是一个始终在线、持续更新的地图，涵盖了 Postman 组织内的 API、服务和依赖关系，为“AI 工程师”在行动前提供基础。代理在沙箱中运行并执行需要人工参与的写入操作；输出内容与现有的 PR（拉取请求）审查流程集成。Asthana 将风险模型描述为一名初级工程师，其工作可以经过代码审查。普通的编程代理可以编写一个函数。但它无法分辨你应该使用你 17 个支付 API 中的哪一个。

> 普通的编程代理可以编写一个函数。但它无法分辨你应该使用你 17 个支付 API 中的哪一个。

这在规模化时更为重要。Asthana 告诉我，Postman 映射出的最大上下文图谱远远超过了任何资深工程师能记在脑子里的内容。他们为一家美国大型电信公司映射了超过 1,100 个 API，为一家全球电信公司映射了超过 2,600 个，并为一家大型科技公司映射了超过 11,000 个。

Postman 一直在其自己的工程团队中运行这些代理，并与我分享了一些数据。Asthana 告诉《The New Stack》，API 涉及该公司 68% 的拉取请求流量，Postman 在与 API 相关的工作中运行“AI 工程师”。他说，迄今为止最重大的发现是一组下游依赖关系的变更，这些变更在审查中可能会通过，但在生产环境中会失败。

> 一份过时的地图可能比没有地图更糟糕，因为代理会凭此自信地采取行动。

Postman 的赌注是，过去十年来客户在其工具内记录 API 的行为，使其在构建该地图方面占据领先地位。这是一个真正的优势，但也伴随着真正的陷阱。地图的好坏取决于其背后的信息，而许多公司的记录已经多年未更新。Asthana 引用了 Postman 自身使用时的一次失败：代理不知道在另一个数据中心运行的一些实时系统，因此它基于不完整的地图工作，导致部分分析出错。失败的原因是地图本身。一份过时的地图可能比没有地图更糟糕，因为代理会凭此自信地采取行动。

## 这是一种类别，而不仅仅是一个功能

无论 Postman 的“AI 工程师”是否会成功，这种框架正开始看起来像是一种趋势。Cursor 和 Windsurf 会索引你的存储库。Claude Code 读取你的 [CLAUDE.md](http://claude.md) 并了解你的代码库。GitHub 正在将 Copilot 更深地集成到依赖图中。主要的编程代理公司似乎正朝着同一个方向前进：上下文和模型一样，都是瓶颈。

预计该领域的更多供应商将宣布上下文层，大多数工程团队最终会选择购买其中一个。当他们这样做时，重要的是要考虑他们组织的上下文是否以机器可读的形式存在（而不是只存在于某人的脑子里）。

基础设施在这个新的代理世界中变得越来越关键。这是代理需要立足的基础。团队需要开始更重视上下文。

上下文债务一直存在。代理使这个问题变得更加紧迫。Pi 的工程师们可能是对的，即“垃圾”最终会找上门来。胜出的团队将不是那些使用最好模型或编程代理的团队，而是那些系统对人类和机器同样友好的团队。
