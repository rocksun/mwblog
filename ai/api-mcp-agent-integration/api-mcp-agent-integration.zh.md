在这个智能体时代，每个人都对“数字同事”的潜力感到兴奋。模型上下文协议（MCP）是大家最近热议的话题，当谈到智能体赋能时。但是，如果你在过去十年里一直为你的组织构建API，该怎么办？你是否必须放弃所有这些工作，转而采用MCP优先的策略？

在你因噎废食之前，让我们首先考虑一下在构建智能体应用程序时，API和MCP集成的多种方法。

## API：古老而可靠的桥梁

API通常被描述为连接两个系统使其相互通信的桥梁，但我发现将API视为餐厅菜单上的选择更为恰当。每道菜的内容都清晰标明，因此你知道你将得到什么。如果你点牛肉，你就不会收到意大利面。就像菜单一样，API由人类预定义，并附有可在使用前查阅的文档。

API不允许推测。例如，如果你想在应用程序中获取客户数据，你会使用API中的客户数据接口。选择客户数据接口意味着你将得到你正在寻找的内容——在这个例子中，是客户数据，而不是天气预报数据。

> “API通常被描述为连接两个系统使其相互通信的桥梁，但我发现将API视为餐厅菜单上的选择更为恰当。”

API的有用之处在于它们是具体的且结构紧凑，要求代码遵循定义的结构。例如，设计良好的REST API遵循清晰的动词（系统可以做什么）和名词（系统中的组件）模式。清晰的动词示例包括“获取”、“创建”、“删除”，名词示例包括“文件”、“用户”、“发票”。这种分离是为了让机器能够基于API的语义，以规范、受控的方式进行交互。

在AI智能体出现之前，构建利用API的客户端应用程序需要自定义代码，这些代码直接与每个特定的API对齐。应用程序只能与被明确告知要使用的接口进行交互。另一方面，AI智能体在获得一个接口后，会反复调用它，尝试理解其工作原理，直到获得成功的响应。

它们学习API，但有时会带来灾难性的后果。智能体已被证明会过度调用接口、检索敏感数据，或者重复尝试API直到意外破坏某些东西。更糟糕的是，一些智能体泄露了API凭据，例如[OpenClaw API密钥泄露](https://www.bitsight.com/blog/openclaw-ai-security-risks-exposed-instances%23:~:text=The%2520traffic%2520included%2520prompt%2520injection,'ve%2520read%2520the%2520source.%25E2%2580%259D)的案例。

即使在这个新的智能体范式下，API在交付成功的智能体方面仍然发挥作用。例如，如果你有可以提高智能体准确性的私有数据，但这些数据高度敏感且需要复杂的授权结构，API可以确保对其进行受控访问。然而，你需要仔细考虑你针对智能体的API策略，因为这确实存在成本……字面上的成本。

例如，如果你有一个包含许多字段的复杂API，你需要精确地文档化如何为智能体使用它。智能体也可能选择开始探索API中的其他接口。

提供关于如何使用API以及具体的API参数的信息，是智能体在其整个会话中需要携带的额外信息，这意味着你将增加花在API信息上的token数量，并占用上下文窗口中更多的空间。

与其依赖可能消耗大量token的详细API，不如考虑使用模型上下文协议（MCP）。MCP为智能体提供了一种更轻量级、基于规范的集成替代方案，有助于在整个会话中节省你的token。

## MCP：智能体交互协议

[模型上下文协议](https://thenewstack.io/model-context-protocol-a-primer-for-the-developers/)（MCP）是为AI智能体而非人类直接与工具、数据源和应用程序交互的世界而设计的。由于MCP是通用的AI集成标准，智能体无需理解自定义客户端代码（例如REST接口）即可与MCP服务器通信。

所有集成都使用相同的协议，因此比API更统一。此外，每个MCP服务器的功能——工具、资源、提示——都是自描述的，因此它们会宣布自己的能力。你不需要额外的工具来解决API中的文档问题。

MCP使智能体能够动态发现和使用工具，无需任何文档，[类似于USB是连接新外围硬件的通用方式](https://modelcontextprotocol.io/docs/getting-started/intro)，通常不需要下载新软件。智能体可以基于标准采取行动，无需额外的工具或解决方案。通过MCP，服务器会“宣传”它们能做什么。

作为同事的智能体可以“看到”可用的功能，并根据MCP服务器提供的描述自主做出决定。AI智能体将判断其独立制定的问题解决方案计划是否需要这些工具。

因此，本质上，[MCP使智能体](https://thenewstack.io/how-mcp-enables-agentic-ai-workflows/)能够理解并找到特定的“工具”来完成任务。与API不同，通过MCP访问的工具不依赖于单独的、详尽的文档。API准确地告诉机器做什么，而MCP告诉AI智能体存在哪些工具，以便智能体可以根据需要使用它们。

如果API是餐厅的实体菜单，那么MCP就像一个食物配送应用程序。一家餐厅只能提供该特定地点（API）的菜单。然而，MCP提供了一个通用的“应用程序”，允许用餐者（智能体）使用单一界面浏览、点餐并与该区域的*任何*厨房进行交流。

用餐者不再需要为每家餐厅下载单独的应用程序，MCP客户端充当统一的店面。它协调对MCP服务器（各个厨房）的访问，使AI能够动态发现新的“菜肴”（工具和数据），而无需手动设置。通过标准化这种握手，MCP将AI从一个受限的、单一菜单选项转变为一个拥有全市访问能力的智能体。

> “如果API是餐厅的实体菜单，那么MCP就像一个食物配送应用程序，提供一个通用的‘应用程序’，允许用餐者（智能体）浏览、点餐并与任何厨房进行交流。”

鉴于使用这些MCP服务器访问工具的LLM的非确定性性质，IT组织必须通过实施MCP网关来考虑MCP服务器的控制和治理。

## 定义你的智能体集成策略

鉴于潜在的风险和治理难题，为什么你会考虑MCP集成？很简单……因为智能体需要动态性。API是静态的，这对于智能体同事来说可能过于僵化。话虽如此，在组织因安全或监管原因需要更受控、确定性方法进行特定实例时，API对智能体仍然具有相关性。

你可能会问，“为什么我不能只用MCP封装我的API？” Tanzu团队已经看到Spring AI成功地利用MCP工具命令封装API的多个案例，从而减少了智能体在交互过程中需要理解和查找的复杂规范数量。

从表面上看，这可能很有吸引力，因为你不会损失在API上数十年的投资，并且可以节省token。然而，封装并非智能体友好型API的万能药。这取决于对每个API的具体分析。需要跨多个数据源进行动态分析以提供建议的场景是MCP封装的良好候选者（例如，零售推荐引擎或实时投资分析）。

这类分层集成可能会使用API来获取客户数据，然后添加一个单独的MCP集成来获取实时数据。然而，每个用例仍然需要仔细评估。

无论你的分析结果如何，有一点是肯定的。MCP服务器和API必将随着时间的推移在AI用例中激增。负责IT性能、成本、合规性和安全的组织将需要对其激增的集成进行**可观测性**、防护和可审计性。
