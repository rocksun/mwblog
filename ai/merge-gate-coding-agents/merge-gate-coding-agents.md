文章指出代码审查已成为AI时代的开发瓶颈。传统合并契约在AI海量产出下失效，应将耗时的系统级验证前置，利用轻量级临时环境在合并前完成全链路测试，确保合并契约的完整性与可靠性。

> 译自：[85% say code review is the new bottleneck. Here's what the AI coding narrative leaves out.](https://thenewstack.io/merge-gate-coding-agents/)
> 
> 作者：Arjun Iyer

合并是一种契约。当一个变更进入主分支（main）的那一刻，组织内的所有其他团队都会基于该变更能够正常工作这一假设进行开发。他们从该分支拉取代码，在其基础上进行部署，并在出现故障时进行调试，坚信前人留下的代码是可靠的。

大多数工程组织从未将这一契约明确写下来。他们拥有一个运行检查的流水线，无论检查涵盖了什么，就是合并所承诺的内容。问问大家，PR 上的绿色对勾实际上保证了什么？你很少能得到准确的答案。

> “合并是一种契约。当一个变更进入主分支的那一刻，组织内的所有其他团队都会基于该变更能够正常工作这一假设进行开发。”

这种模糊性在过去是完全可以接受的。代码是以人类的速度到达的，由那些将上下文记在脑海中并持续跟进的人编写。这两个条件现在都已经不存在了。

氛围编程代理（Coding agents）正在成倍增加 PR 的数量，越来越多的代码变更是由没有任何持久上下文的工具编写的。业界已经意识到了后果：在 GitLab 最近的一项调查中，[85% 的受访者表示瓶颈已经从编写代码转移到了审查代码上](https://thenewstack.io/gitlab-ai-code-governance/)。现在是明确定义契约并诚实审视当前流水线在哪些方面未能满足契约的时候了。

## 四层信心模型

剥离掉工具层面，一个变更在组织其余部分基于它进行开发之前，需要确立四个方面的保证。

首先，它格式规范。它能编译通过，通过静态分析和 lint 检查，并能生成可部署的制品。

其次，它内部正确。单元测试在隔离状态下确认了逻辑符合作者意图，并模拟了依赖项。

第三，它在边界上兼容。它遵守消费者所依赖的 API 契约及其生产者发出的模式（schemas）。

第四，这是在分布式系统中最重要的一层，即它在真实系统中行为正确。这既包括功能性行为，也包括非功能性行为：在功能上，它返回正确的结果并符合邻近服务期望的流程；在非功能性上，它在真实流量下能经受住延迟、错误率、资源使用和向后兼容性的考验。这一点不是通过模拟（mocks）或记录的固定数据（fixtures）来检查的，而是针对它所调用的服务以及调用它的服务的实时版本，使用真实的数据形态和真实的故障模式进行检查。

> “第四层是最昂贵的。而且第四层是唯一能够捕捉到微服务中真正致命问题的层级：局部正确但系统性错误的代码。”

前三层很便宜。它们不需要特定的环境，可以轻易并行化，并且在几分钟内就能完成。第四层是昂贵的，因为它需要一个真实的环境来运行。而且第四层是唯一能够捕捉到微服务中真正致命问题的层级：局部正确但系统性错误的代码。

## 为什么最高层级被排除在关口之外

在过去十年中的大部分时间里，第四层并没有在合并前运行。因为这根本做不到。建立真实系统行为意味着需要一个所有服务都在运行的环境，而这样的环境稀缺、昂贵且创建缓慢。团队通常只有一两个共享环境，且经常被占用或损坏。

因此，业界做了一个沉默的妥协。在满足前三层后合并，然后在共享环境中发现系统行为，所有未经完全验证的变更都在这里堆积。我们构建了发布管理、冻结窗口和预发布值班轮换，很大程度上就是为了应对这种妥协带来的后果。

这种妥协在人类开发速度下勉强维持。当共享环境经历十次变更时，有人可以重新梳理时间线并通过询问找到罪魁祸首。代码作者当时仍在附近，上下文完整，准备好推送修复程序。

> “系统验证属于合并后，这从来都不是一种原则性立场。这只是对基础设施限制的妥协。”

请注意这种妥协的本质。系统验证属于合并后，这从来都不是一种原则性立场。这只是对基础设施限制的妥协：预合并环境不切实际，所以关口缩小到了基础设施所能提供的范围。

## 代理将妥协变成了负债

现在改变代码产出的速度。代理让百人规模的工程团队能[产生出](https://thenewstack.io/ai-generated-code-crisis/)曾经需要数百人才能完成的 PR 数量。每一个变更都能轻松通过前三层，因为这些正是代理最擅长处理的检查。代理会不断迭代，直到构建通过、单元测试通过、契约对齐。

代理自己无法发现的是，其局部完美的变更可能会破坏三个服务之外的消费者，因为其反馈循环中没有任何关于该消费者的信息。这不是一个假设的风险：对开源 PR 的分析发现，[AI 编写的变更带来的问题大约是人类编写的 1.7 倍](https://leaddev.com/ai/ai-generated-code-sparks-production-confidence-crisis)，逻辑和正确性错误尤为突出。如果合并关口止步于第三层，那么现在系统性地放行了那些最危险的故障模式从未被检查过的变更。

合并后的发现机制也因为同样的算术原因而崩溃。在共享分支上，十个批次内的变更归因是可容忍的。如果有五十个，其中大部分是机器编写的，且作者在 PR 关闭的瞬间就失去了上下文，那么修复错误就变成了一种“考古工作”。批次中每增加一个变更，诊断每个故障的成本就越高，而这恰好发生在批次规模不断增长的时候。

旧妥协的两部分同时失效了。关口放行了不该放行的变更，而事后的安全网已无法再消化它们。

## 曾证明这一切合理的限制已经消失

这就是[基础设施方面发生的变化](https://thenewstack.io/how-autonomous-agents-are-changing-infrastructure-management/)。你不再需要整个技术栈的完整副本就能为一个变更提供运行的真实系统。

取代复制的模式很简单。你的 Kubernetes 集群运行一个共享、稳定的环境，其中包含每个服务的当前版本，并通过标准的部署流水线保持更新。当一个 PR 需要验证时，只需将该 PR 更改的一两个服务与稳定版本一起部署即可。

隔离来自路由流量，而不是复制整个技术栈。属于该 PR 的请求在头部带有一个小的识别标签。路由层读取标签，并将这些请求定向到它们存在的已更改服务，并将其他所有请求定向到稳定版本。每个 PR 都能看到一个完整的、类生产的系统，其中只包含它自己的变更，数十个 PR 可以同时在同一个集群中验证，而互不影响。

> “隔离来自路由流量，而不是复制整个技术栈。每个 PR 都能看到一个完整的、类生产的系统，其中只包含它自己的变更。”

由于只需要启动更改过的服务，这种轻量级的临时环境在几秒钟内即可就绪。第四层信心模型，即曾经因太过昂贵而无法在合并前建立的层级，现在的成本大约相当于一次单元测试的运行。这就是像 [Signadot](https://www.signadot.com/) 这样的平台所提供的能力，它消除了合并关口止步于第三层的最后一个理由。

## 重写契约

随着限制的消失，流水线围绕明确的分工进行了重组。

CI 保留前三层，并通过剥离其他所有内容来提速。构建、静态分析、单元测试、契约检查，几分钟内完成。它的工作是回答一个问题：这是一个格式规范的组件吗？

第四层移动到 PR 内部。每个 PR，无论是人类编写还是代理编写，都有自己的[临时环境](https://thenewstack.io/smart-ephemeral-environments-share-more-copy-less/)，并在合并前并行运行集成和端到端验证。合并后沉重的集成阶段——那个使整个流水线变慢的阶段——缩减为一个简单的冒烟测试，或者完全消失。

更大的变化在于各阶段意义的改变。合并后的流水线不再是发现问题的地方，而是确认问题不存在的地方。发现问题应该在签署契约之前，而不是之后。

合并契约因此有了书写的价值，因为每一条条款都得到了强制执行：该变更可以构建，逻辑经过测试，契约兼容，并且已经针对真实系统正确运行。绿色意味着经过验证，而不是“可能没问题”。

## 关口现在是一种选择

多年来，关于合并应该卡控什么的诚实回答是：我们的环境能支持什么。那个答案现在不再准确了。针对类生产系统的每变更验证不再是流水线中昂贵的部分，代码量也不再是软件交付的限制因素。验证才是。

那些最快适应氛围编程辅助开发的团队，并不是产生代码最多的团队。他们是那些将系统验证前置到合并左侧的团队，这样速度就转化为已交付的功能，而不是变成一长串未经验证的变更队列。基础设施允许这样做。代码量需求也要求这样做。
