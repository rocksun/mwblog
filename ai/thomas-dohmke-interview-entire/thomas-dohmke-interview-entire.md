GitHub前CEO Thomas Dohmke创立Entire平台，获6000万美元种子轮融资，旨在打造面向AI代理时代的新开发者协作平台，核心为三层架构和Checkpoints工具，解决代码审查瓶颈，并计划引入大量AI代理。

> 译自：[GitHub's former CEO launches a developer platform for the age of agentic coding](https://thenewstack.io/thomas-dohmke-interview-entire/)
> 
> 作者：Frederic Lardinois

当GitHub首席执行官 Thomas Dohmke 于2025年8月[离开](https://thenewstack.io/github-loses-its-ceo-and-independence/)微软旗下的公司时，他说此举是为了回归他的创业本源。现在，经过几个月的开发，他正在推出[Entire](https://entire.io/)，这是一个新的开源开发者平台，它重新构想了开发者和代理之间如果从头开始构建，协作会是什么样子。

Entire 获得了6000万美元的种子轮融资，这是开发者工具历史上最大的一笔，由 Felicis 领投，Madrona、Basis Set 和微软的风险投资部门 M12 参投。

无论以何种标准衡量，这都是一笔巨额融资，但 Dohmke 的声誉——他曾领导 GitHub 从代码仓库演变为以 Copilot 为核心的 AI 平台——无疑起到了帮助作用。此外，鉴于软件开发的快速发展，需要大量投资才能跟上市场不断变化的需求。

在接受《The New Stack》采访时，Dohmke 解释了他创立 Entire 的愿景。“我们正在摆脱将工程视为一种手艺，即手动在文件和文件夹中编写代码的方式——这实际上就是过去15年 Git 仓库的作用，对吗？如果你查看任何一个仓库，它实际上就是一个文件浏览器，你可以在搜索中点击文件等等，” Dohmke 说。“我们正在从这种方式转向一个更高的抽象层次，即规范——推理、会话日志、意图、结果。我们相信这需要一个与今天的 GitHub 截然不同的开发者平台。”

Dohmke 指出，GitHub 是为人与人之间的交互而构建的。整个系统，从问题到拉取请求再到部署，都不是为这个 AI 代理的新时代而设计的。然而，当今最先进的开发者通常并行使用十几个代理，甚至我们定义软件项目的方式也在迅速变化。在 GitHub 上构建这一点——并带动其现有用户群——将是一个“非常不同的尝试”，Dohmke 说。

## 离开 GitHub 再次成为创始人

值得注意的是，Dohmke 强调他离开 GitHub 是完全友好的。他离开 GitHub 并非为了打造一个竞争对手。

“在微软工作了十多年，其中七年在 GitHub，担任了四年首席执行官之后，我再次感受到了成为创始人并创造新事物的冲动，”他说。“今年6月，我与 [微软首席执行官] Satya Nadella 进行了一次愉快的对话，我向他阐述了我的想法。他的回应是，‘嘿，你知道的，第一，请继续努力直到你离开的那一天。第二，让我们看看我们能在微软生态系统中一起做些什么。”

鉴于此，微软的风险投资部门投资 Entire 也就不足为奇了。

## Entire 正在构建什么？

同样值得注意的是，尽管 Entire 是一个平台项目，但它不一定会与 GitHub 竞争。Dohmke 表示，其理念是在技术栈中构建一个更高层次的层，让开发者能够管理代理的推理过程并与它们协作。代码仓库仍将是其核心。

Entire 正在构建一个三层平台，以一个从头开始构建的与 Git 兼容的新数据库作为基础，中间是语义推理层，顶部是用户界面。

该团队认为，新的数据库层是必要的，因为这些新仓库中存储的信息是不同的——具体来说，代理在使用这些工具时可以发出比人类更多的上下文信息。这个新数据库将允许人类和代理不仅可以查询代码，还可以查询代码背后的推理过程。

鉴于代理使用此数据库及其 API 端点的频率可能远高于人类使用 Git 仓库的频率，该团队还需要考虑性能。

Dohmke 还表示，这个新数据库，与传统的集中式 Git 仓库不同，可以是一个全球分布式节点网络。对于需要（或希望）确保数据主权的用户来说，这是一个主要卖点。

## 检查点

该团队推出的第一个产品是中间层的一部分。这款名为 Checkpoints 的新开源工具集成了 Claude Code 和 Google 的 Gemini CLI（即将支持 Open Codex），并自动提取和记录代理的推理、意图和结果。

“中间层旨在向人类和代理提供所有导致软件产品的信息，” Dohmke 解释道。“而今天，在 GitHub 仓库中，只有代码、有时是文档和依赖项，但它基本上缺失了所有关于你是如何达到这个状态的[信息]。”

![](https://cdn.thenewstack.io/media/2026/02/dc91247f-entire-cli-scaled.png)

Entire 的 CLI 中的 Checkpoint 工具（图片来源：Entire）。

那是因为这些系统是为人类开发者构建的，尽管开发者在完成代码编写后可能会编写测试用例和文档，但记录其确切的推理步骤从未是该过程的一部分。然而在传统的非代理工作流中，许多机构知识从未被记录下来。

“这是我们更大愿景的第一步，即在软件项目的整个生命周期中提供语义推理层，这样你就可以在未来的任何时候，作为人类和代理，追踪决策是如何做出的，” Dohmke 解释说。

随着所有这些数据的保存，Checkpoints 将允许开发者审查代理是如何生成代码的。

尽管用户界面仍在开发中，但 Entire 已经构建了部分功能来帮助可视化存储在 Git 中的检查点。然而，目前团队主要专注于命令行体验。

## 审查瓶颈

Dohmke 强调的当前开发者/代理工作流中的一个问题是当今我们经常听到的：代码交付的瓶颈不是编写代码，而是审查代理编写的代码。这已经导致了[开发者倦怠](https://thenewstack.io/is-ai-creating-a-new-code-review-bottleneck-for-senior-engineers/)。

Dohmke 认为，未来是更多的代理。

“如果你在软件生命周期中继续这样做，编写代码之后的下一步就是审查代码——无论是你自己的代码还是团队成员在拉取请求中的代码，” Dohmke 说。“但是拉取请求在理解代码方面存在同样的问题。它向我展示了我从未编写过的文件的更改。而像 Copilot 代理这样的代码审查代理会给我关于其代码的反馈，这在我仍然有基本理解时很棒，但如果我根本不理解该代码的作用，它就会变得毫无意义或多余。”

当代码更多而上下文更少时，解决方案可能是使用代理和确定性工具来测试代码，并确保其合规和安全。

“它正变得越来越成为一个瓶颈，因此你必须将这一步从流程中移除，”他解释道。“我认为这是行业面临的最大挑战之一，因为与此同时，我们正在与越来越多的网络攻击作斗争，许多组织引入了零信任流程，这意味着没有人工审查，任何东西都无法部署。所以，我认为，在我们的愿景中，很多创新将在这里发生，我们希望成为其中的一部分。”

## 招聘人类和代理

随着本轮融资的结束，Entire 计划将其员工人数从目前的15人增加到约30人，并尽快建立其平台。但正如 Dohmke 强调的，现在不再仅仅是人类员工重要。即使在其新闻稿中，Entire 团队也指出计划将其团队扩展到“数百个代理”。

“我认为在2026年，任何领导者都需要将员工数量不再仅仅视为工资、福利、差旅和开销，而是视为代币。我曾与工程师们交流，包括我自己的团队成员和湾区的工程师，他们都在谈论每月数千美元的代币费用，” Dohmke 说。

至于商业模式，Dohmke 告诉我们，该团队计划遵循成熟的开源策略，即在宽松许可下提供平台的大部分功能，然后提供带有额外功能的托管服务以实现盈利。
