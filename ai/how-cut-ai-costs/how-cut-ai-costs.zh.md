AI成本正在不断上升，原因并非大多数人所想的那样。诚然，模型本身确实越来越昂贵，且团队在使用时也缺乏节制。

但导致成本上升的另一个原因是：混乱的语境（context）。

语境管理是当今企业面临的最大未解难题之一。企业内部堆积如山的信息需要转化为语境，以便AI代理能够可靠（且低成本）地使用。为了让代理能够获取这些知识，工程师们通常会将代理连接到数十个 [MCP 服务器](https://thenewstack.io/15-best-practices-for-building-mcp-servers-in-production/)。

根据 Gartner 的说法，“语境累积是主要的成本驱动因素。每个代理任务都会从企业文档、工具架构、合规规则和对话历史中组装出一个语境窗口。每一步操作都会重新摄取这种不断增长的语境，从而在整个工作流程中复合了成本。”¹

> “语境管理是当今企业面临的最大未解难题之一。”

因此，当一个代理需要弄清楚某件事时，比如[服务中断](https://thenewstack.io/a-cascade-of-failures-a-breakdown-of-the-massive-aws-outage/)是否与最近的部署有关，它会先问 GitHub MCP 谁拥有该服务，然后问 Jira 最近的工单，接着问 PagerDuty 事件情况。这涉及三次独立的调用，并且每当出现类似问题时，这些调用都会以不同的组合方式重复进行。

偶尔发生一次问题不大，但如果每月发生数百次（例如在工作流程中，或者因为它是热门问题），成本就开始累积了。抛开缓存不谈，第 1,000 次运行的成本与第 1 次是一样的。

这正是我们的数据所显示的。代理对 AI 的查询每月重复数千次，且形式略有不同。

每次运行这些查询时，代理都会拉取数十万个 token 的输入语境（这是代理查询中语境的主要部分）。

![显示输入与输出 token 比例随时间变化的图表。](https://cdn.thenewstack.io/media/2026/06/7e4d2ffb-image1-1024x576.png)

因此，我们决定进行一项实验，看看在获取语境时能否降低 AI 成本。

我们的假设是：通过预先关联语境并构建一个匹配流行查询模式的语义层，可以减少代理必须执行的数据跳跃（hops）和推理。更少的跳跃意味着消耗的 token 更少，从而降低成本。

## 实验过程

我们从生产环境中提取了数千个代理查询，对它们进行了分类，并创建了一个包含 1000 个常见 SDLC 查询的测试集。

然后，我们创建了以下四种条件来运行这些查询：

**Claude + MCPs（基准测试）。** 代理通过各自的 MCP 服务器直接访问 GitHub、Jira 和 PagerDuty，这也是大多数团队目前采用的设置。代理必须自行决定调用哪个工具、调用顺序，以及如何协调以不同格式和命名约定返回的结果。每个涉及多个工具的查询都需要代理在运行中解决这些不匹配问题，从而在每次尝试时消耗 token。

**Claude + MCPs + 技能文件。** 与前一条件相同的工具访问权限，但我们添加了一个 markdown 格式的技能文件，描述了每个工具的权限范围，解释了代码库结构和潜在的命名不匹配，并概述了每种问题的查找策略：从 GitHub 开始，找到所属团队，然后使用该团队名称查询 PagerDuty 排班。

[**语境湖（Context lake）**](https://thenewstack.io/context-lake-ai-agents/)**。** 代理只能访问 Port 的 MCP 服务器，该服务器公开了一个统一的软件目录，其中 GitHub、Jira 和 PagerDuty 的数据已经预集成。目录中的服务已经将 PagerDuty 服务、GitHub 仓库和 Jira 项目作为关系进行了关联。

**语境湖 + 技能文件。** 与前一条件相同的目录访问权限，外加一个针对目录结构优化的技能文件。它不再是查找策略，而是一个路由表：将查询类型映射到回答它的实体类型和属性字段。“谁负责服务 X 的待命（on-call）？” → 读取 `service[X].on_call`。代理的唯一工作就是找到正确的实体并读取正确的字段。

然后，我们针对所有 4 种条件和三种模型运行了这些查询。

总共运行了 12,000 次查询，并测量了每次查询的 token 使用量。

我们的主要发现：

* 在所有模型中，带有技能的语境湖始终比 Claude + MCPs 便宜 80%。
* Haiku 的成本从 $0.087 降至 $0.018，Sonnet 从 $0.333 降至 $0.059，Opus 从 $1.761 降至 $0.354。无论使用哪种模型，节省效果都成立。
* 昂贵的模型通过语境湖变得更适合日常查询。Opus 原本每查询 $1.761 的成本通常只保留用于最关键的任务。在语境湖的价格下，同样的 Opus 查询成本为 $0.354——虽然仍是最昂贵的选择，但在常见用途的承受范围内。

技能文件的效果在两个方面令我们感到惊讶：

为 Claude + MCPs 添加技能实际上让情况变得更糟：Haiku 上成本增加了 18%，Sonnet 上增加了 13%，Opus 上增加了 24%。我们原本期望路由引导能帮助代理走更直接的路径。但实际发生的情况是，代理像对待清单一样盲目遵循技能文件，按顺序执行每一个步骤，而不是推理它真正需要什么。

在语境湖上，技能文件确实起到了作用，但方式不同。由于数据已经整合，代理不需要进行冗余调用。技能文件的唯一任务就是指向正确的入口点。

技能文件本身并非坏事，但它们不是万能药。当底层数据本身结构良好时，它们似乎效果更好。

### 什么让语境湖更高效

语境湖是一个持续更新的统一知识层，它将企业的所有数据（如服务、团队、事件、部署和工单）连接到一个具有显式关系的单一结构化模型中。代理查询它，而不是查询单个工具。

> “代理永远不需要推理事物是如何连接的，因为这已经内置在数据模型中了。”

语境湖的两个特性帮助代理更高效地获取语境：

1. **预集成数据。** 在直接对接 MCP 的设置中，代理负责连接实体，使用来自彼此不了解的工具的部分结果。每次查询都在重复同样的连接工作。在目录中，一个 `service` 实体已经知道它的团队、仓库、PagerDuty 服务和 Jira 项目。代理只需读取一个对象即可获得全貌。效率提升不仅仅是调用次数减少了。更重要的是，代理永远不需要推理事物是如何连接的，因为这已经内置在数据模型中了。
2. **数据快捷方式。** 镜像和聚合属性可以缩短数据点之间的路径。例如，`service.on_call` 不是代理需要遍历的关系，而是在同步时通过镜像 `team.on_call_user` 已经填充好的字段。`open_incident_count` 也不需要代理去计算；它直接存储在那里。当代理通过逐个查询每个服务来统计事件时，它是在把 token 浪费在本来可以只做一次的算术上。目录在摄取时完成了这项工作，并直接提供结果。

### 这意味着什么

如果你现在正在开发 AI 代理，以下是你应该了解的：

1. 对于大多数查询，你可能不需要最昂贵的模型。Haiku 足以处理我们很大一部分测试案例，而且如果你的数据层高效，较便宜的模型本来需要处理的语境就更少。
2. 平台工程需要负责语境管理。目前，大多数组织将代理成本视为“代理问题”：调整提示词、添加更多工具，然后祈祷模型能自己搞定。但代理只是在响应所给出的任何结构。如果数据是碎片化的，代理就会频繁跳转。而这非常昂贵。
3. 语境窗口是一个需要管理的资源，且有预算限制。必须有人来负责，而平台工程是这项工作的合适人选。
4. 将数据预先集成到共享模型中，将工作从推理时间转移到了摄取时间。你为建立这种关系支付了一次成本；此后的每次查询都会更便宜。使用得越多，节省的效果越明显。
