**当 Gavriel Cohen 第一次看到 OpenClaw 时**，他知道这就是他想要的。当时，Cohen（很快将成为 [NanoClaw](https://nanoclaw.dev/) 和 [NanoCo AI](https://nanoco.ai/) 的创始人）已经停止了开发工作，转而投身于一个营销项目。

[Cohen](https://www.linkedin.com/in/gavrielco/) 告诉 *The New Stack*，当时他正沉浸在 Anthropic 的 Claude Code 中，重新唤醒了他的编程直觉，就在那时他发现了当时的 Clawd Bot（OpenClaw 最初的名字）。

“我进行了第一次小尝试，”他说。“我安装了它，将它连接到我的 WhatsApp，来回发送了几条消息。所以，我接触它确实是因为我们在构建一个 AI 原生营销机构时的实际需求。”

在此之前，正是带有终端 LLM 的 Claude Code 首先触动了他。“我立即尝试了它，这在思想上是一次巨大的解放，所以大约在 2025 年 3 月，我知道我需要重新开始构建了。”Cohen 此前曾是 [Wix](http://wix.com) 的开发人员，这是一家同样总部位于特拉维夫的无代码网站开发平台。

他知道自己的工作流程中缺少了关键的一环。“我正在规划我需要的东西，实际上我还买了一台 [专用于在后台运行 Claude Code 并设置定时周期性任务的 Mac mini](https://thenewstack.io/mac-mini-agent-infrastructure/)。”因此，起初他与 OpenClaw 的关系发展得很顺利。

## 偶遇自建包的意外惊喜

用了多长时间才发现潜在的问题？

“几乎是立刻就发现了，”Cohen 回忆道。“我正在看安装设置，里面有各种不同包的安装选项——其中一个是我几个月前构建的名为 NanoPDF 的包。我看到后心想：‘他们为什么要包含那个不起眼的小包？’”

你可能会认为 Cohen 看到自己的包被推荐会感到受宠若惊，但工具用户期望看到的只是经过实践检验、广受好评的第三方包。他深知自己的包两者都不是。它只有几个 star，而且好几个月都没有更新了。

但还不止于此：“在第一天或第二天调试时，由于一个定时任务没有触发，我查看了所有 WhatsApp 消息的日志——不仅是我连接的那一个群组，而是所有的群组。”

## 致命的五十万行代码

对于大多数开发者来说，粗心大意和糟糕的安全性的结合足以亮起红灯。

Cohen 在接受 *The New Stack* 采访时多次提到这一观察结果，原因在于：你可以更改政策或执行更严格的编码标准，但是一旦项目的代码库失控，变成无法维护的混乱状态，这个项目就完蛋了。即使是开源项目，能关注它的眼睛也只有那么多。到 2 月份，OpenClaw 已经有 3000 多个拉取请求（pull request）等待解决。

“但最重要的是，我看了看代码库，它居然有大约五十万行代码。”

Cohen 的营销业务只有三名员工。“我们有客户，所以我希望为每个客户配备一个智能体，但我无法将这东西连接到我的客户数据，我也无法在其之上构建业务，”他说。

于是 Cohen 做了他知道自己能做的唯一一件事。“我坐下来开始构建 NanoClaw。我必须把它做得极其精简，因为任何关心安全性的用户想要使用它，都必须能够审查代码，真正看清发生了什么并对此感到放心。”

> “但最重要的是，我看了看代码库，它居然有大约五十万行代码。”

于是 Cohen 开始着手解决这些问题。此时，有一个问题值得思考。从架构上看，OpenClaw 到底是什么？

“在核心层面，”Cohen 回答道，“你有一个编码智能体。它可以编写代码并运行 Bash 命令，因此你需要一个持久的环境会话。然后你需要连接到一个即时通讯应用。第四点是连接到互联网。”

Cohen 继续说道。“从这 4 种基本能力出发，你就可以构建出其他所有功能。而且它是主动的，因为一旦你开始调度任务，它就不再只是对你所说的话做出反应。你最少只需 25 行代码就可以写出一个 claw 智能体。”

虽然 OpenClaw 显然还没有做好面向大众的准备，但 Cohen 坚持自己对主要问题的结论。“我认为它从一开始就存在根本性的缺陷，而致命的缺陷就是那五十万行代码。”（快速查询证实，OpenClaw 现在拥有超过 80 万行代码）。即使在 [OpenAI 收购 OpenClaw](https://x.com/sama/status/2023150230905159801?lang=en) 之后，易用性问题依然存在，社区也在不断萎缩。

那么 NanoClaw 是否简化了安装设置？Cohen 在回答前停顿了一下。“在很大程度上，我仍然认为 NanoClaw 适合技术人员——不一定是开发者，而是那些熟悉终端和 GitHub 的人。NanoClaw 并不以二进制文件形式交付。每个人都是通过源码运行它。”

新版本现在有了一个基于终端的设置脚本，可以引导用户一步步操作。任何安装问题都会交由 Claude 来修复。Cohen 对此并不完全满意；让 Claude 参与其中会带来安全隐患。“如果你不了解安全模型，只是盲目运行 Claude，它可能会破坏安全模型或移除沙箱限制。”

说到这里，为什么 NanoClaw 选择使用容器？这一决定现在是否奠定了它的市场地位？

## 为什么容器最终胜出

“当我真正坐下来编写 NanoClaw 时，我只是从一个空项目开始，我没有让 Claude Code 去参考 OpenClaw，”Cohen 回答道。“我只是描述了所需的功能。我说我需要一个即时通讯应用、一个编码智能体；我希望它处于循环中，并且我需要内存，然后我从头开始构建了它。”

> “我不能让它直接在我的机器上运行，并允许一个自主智能体做任何事情，所以我把它放在了一个隔离的容器中。”

当 Cohen 开始动手时，新的考量也随之产生。“我想：我应该给它提供哪些工具？我不想让它受到限制而无法运行 bash 命令，但如果我允许它运行任何命令，它就必须处于一个隔离的环境中。我不能让它直接在我的机器上运行，并允许一个自主智能体做任何事情，所以我把它放在了一个隔离的容器中。”在那个时候，容器的使用赋予了 NanoClaw 独特的卖点。

Cohen 在这里透露了一个真正令人惊讶的消息：“我们最初使用的是 Apple 容器。”

真的吗？“我当时在一台 Mac mini 上运行它，隐约知道 Apple 增加了这种新的原生容器功能。最初效果非常好，但当成千上万的人开始使用它时，大家都纳闷：‘你用 Apple 容器干什么？Docker 才是默认选择，它是一个随处可用的标准。’所以我对两者都提供了支持，然后在几周内我就把默认设置改成了 Docker 容器。”

NanoClaw 与 OneCLI 合作，使用其凭据和代理层来增强安全性。但这带来了一些范式转变。“用户群和社区都理解这一点，”Cohen 说。“在我们达成该合作之前，存在一个我们试图防止的重大问题，即凭据进入智能体环境的问题——但是，例如，你需要配置好 Anthropic 的 token，以便智能体可以进行连接。”

起初 NanoClaw 创建了自己的代理解决方案，但这变得非常复杂。“与 OneCLI 的合作是非常自然的，用户群也乐于看到有一个妥善的解决方案。我们还引入了‘人机协同审批’机制，并针对——例如——智能体如何使用 Gmail 设置了策略。”

说到合作，在有关 NanoClaw 的讨论中经常会提到 Vercel。实际的集成是怎样的，它在路线图中又处于什么位置？

“Vercel 创建了名为 [Chat SDK](https://github.com/vercel/chat) 的优秀开源包，它能为你提供与大约 15 到 20 个不同即时通讯应用的标准化连接。我带给 NanoClaw 的核心理念之一就是‘不要重复造轮子’。为什么每个人都要重复实现与即时通讯应用的相同集成呢？让我们围绕一个大家共同拥护的库来做。”

我们可以看到，OpenClaw 现在已经自成一个软件工具类别。那么，NanoClaw 在接下来的 12 个月里需要做些什么，才能不仅仅是继承衣钵，而是重新定义这一类别？

“我们一直在通过凭据代理、人机协同审批以及在智能体隔离环境下进行构建，来正面迎击那些棘手的难题和挑战。这些问题和挑战阻碍了在确保安全的前提下，充分发挥自主智能体的全部威力。”

## 为企业而建

“在接下来的几个月里，我们正将目光投向那些具有严格安全要求、合规性要求、监管要求等的大型企业。因此，我们已经发展到了这样一个阶段：我们能够提供这样的方案，允许你连接到自己的电子邮件、日历，并在确保安全的的前提下进行操作，任何敏感操作都需要经过审批。”

显然，对于任何小型企业来说，进入企业级市场都是一个挑战。但 NanoClaw 很清楚自己的目标，因此在竞争对手争夺 OpenClaw 掉落的王冠之际，他们可能是最值得支持的“甲壳动物”（注：Claw 意为爪子，此处双关）。
