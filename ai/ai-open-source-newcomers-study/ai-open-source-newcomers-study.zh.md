关于AI会对开源产生什么影响，一直存在一种普遍的恐惧。人们担心编程代理会占用那些帮助新贡献者入门的简单任务；它们生成的代码更难维护，最终导致新维护者的来源枯竭。这听起来确实是一个合理的假设。但根据一项新的研究，这种情况似乎并未发生。

7月2日提交至 arxiv.org 的一项[北京大学的研究](https://arxiv.org/abs/2607.01810)追踪了1,888个采用AI编程代理（如 Cursor 和 Claude Code 等工具）的 GitHub 仓库，以观察这些项目在引入 AI 后发生了怎样的变化。研究人员将项目提交第一个代理配置文件（如 .cursorrules 或 CLAUDE.md）作为采用 AI 的时间点，并将其与一组未采用 AI 的对照项目进行了对比。

他们使用了双重差分法（difference-in-differences），这是区分工具带来的实际影响与项目既有趋势的行业标准方法。研究发现……结果并不明显。新人的参与度保持稳定或有轻微上升。在最保守的统计规范下，最坏的情况也仅发现了1.5%的下滑，且在统计学上并不显著。

## 复杂度上升，贡献者保持稳定

循环复杂度（即函数中独立路径的数量）在采用 AI 后，各编程语言中平均上升了3%到4%。认知复杂度（这是一个更复杂的指标，会对深度嵌套的逻辑和混乱的控制流进行严重扣分）在 Python 项目中跳升了约 11%。这听起来很糟糕，但如果与[去年发表的一项卡内基梅隆大学的研究](https://arxiv.org/abs/2511.04427)相比，该研究发现 Cursor 的采用导致了同一指标激增41%。北京大学的团队通过研究更大规模、更成熟的项目集并结合更严谨的统计控制，得出的结果仅为之前估算的四分之一左右。

这项研究真正有趣的地方在于，他们没有将复杂度与新人数量作为两个独立的发现，而是锁定了复杂度确实有所增加的128个 Python 项目进行深入分析。

在这些仓库中，新人的加入没有减少，留存率保持稳定，活跃贡献者基础甚至有所增长。这两个效应都是真实的，但它们之间似乎并没有关联。虽然 AI 生成的代码变得稍微复杂了一些，但这种额外的复杂度似乎并没有阻碍新人的参与，至少在本研究发现的水平上是这样。

> AI 生成的代码变得稍微复杂了一些，但这种额外的复杂度似乎并没有阻碍新人的参与，至少在本研究发现的水平上是这样。

## 研究的局限性

研究存在一些重要的注意事项。最大的问题是该研究主要聚焦于成熟的开源项目。近三分之二采用 AI 编程工具的仓库在创建之初就使用了它们，这使得研究人员没有足够的 AI 介入前的基准数据进行比较。这些仓库被排除在主要分析之外，分析重点转向了603个在引入 AI 之前至少有六个月历史的项目。

研究人员还检查了完整的数据集，其中新人的参与度似乎有所下降。但进一步观察发现，这些项目在采用 AI 之前就已经在流失贡献者了，因此无法将下降归咎于这些工具本身。

## 衡量采用率的盲点

此外还有另一个局限性。该研究是通过寻找与 Cursor 等工具相关的配置文件来衡量 AI 的采用情况，而不是追踪开发者实际使用的频率。这意味着它能告诉我们项目采用 AI 后发生了什么，但无法区分是深度依赖 AI 的团队，还是仅进行试验的团队，两者在结果上是否存在差异。

GitHub 表示，全平台的合并拉取请求（pull request）数量已从2023年初的每月约2500万增加到如今的每月约9000万。

GitHub 也开始做出回应。该公司最近限制了外部贡献者同时可以拥有的开放拉取请求数量，并推出了新的工具来帮助维护者梳理日益增长的审查队列。

> 所谓的“挤出效应”担忧，目前可以告一段落了。

## 这对我们意味着什么

所谓的“挤出效应”担忧，目前可以告一段落了。在成熟的开源项目中，以目前的采用水平来看，AI 代理并没有把新人拒之门外。然而，拉取请求的数量已经几乎翻了两番。这些拉取请求中包含的代码确实稍微复杂了一些。提交这些代码的人可能并不完全理解他们所提议的内容。而所有这些压力都落在了维护者身上，但维护者的队伍增长速度远赶不上贡献速度。

研究人员表示，未来的工作应该研究项目对 AI 编程工具的依赖程度，并寻找更好的方法来研究那些几乎是“AI原生”的仓库。这些都是重要的后续步骤。但最紧迫的问题或许不是：AI 是否改变了谁来贡献，而是它是否随着时间的推移改变了维护一个开源项目所需付出的努力。

> 最紧迫的问题或许不是：AI 是否改变了谁来贡献，而是它是否随着时间的推移改变了维护一个开源项目所需付出的努力。
