随着AI Agent的普及，云巨头正将计算单位从请求转向会话。通过会话级路由、隔离和生命周期管理，不仅解决了状态保持问题，更满足了生产环境对安全性与运行时的苛刻要求，标志着企业AI基础设施已进入精细化控制阶段。

> 译自：[AWS, Microsoft, and Google agree the session is the new unit of compute. They disagree on how to isolate it.](https://thenewstack.io/agent-session-aware-runtime/)
> 
> 作者：Janakiram MSV

**在过去几个月中**，四家AI巨头几乎同时悄悄重建了同一件事：AWS、微软、Google和Anthropic各自发布了Agent运行时更新，指向了相同的架构转型。

微软在四月围绕“按会话隔离”重建了其Foundry托管的Agent。Anthropic的[托管Agent](https://www.anthropic.com/engineering/managed-agents)将Agent虚拟化为会话、执行器（harness）和沙箱。AWS将每个会话路由到其专属的微型虚拟机（microVM），而Google则将Agent代码执行隔离在专用沙箱中。这种共同模式并非又一个模型功能或开发工具，而是从“请求级负载均衡”向“会话感知执行”的转变。

这一点至关重要，因为企业级Agent并非普通的API调用。它们是长期运行、有状态且会使用工具的进程，经常运行受用户输入影响的代码。本质上，Agent运行时正成为控制状态、身份、隔离和生命周期的控制平面，其调度单位已从单个请求转移到Agent会话。四大平台的这种趋同表明，会话感知执行正成为生产级Agent的基本要求。

## 传统的云扩展模型

要理解为什么这一点很重要，最好从传统模型做得正确的地方谈起。[NGINX](https://www.nginx.com/)和HAProxy层通常位于工作节点池的前面，将每个传入请求路由到下一个可用的后端。状态被刻意保存在其他地方（Redis或数据库中），因此任何工作节点都可以服务于任何请求。这种状态外置实现了弹性扩展、容错和基础设施更换，且不会干扰应用程序。“粘性会话”虽然可用，但企业通常将其视为有状态应用程序的例外，而非默认设置。

![](https://cdn.thenewstack.io/media/2026/06/a6e2d290-model-a-load-balancer-1024x753.png)

该模型基于两个假设：请求互不依赖，且任何后端都可以服务于任何请求。对于Web API和微服务而言，这些假设在近二十年来一直成立，并支撑了有史以来最大规模系统的扩展。Agent同时挑战了这两个假设，这就是为什么单纯依靠传统负载均衡已不再足够。

## 为什么Agent破坏了该模型

Agent带来了两个独特的挑战：在对话轮次间保持上下文状态，以及强制执行足够强大的安全边界以抵御不可信代码。只有前者可以通过路由解决。

## 为什么Agent状态无法池化？

想象一个处理退款的企业支持Agent。它读取订单、调用工具，然后等待模型响应。下一轮对话提出了澄清问题。如果该轮次被路由到另一个没有先前上下文访问权限的副本，Agent就会丢失完成工作流所需的状态。

> 负载均衡器开始表现得不像流量分配器，而更像会话路由器。

业界已经为这种权宜之计起了一个名字：会话亲和性（session affinity），且它已经承认了其代价。负载均衡方法越来越依赖一致性哈希，使用对话标识符来保持相关步骤处于热状态。简单来说，负载均衡器开始表现得不像流量分配器，而更像会话路由器。

## 为什么信任边界强制要求隔离

第二个挑战更为严峻，因为它不仅关乎状态，更关乎隔离。Agent可以执行由用户输入塑造的模型生成代码，因此后端必须被视为一个安全边界，而非通用的计算目标。共享内核无法提供企业安全团队所需的租户隔离。会话亲和性可以保持路由的连续性，但它本身无法为不可信执行提供按租户隔离。对于平台和安全团队而言，这就是调优后的负载均衡器不再适用的界限。

这不是理论上的担忧。当[Asana](https://www.upguard.com/blog/asana-discloses-data-exposure-bug-in-mcp-server)在2025年6月披露其MCP服务器的漏洞时，该服务器自5月1日起就已上线，窗口期约为五周。由于租户隔离检查失败，大约1,000个组织能够查看属于其他客户的项目数据。根据报告的细节，安全研究人员将问题追溯到一个验证了用户但未能持续强制执行缓存响应背后Agent和租户上下文的服务器。没有外部攻击者参与，数据却跨越了组织边界。该事件展示了为什么用户身份、Agent上下文和会话状态必须始终绑定在一起。

## 会话作为计算单位

一旦这种绑定成为正确性和安全性的要求，会话就成为了平台调度的单位。最明显的证据在于生命周期。一个会话现在拥有了前所未有的状态。AWS[文档](https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-how-it-works.html)显示，AgentCore会话在处理工作时保持“活跃（Active）”，在配置完成但等待时处于“空闲（Idle）”。在15分钟空闲超时或8小时最长生命周期后，它会变为“终止（Terminated）”。这是一个长期运行的执行环境的生命周期，而非短暂的HTTP请求。

![](https://cdn.thenewstack.io/media/2026/06/9a8ae380-model-b-control-plane-1024x518.png)

经济效益遵循同样的逻辑。当计费与活跃会话绑定时，并发性、空闲时间和Agent规格成为成本驱动因素，而非请求量。这给平台团队和财务部门增加了一个新的项目。更好的心理模型不再是传统负载均衡器，而是更接近虚拟Actor运行时，其中可寻址的身份按需实例化，在需要时保持活跃，在空闲时停用，且每个键对应一个活跃实例。

## 针对同一转变的四种方法

所有四个平台都在远离将有状态、不可信的Agent工作视为普通负载均衡流量的做法。关键区别在于每个供应商为隔离执行环境选择的计算原语。

AWS AgentCore是四者中最具特定偏见的。每个会话获得一个专用的[Firecracker](https://aws.amazon.com/blogs/aws/firecracker-lightweight-virtualization-for-serverless-computing/)微型虚拟机，具有隔离的计算、内存和文件系统，携带相同`runtimeSessionId`的请求通过会话头路由回该微型虚拟机。当会话结束时，微型虚拟机被终止，其内存被清理。

托管在Microsoft Foundry上的Agent通过一组不同的原语实现了类似的模型。平台按需创建按会话虚拟机隔离的沙箱，运行它并在会话结束时销毁它，没有副本计数，也没有需要调整大小的热池。每个Agent获得一个专用的Microsoft Entra身份，会话运行15分钟空闲超时，最长生命周期为30天。

Google [Agent Engine](https://cloud.google.com/vertex-ai/generative-ai/docs/agent-engine/overview)是最具启发性的混合体。其运行时将请求扩展保持在推理循环内——这是一个具有可配置最大/最小实例数量的托管环境，且`container_concurrency`默认值为9。即使Google在循环中保留了请求扩展，它也将不可信代码执行分离到一个隔离的代码执行沙箱中，并将对话状态外置到会话和内存库（Memory Bank）。即使是保留了负载均衡器的供应商，也不会将其指向有状态的不可信工作。

Anthropic提供了最清晰的架构分解。托管Agent将Agent虚拟化为三个组件：记录所有事件的会话、运行循环和路由工具调用的执行器，以及运行代码的沙箱。执行器成为一个近乎无状态的控制平面，而沙箱成为一个可调用、可重建的资源。 [Cloudflare](https://blog.cloudflare.com/claude-managed-agents/)的集成表明，底板可以解耦。Agent循环在Anthropic上运行，而每个工具调用在Cloudflare沙箱中运行，这可以是完整的微型虚拟机或轻量级的V8隔离。

正如表格所示，四个平台在路由和生命周期模型上趋同，而在执行底板上存在分歧。

| 平台 | 每个会话的计算原语 | 对企业买家的意义 |
| --- | --- | --- |
| AWS AgentCore | 专用Firecracker微型虚拟机，由会话ID路由，8小时上限 | 最具特定偏见的方案，隔离和路由融合在微型虚拟机中 |
| Azure Foundry托管Agent | 按会话虚拟机隔离沙箱，专用Entra身份，30天最大值 | 身份主导的隔离，适合较长的多日工作负载 |
| Google Agent Engine | 请求扩展循环加上独立的隔离代码执行沙箱 | 对推理进行负载均衡，对执行进行隔离的混合模式 |
| Anthropic托管Agent | 解耦的执行器和沙箱，循环背后的微型虚拟机或V8隔离 | 可移植性最强，底板被视为可替换层 |

没有单一架构是每个企业工作负载的正确答案。多日研究Agent适合Azure较长寿命的会话，代码密集型Agent需要微型虚拟机级别的隔离，而高容量自动化则倾向于更轻量的隔离。许多企业平台最终将结合不止一种模式。

## 应用程序仍拥有的绑定

平台可以隔离和路由会话，但它不知道是哪个人拥有它。AgentCore明确指出它不强制执行会话到用户的映射，因此应用程序后端必须维护用户及其会话ID之间的关系，并设置每用户会话限制。这就是Asana事件与企业架构相关的原因。

> 隔离得以维持，而用户到会话的绑定中断了。

在那起案例中，隔离得以维持，而用户到会话的绑定中断了。平台解决了隔离和生命周期问题，并将身份映射、授权和租户上下文交还给应用程序。对于企业买家而言，关键问题是谁拥有这种绑定，以及它如何在并发的多租户负载下进行测试。

## 这与粘性会话有何不同

一个合理的反对意见是，这看起来像是粘性会话与微型虚拟机底板上的自动扩展相结合。值得强调区别：传统的粘性会话通常是性能优化，而在Agent运行时中，会话绑定成为了正确性和安全性的要求。传统的负载均衡器将流量路由到现有的后端，并不拥有执行环境的生命周期。

新的控制平面在首次看到会话键时配置环境，将工作路由到它，并在空闲或生命周期限制后将其销毁。底层的原语实际上很古老，正是因为其古老，这种转变才清晰可见。AWS在2018年的re:Invent上开源了Firecracker，将虚拟机级别的隔离与容器速度相结合。我当时为 *The New Stack* [报道了](https://thenewstack.io/how-firecracker-is-going-to-set-modern-infrastructure-on-fire/)它，并在单台笔记本电脑上运行了一百多个微型虚拟机来亲身体验这种权衡。同样的微型虚拟机现在每月支撑着Lambda和Fargate数以万亿次的执行。这就是云架构进化的方式：现有原语围绕一个新的抽象单位重新组合。

> Agent会话正在成为企业AI基础设施的操作单位。

总之，重要的转变不是云提供商为Agent找到了更好的负载均衡器。而是Agent会话正在成为企业AI基础设施的操作单位。传统的负载均衡将独立请求路由到可互换的工作节点。而会话感知运行时将工作路由到隔离的环境，在轮次间保持状态，并拥有执行上下文的生命周期。分发层在Agent出现时并未消失，它变得具备了会话感知能力、状态感知能力和生命周期感知能力，使其成为控制平面而非传统的负载均衡器。

三个实际问题将决定未来走向，企业需要评估每一个问题。按会话计费的经济性将决定成本如何随并发性扩展。在Kubernetes上重建此路由模型的开源工作将决定其可移植性。微型虚拟机和轻量级隔离之间的争论将决定隔离与密度之间的权衡。对于构建Agent的企业而言，一个拥有隔离、路由和生命周期的运行时，将后台管道变成了深思熟虑的架构选择。它为平台、安全和运维团队提供了一个可以标准化、而非为每个工作负载重复构建的基础。
