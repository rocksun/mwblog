## 总结概述

* AI 生成的 Terraform 代码常出现四类重复性故障（权限过宽的默认设置、缺失的安全代码块、硬编码值以及陈旧的提供商模式），这些错误与现实中导致云泄露的配置错误如出一辙。
* 来自 NeurIPS 2024、FSE 2026 和 ICSE 2026 的同行评审基准测试显示，前沿大模型在真实 Terraform 任务中的通过率不足 40%，且相比 YAML 和 JSON，HCL 对 AI 而言更难处理。
* `terraform validate` 和 `terraform plan` 无法捕获这些故障，因为它们仅验证语法和状态增量，而非需要语义分析的安全态势。
* SonarQube 可覆盖 Terraform 从编写到合并的全过程：通过 SonarQube for IDE 提供 IDE 反馈，通过 SonarQube Cloud 或 SonarQube Server 进行拉取请求（PR）和合并检查，并通过 SonarQube Cloud 上的 Agentic Analysis 进行代理端验证。

根据 Sonar 2026 年的[代码开发者调查](https://www.sonarsource.com/resources/developer-survey-report/)，目前 42% 的已提交代码是由[人工智能代理](https://www.sonarsource.com/resources/library/what-is-an-ai-agent/)编写或辅助完成的，预计到 2027 年这一比例将增长至 65%。那些负责交付应用程序代码的 AI 编码代理，现在也在编写你的[基础设施即代码](https://www.sonarsource.com/resources/library/infrastructure-as-code/)：构建模块、生成资源块并即时填充 IAM 策略。

代理生成 IaC 的速度目前已超过了传统的人工审查和流水线所能承受的范围，配置错误进入生产环境的速度比以往任何时候都快。这引发了两个重要担忧：当 AI 代理编写你的 [Terraform](https://developer.hashicorp.com/terraform) 时，它会犯哪些错误？以及在错误进入生产环境之前，由谁来拦截？

## AI 犯错的频率有多高？

高到足以引起重视。NeurIPS 2024 的 [IaC-Eval 基准测试](https://proceedings.neurips.cc/paper_files/paper/2024/hash/f26b29298ae8acd94bd7e839688e329b-Abstract-Datasets_and_Benchmarks_Track.html)在 458 个由人工策划的 AWS Terraform 场景中对 GPT-4 进行了评估。其 *Pass@1* 准确率仅为 19.36%。同一模型在 EvalPlus 的同等 Python 任务中得分 86.6%，这意味着它在基础设施代码上的表现比在应用程序代码上差四倍。

在更新的模型上，这种模式并未扭转。FSE 2026 接收的 [DPIaC-Eval](https://arxiv.org/abs/2506.05623) 研究在 153 个真实 IaC 任务中测试了六个前沿 LLM。首次部署成功率仅在 20.8% 到 30.2% 之间。在标准 Checkov 策略检查下的合规率甚至更低，仅为 8.4%。该基准测试虽以 [CloudFormation](https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/Welcome.html) 为主，但其底层逻辑（训练语料库过时、缺乏意图上下文、默认配置权限过宽）同样适用于各种 IaC 格式。

ICSE 2026 接收的 [TerraFormer](https://arxiv.org/abs/2601.08734) 论文测试了 17 个前沿 LLM，并解释了为何 Terraform 比其他 IaC 格式受到的冲击更大：与 Kubernetes、CloudFormation 和 Ansible 使用的 YAML 和 JSON 语法相比，HCL 在 LLM 训练数据中的普及度较低，因此代理能参考的正确 Terraform 示例较少。

生成看起来“正确”的 Terraform 是一回事，生成安全、最新且没有虚假引用的 Terraform 则是另一回事。在这些研究的交汇处，存在着一组可识别的模式。

## AI 在 Terraform 上犯错的四种方式

以下四种故障模式中的每一种都指出了代理“做了什么”，而不是最终的 [CVE](https://www.sonarsource.com/resources/library/common-vulnerabilities-exposures/) 分类（这是行为的下游结果）。

### 1. “盲目使用 *”的问题

当 AI 代理无法推理团队预期的访问权限时，它通常会默认选择权限过宽的设置：通配符 IAM、公开的 S3 ACL、对 SSH 或 RDP 开放 `0.0.0.0/0` 的安全组等。限制性配置需要提示词中很少提供的边界上下文；而权限过宽的默认设置可以满足即时请求并被交付。

生成的实际模式：

```
# Generated by an AI assistant: it looks reasonable, but isn't
resource "aws_iam_policy" "app_policy" {
  name = "app-service-policy"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = "*"
      Resource = "*"
    }]
  })
}
```

代理本应编写的受限版本：

```
resource "aws_iam_policy" "app_policy" {
  name = "app-service-policy"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = ["s3:GetObject", "s3:PutObject"]
      Resource = "${var.bucket_arn}/*"
    }]
  })
}
```

在我们的示例中，SonarQube 通过规则 S6304 捕获此类问题，该规则会标记授予账户所有资源访问权限的策略。

### 2. “静默遗漏”的问题

有时，AI 生成的资源看起来很完整，但却跳过了与安全相关的代码块：例如没有 `storage_encrypted` 的 `aws_db_instance`，没有 `logging_config` 代码块的 `aws_cloudfront_distribution`，或没有加密配置的 Azure 存储账户。在这种情况下，资源确实启动了，但保护措施并未生效。加密和日志记录对于“资源是否存在”并不起核心支撑作用，因此很容易被忽略，除非审查者知道要查看什么，否则在代码审查中很难发现这种遗漏。

生成的模式：

```
resource "aws_db_instance" "primary" {
  identifier        = "app-db"
  engine            = "postgres"
  instance_class    = "db.t3.medium"
  allocated_storage = 100
  username          = "appadmin"
  password          = var.db_password
}
```

安全等效项：

```
resource "aws_db_instance" "primary" {
  identifier        = "app-db"
  engine            = "postgres"
  instance_class    = "db.t3.medium"
  allocated_storage = 100
  username          = "appadmin"
  password          = var.db_password
  storage_encrypted = true
  kms_key_id        = aws_kms_key.db.arn
}
```

在我们的示例中，SonarQube 规则 S6303 捕获未加密的 RDS 资源。姊妹规则 S6258 则捕获 AWS、Azure 和 GCP 服务中禁用日志记录的变体。

### 3. “硬编码一切”的问题

代理将以下内容硬编码在 `*.tf` 或 `*.tfvars` 中并不罕见：字面量密钥、看起来足够真实以至于被提交的虚假凭证，以及团队本应使用变量或数据源的地方却使用了魔术数字和内联 CIDR。根据我们的经验，当提示词未提供变量脚手架时，代理就会倾向于使用字面量。一旦硬编码的凭证进入 Git 历史记录，它就会一直存在。

生成的模式：

```
resource "aws_db_instance" "primary" {
  identifier = "app-db"
  username   = "appadmin"
  password   = "P@ssw0rd-Summer-2026!"
}
```

从密钥管理器获取的安全模式：

```
data "aws_secretsmanager_secret_version" "db_creds" {
  secret_id = aws_secretsmanager_secret.db.id
}

resource "aws_db_instance" "primary" {
  identifier = "app-db"
  username   = "appadmin"
  password   = data.aws_secretsmanager_secret_version.db_creds.secret_string
}
```

Sonar 的[密钥检测](https://www.sonarsource.com/solutions/secrets-detection/)功能可标记项目中源文件里的硬编码凭证，Sonar 的 IaC 分析也包含其自身的凭证检测规则。两者均可在 [SonarQube Server](https://www.sonarsource.com/products/sonarqube/server/) 和 [SonarQube Cloud](https://www.sonarsource.com/products/sonarqube/cloud/) 中使用。

### 4. “训练数据陈旧”的问题

AI 代理有时会输出一两年前正确、但现在已被弃用、重命名或移除的属性名称和资源参数。AWS 提供商 v4 到 v5 的过渡就是典型的例子：在 v4 中，可以在 `aws_s3_bucket` 上内联设置 ACL；而 v5 将其拆分为专门的资源类型并移除了内联属性。

以 v4 风格生成的模式：

```
# Generated by an AI assistant: works on AWS provider v4, broken on v5+
resource "aws_s3_bucket" "app_data" {
  bucket = "my-data-bucket"
  acl    = "private"
}
```

当前的 v5+ 模式：

```
resource "aws_s3_bucket" "app_data" {
  bucket = "my-data-bucket"
}

resource "aws_s3_bucket_acl" "app_data" {
  bucket = aws_s3_bucket.app_data.id
  acl    = "private"
}
```

TerraFormer 解释了这一机制：训练语料库会老化，而 Terraform 提供商不会。

## 为何你现有的检查手段会错过这些问题

`terraform validate` 会检查你的 HCL 在语法上是否有效且内部是否一致。[HashiCorp 的文档](https://developer.hashicorp.com/terraform/cli/commands/validate)明确指出：“它不验证远程服务，例如远程状态或提供商 API。” [`terraform plan`](https://developer.hashicorp.com/terraform/cli/commands/plan) 会预览提供商将要应用的状态增量，但它不会评估生成的配置是否安全或合适。

一个语法完美的 `aws_s3_bucket` 如果配置了公开的 ACL，它会通过这两项检查。带有 `Action = "*"` 的 `aws_iam_policy` 也是如此。没有加密代码块的 `aws_db_instance` 同样会通过。本文强调的四种故障模式构成了这些工具设计上留下的缺口，要捕获它们，需要进行能够理解配置“含义”的分析，而不仅仅是检查它是否符合语法。

## SonarQube 作为 IaC 验证层

SonarQube 的[基础设施即代码分析](https://www.sonarsource.com/solutions/infrastructure-as-code/)涵盖了 [Terraform](https://www.sonarsource.com/knowledge/languages/terraform/)、Azure Resource Manager 和 CloudFormation，并在 [SonarQube Server](https://www.sonarsource.com/products/sonarqube/server/) 和 [SonarQube Cloud](https://www.sonarsource.com/products/sonarqube/cloud/) 中分析相邻的平台和流水线配置（如 Ansible、Docker 和 Kubernetes）。分析器将你的 HCL 解析为抽象语法树（AST），并对 AWS、Azure 和 GCP 的配置错误模式进行了广泛覆盖，包括上述四种故障模式。规则库会随着新提供商版本的发布而维护（最近在 [SonarQube Server 2026.1 LTA](https://docs.sonarsource.com/sonarqube-server/server-update-and-maintenance/release-notes) 中有所扩展），这就是检测能力如何跟上故障模式 4 背后的模式漂移。

相同的规则在四个接触点运行：

* **在 IDE 中：** [SonarQube for IDE](https://www.sonarsource.com/products/sonarqube/ide/) 会在编写 Terraform 时标记问题。
* **在代理会话中：** SonarQube Cloud 上的 [Agentic Analysis](https://docs.sonarsource.com/agent-centric-development-cycle/features/agentic-analysis) 允许 AI 代理在代码到达 PR 之前，在代理模式和 CLI 中让 SonarQube 检查其工作。
* **在拉取请求中：** SonarQube Cloud 和 SonarQube Server 会在 GitHub、GitLab、Azure DevOps 和 Bitbucket 的 PR 中显示规则发现的问题。
* **在合并门禁（Merge Gate）处：** 质量门禁决定通过或失败。如果你将 SonarQube Cloud 与 Terraform Cloud 一起使用，Run Task 集成会在预计划（pre-plan）阶段检查最新的门禁状态。

那份显示 AI 贡献了 42% 代码的 Sonar 调查还发现，SonarQube 用户报告称，由 AI 生成代码引起的停机事故的可能性降低了 44%。这就是持续的质量门禁带来的好处：每次涉及 Terraform 的提交都会运行相同的规则并面临相同的合并决策。

## 应用于基础设施的以代理为中心的开发周期

AI 代码生成工具的发展速度极快，以至于传统的人工 IaC 审查和 CI/CD 流水线难以跟上传入变更的数量，这需要一个能够随产生错误即时捕获错误和配置问题的验证层。

这就是[以代理为中心的开发周期](https://www.sonarsource.com/blog/the-future-is-ac-dc-the-agent-centric-development-cycle/)在基础设施方面的实践。无论 AI 代理是输出了通配符 IAM 策略、遗漏了加密块，还是内联了硬编码凭证，错误都会在源头被捕获，并在进入生产环境之前得到纠正。
