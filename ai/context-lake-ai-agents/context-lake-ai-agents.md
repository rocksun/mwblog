AI智能体因安全限制、工具过载及缺乏企业常识而难以规模化落地。引入“上下文湖”（Context Lake）作为统一的企业知识层，能让智能体真正理解组织关系与业务上下文。

> 译自：[Why AI agents need a Context Lake](https://thenewstack.io/context-lake-ai-agents/)
> 
> 作者：Zohar Einy

你的 AI 智能体已经拥有了工具访问权限。但它所缺失的，是让这些工具发挥作用的知识。

你可能已经将 Claude Code 连接到了你的 GitHub，或者你的 Jira。它确实起作用了——对你而言。但当有人问起：“我们能把它推广到整个团队吗？”时，你才意识到，有多少事情是依赖于只有你才掌握的上下文才维系在一起的。

在整个组织中扩展 AI 智能体很快就会遇到三堵墙：安全审批受阻、过多的 MCP 工具挤爆上下文窗口，以及你的智能体仍然无法回答诸如“谁负责这个服务？”之类的基本问题。

![连接到多个工具的 AI 智能体示意图，这使得回答简单问题变得令人困惑。](https://cdn.thenewstack.io/media/2026/05/2f1795c7-1-1024x536.png)

**连接到多个工具的 AI 智能体，这使得回答一个简单的问题变得非常混乱。**

## 安全性阻碍了你

你可能遇到的第一个障碍是安全性。在大多数组织中，MCP 服务器在投入使用之前，需要对每个数据源进行审查和法律批准。一些客户告诉我们，他们为此设立了 AI 委员会。在最好的情况下，批准新模型和 MCP 服务器需要几个月的时间；在最坏的情况下，它们会完全被封锁。

> *“我们花了 9 个多月的时间才让 GitHub Copilot 在内部组织中获得批准使用。MCP 服务器完全被禁用了。”*
>
> —— 某大型金融机构的平台负责人

当你确实获得了引入 MCP 的批准时，你自然希望你的智能体更聪明、连接更多数据，因此你会启用尽可能多的工具。结果，智能体最终配置了 10 多个 MCP 服务器和数百个 (MCP) 工具。但是，如果每个工具定义都被加载到上下文窗口中，你最终将面临成本飙升和延迟增加的问题。

Anthropic 的工程团队[记录了这一问题](https://www.anthropic.com/engineering/code-execution-with-mcp)，并估计智能体仅在加载 MCP 工具定义时就消耗了 150,000 个 token。你甚至可能在 Cursor 或其他 IDE 中亲自见过这种情况，它们会显示有关响应质量下降的错误和警报。

![Cursor IDE 截图，显示关于拥有过多 MCP 工具的警告，这可能会降低性能。](https://cdn.thenewstack.io/media/2026/05/b10d59ec-2-1024x536.png)

*Cursor IDE 显示关于拥有过多 MCP 工具的警告，这可能会降低性能。*

## 准确性失效

假设你已经解决了前面安全和工具过载这两个障碍，你的智能体仍然无法回答一些基本问题。

如果你试着问你的智能体：*“我有哪些开启的 PR（拉取请求）？”*

* 它不知道“我”指的是谁。
* 它不知道“开启”是指草稿、已批准，还是仅仅未合并。
* 它不知道应该搜索哪些代码库或组织。
* 它可能会花几分钟查询不同的代码库，却仍然漏掉了一半。

或者，如果你尝试提问 *“这个服务的部署状态是什么？”，它将无法得知以下问题的答案……*

* 什么是服务？
* 什么是部署？
* 哪条流水线（pipeline）代表它？你所有的流水线名称都很相似。

因此，智能体只能求助于猜测。当然是有根据的猜测，但仍然是猜测。

你不会雇佣一个新人，并指望他们能立刻知道所有这些问题的答案。智能体也是如此。它们和人类一样，都需要一个“入职培训”的过程，以理解你公司的专业术语和工作方式。

## AGENTS.md 无法扩展

你可能已经尝试过 [AGENTS.md](https://agents.md/) 或类似的方法——一个向智能体解释你代码库的 markdown 文件。对于单个代码库，这确实有效。但如果你的公司有 1,000 个代码库呢？你如何保持一切同步？这些规则是否同样适用于所有团队？你如何在那种规模下管理上下文？

这就是[上下文工程 (context engineering)](https://www.port.io/glossary/context-engineering)的用武之地——大规模管理、策划并向 AI 系统传递正确的上下文。如果你已经为开发者解决了这些问题，那么你也可以为 AI 智能体解决它们。

目前缺失的是一个上下文湖（Context Lake）——一个供你的智能体查询的组织知识层。它不仅提供工具访问，还提供理解力：包括所有权、依赖关系，以及在当前环境下“生产就绪（production-ready）”意味着什么。

## 什么是上下文湖（Context Lake）？

你需要在智能体和工具之间建立一个层——一个保存着原始 API 访问无法提供的知识的层。谁拥有这个？什么依赖于它？在这里“生产就绪”到底意味着什么？

这就是上下文湖：一个统一的组织知识层，AI 智能体可以对其进行查询。

你的智能体拥有 GitHub 的访问权限。它可以列出每一个代码库。但如果问它“支付（Payments）团队拥有哪些服务？”它就会卡住——因为这种关系并不存在于任何 API 中。MCP 赋予了智能体工具，而上下文湖则赋予了它们理解力。

> “MCP 赋予了智能体工具，而上下文湖则赋予了它们理解力。”

可以把它想象成一个新入职的工程师在最初几个月里所学到的一切，只是它被结构化了，既适用于机器，也适用于人类。

你可能已经有了一个服务目录（service catalog）。但目录是供人类浏览的，而上下文湖则是为智能体查询而构建的——它为程序化访问而进行了结构化，而不仅仅是文档。

## 使用场景

*“谁负责这个服务？谁在值班？如果凌晨 2 点有东西坏了，会呼叫谁？”*

没有上下文湖，你的智能体将一无所知。有了上下文湖，它就知道支付服务属于支付（Payments）团队，Sarah 这周值班，并且故障升级会发送到 #payments-incidents Slack 频道。

*“如果我修改这个，会有什么东西损坏？哪些服务在使用这个 API？哪些下游系统依赖这个数据库？”*

在做出更改之前，你的智能体需要知道“爆炸半径”（blast radius，即波及范围）。上下文湖显式地映射了这些关系，因此它可以在触碰任何东西之前，回答“还有什么会坏掉？”。

> “在做出更改之前，你的智能体需要知道爆炸半径。上下文湖显式地映射了这些关系。”

*“什么是服务？GitHub 代码库是如何变成服务的？部署（deployment）和发布（release）之间有什么区别？”*

每个组织都有自己的术语。[上下文湖将原始数据转换](https://thenewstack.io/better-context-will-always-beat-a-better-model/)为你们公司的语言。GitHub 代码库不仅仅是一个代码库，它被映射到一个服务，由一个团队拥有，并在特定的环境中运行。

*“这有多关键？它会影响谁？我该如何确定优先级？”*

你的 AI 智能体无法理解“生产就绪”在你们组织中意味着什么。它不知道 checkout-service 每天要处理 200 万美元的交易并需要 P1 级别的响应时间，而 internal-tools 可以等到周一再处理。

上下文湖捕获了这种[业务上下文](https://thenewstack.io/how-precog-adds-business-context-to-make-enterprise-data-ai-ready/)：收入影响、客户层级、SLA 要求和合规范围。现在，智能体和人类都可以根据真正重要的事情来确定优先级。

## 这将使什么成为可能

有了上下文湖，你不会再因为提问方式的不同或者连接工具的不同而得到不同的答案。现在，你可以获得一致且确定的答案。提问“我们团队有哪些开启的 PR？”时，每次都能得到同样可靠的回答，因为这些关系是被明确定义的，而不是推断出来的。

在 VS Code 中，开发者[使用 GitHub Copilot](https://thenewstack.io/microsoft-makes-github-copilot-free-in-vs-code/) 来了解某项修改的爆炸半径。智能体通过查询上下文湖来寻找下游依赖项。同一位开发者还可以检查哪些库是经过他们团队批准的，或者接下来应该承接哪些任务。

![智能体在有和没有上下文湖的情况下回答查询的截图，展示了在有上下文湖时更可靠的输出。](https://cdn.thenewstack.io/media/2026/05/3a357981-4-1024x536.png)

*有了上下文湖，AI 智能体可以可靠且准确地回答问题。*

这开启了以前无法实现的全新工作流：

* PR 评审路由 —— 智能体根据组件所有权、贡献者可用性以及代码库中的近期活动来分配评审人员。不再需要猜测谁最了解这段代码。
* 日程规划器 —— 新入职的工程师询问：“我应该做点什么？”智能体会将他们团队的活跃事件、Jira 工单、紧急 Bug 和失败的监控聚合到一个排序好的视图中。
* 事件分诊 —— 智能体收到警报后，会检查服务关键性和 SLA 层级，呼叫正确的轮值人员，并开启一个附带相关运行手册（runbook）的 Slack 频道。
* AI 智能体任务委派 —— 当你将任务移交给 AI 智能体时，它能获取到完整的视图：业务优先级、衡量进度的记分卡、历史相关任务以及它需要达到的标准。

## 它是如何工作的

随着 AI 越来越多地融入我们的工作，我们重新设计了 Port 的核心支柱，以便同时为人类和 AI 智能体服务。

![展示 Port 上下文湖架构的图表，它由集成构建，包含护栏、工具和内存。](https://cdn.thenewstack.io/media/2026/05/5712442f-5-1024x536.png)

*展示 Port 上下文湖架构的图表，该架构基于集成构建，具有安全护栏、工具和内存。*

Port 的上下文湖从[集成](https://docs.port.io/integrations-index)开始。数据从 GitHub、Jira、AWS、PagerDuty 和数十种其他工具流入 Port。然后，你将其映射到你的业务术语中：GitHub 代码库变成了一个服务，Jira 项目变成了团队的待办任务（backlog），而 AWS 资源变成了环境的一部分。这把一切紧密地联系在一起。你为服务、团队、环境和部署定义蓝图，然后在它们之间建立关系。当你查询一个服务时，每次都会得到相同的答案：谁拥有它、什么依赖它以及它是否符合你的标准。

为了跟踪你组织的 SDLC（软件开发生命周期）标准，我们会跟踪你的记分卡。你可以定义“生产就绪”的含义、哪些服务符合安全要求以及要达到的目标。

为了控制哪些智能体可以查看或执行什么，我们使用与人类用户相同的访问控制，因此它们只能看到发起请求的用户有权看到的内容。

人类通过仪表板和 Port UI 进行交互。AI 智能体则通过 [Port MCP 服务器](https://docs.port.io/ai-interfaces/port-mcp-server/overview-and-installation)或 API 进行交互。两者都从同一个单一事实源中获取信息。

## 下一步是什么

我们正在与客户紧密合作，以了解[他们的智能体还需要什么](https://thenewstack.io/the-architects-guide-to-understanding-agentic-ai/)。

构建上下文湖仍然需要付出努力。我们正在添加自我发现功能——让 AI 自动发现数据中的关联并建议如何对其进行建模。我们的目标是构建一个能够根据你的反馈和使用模式进行自我构建的上下文湖。

此外，我们意识到并非所有数据都属于上下文湖。临时数据（如日志、Slack 消息和长文档）会不断变化，不需要进行持久建模。然而，这些数据对于某些特定场景依然非常有用。我们正在尝试使用只读操作等解决方案，并探索通过 MCP 连接器来帮助处理此类数据。
