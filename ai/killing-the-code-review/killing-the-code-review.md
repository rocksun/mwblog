本文指出传统代码审查已无法应对 AI 生成代码的高产出。作者建议将代码审查拆分为“意图对齐”与“标准检查”两部分，通过捕获意图与建立不变量（Invariants）替代人工读码，从而实现代码审查的转型。

> 译自：[How to kill the code review](https://thenewstack.io/killing-the-code-review/)
> 
> 作者：Ankit Jain

*本文是 [“我们还需要多久才会停止阅读代码？”](https://thenewstack.io/future-of-code-reviews/) 一文的后续。该文指出，在 AI 加速的软件开发生命周期（SDLC）中，传统的代码审查已不再是一个可行的质量门禁。接下来的自然问题是：如果不进行代码审查，那该怎么办？*

在 AI 编程工具出现之前，代码审查就已经出现了问题。但由于编写代码的速度仍然慢于阅读代码的速度，我们可以假装一切正常。

随着 AI 代理的出现，这种平衡被打破了。它们可以在我写下这句话的这段时间内[编写数千行代码](https://thenewstack.io/5-hacks-of-kindness-learned-by-writing-thousands-of-lines-of-iac/)。

今年 4 月，Faros 在一份[报告](https://www.faros.ai/blog/ai-software-engineering)中分享了他们对 22,000 名开发者进行监测后的观察结果：

* 代码流失率（删除与添加之比）上升了 861%
* 每个 PR（拉取请求）的事故率上升了 243%
* 审查时间上升了 441%。
* 有 31% 的 PR 在没有任何审查的情况下被合并了。

> “那些能在比我写下这句话还短的时间内编写数千行代码的代理，彻底打破了现有的平衡。”

工程师们并非决定停止审查，而是他们实在跟不上了！对于日益庞大的代码量，审查实际上已经无法进行。现在的问题不是是否要终结代码审查，而是用什么来取代它。

## 代码审查不仅仅是审查代码

代码审查从来都不只是一件事。在过去的十五年里，它成为了团队协作的基础——知识得以传播，初级工程师学习高级工程师的思维方式，架构受到审视，所有权得以共享。它同时也起到发现 Bug 和执行规范的作用。

代码审查实际上有两个任务。

第一个任务是**对齐（Alignment）**。这是知识在团队中传递的过程，初级工程师在这里观察高级工程师的思考方式，架构决策在这里受到质疑，共同的所有权在这里形成。它是团队就“正在构建什么”以及“为何构建”达成一致的地方。

第二个任务是**标准检查（Standards-checking）**。包括捕获 Bug、执行规范、标记安全问题，以及验证输出是否符合团队对代码本身的期望。

这两个任务在外部看起来是一样的——它们都发生在 PR 中，且都涉及某人阅读代码并留下评论。但它们在结构上是不同的问题，需要不同的解决方案。将它们混为一谈，就是为什么关于 AI 代码审查的讨论总是陷入循环的原因。那些想要保留审查的人通常是在试图保护“对齐”；而那些希望通过自动化解决问题的人，通常是在解决“标准”问题。

## 任务一：对齐

对齐是更难的问题，也是 AI 无法解决的问题。当人类编写代码时，意图会伴随作者通过审查过程。他们可以解释自己考虑了什么、拒绝了什么，以及在何种约束下工作。即使不言明，这种背景信息也是可获取的。

> “代码审查实际上有两个任务。第一个任务是对齐。第二个任务是标准检查。”

当 [AI 代理编写代码](https://thenewstack.io/ai-agents-software-engineering/)时，意图存在于未保存的提示词（prompt）中，或是存在于无法记录决策过程的工单（ticket）中，抑或仅仅存在于工程师的脑海中；只有实现结果被保留了下来，而背后的推理过程却丢失了。

对此，流行的应对方式是规范驱动开发（spec-driven development）：在生成开始之前完整地写下意图，将规范作为事实的单一来源。直觉是正确的。然而，规范是在工作开始前编写的，此时你对工作还一无所知，因此它在很多方面都是错误的。实现过程会浮现出一百个规范从未预料到的微小决策，而这些决策从未回到文档中。

意图并不存在于规范中。它存在于你的 JIRA 工单、设计文档，以及最重要的——你的提示词中。与 AI 代理的反复交流是一系列真实决策的实时流。规范是在实时生成的，并在会话结束时被丢弃。

[意图驱动开发（Intent-driven development）](https://www.aviator.co/verify?utm_source=tns&utm_medium=content&utm_campaign=q3-2026-tns-verify&utm_term=net-new&utm_content=awareness)意味着在决策实际发生的地方捕捉它，而不是在工作开始前编写的文档中，而是在做出决策的会话中。

捕捉意图比听起来容易。一段简短的范围描述、一份验收标准列表，或者关于明确不在范围内的说明。这些内容通常可以直接来自提示词——工程师在与代理交流时做出的决策已经存在于那里了。关键在于要在提交变更之前将其捕捉下来，而不是让它们流失。

审查者的工作发生了变化。他们不再是阅读 600 行差异（diff）并询问“这看起来对吗？”，而是阅读八行意图并询问“我们是否用正确的约束解决了正确的问题？”。这对高级工程师的时间利用率有了根本性的提升。这也是审查的知识共享功能得以保留的地方。阅读验收标准的审查者是在阅读实现背后的决策，而不是实现本身。

## 任务二：AI 垃圾记录册

标准检查是每个人都想自动化的部分，而这恰恰是 AI 代码审查表现最差的地方。给 LLM 一个带有清单的差异检查表并不能构建质量门禁。当同一个模型既编写代码又审查代码时，它们拥有相同的盲点。验证必须在结构上与代码生成分离。

每个代码库都有 AI 持续出错的模式。这些评论在代码审查中反复出现，因为没有机制使它们成为永久性检查。每一条重复出现的审查评论，都是一条尚未编写的护栏。

> “每一条重复出现的审查评论，都是一条尚未编写的护栏。”

我们称之为[不变量（Invariants）](https://docs.aviator.co/verify/concepts/invariants?utm_source=tns&utm_medium=content&utm_campaign=q3-2026-tns-verify&utm_term=net-new&utm_content=awareness)，或者是你工程团队的 [AI 垃圾记录册（AI slop register）](https://thenewstack.io/engineering-ai-slop-registry/)。这是你团队不断纠正的模式清单。当一位高级工程师第四次输入相同的评论时，该评论就应该进入记录册。然后它就变成了一个不变量：一个在每次变更时自动运行的常设检查。记录册本身也会自动构建：每周一次，它会阅读合并 PR 上的评论并提出新的不变量。

当团队提取最近的一百条 PR 审查评论并进行分类——确定性的、需执行测试的、纯判断性的——比例往往落在 45/30/25 左右。四分之三的审查反馈是可以编码的。一旦检查编写完成，它就再也不需要人工审查了。

对于剩下的 25%，则保留了纯粹的判断。例如基于上下文的模式识别、架构问题，以及那些看起来不对但规则无法解释原因的情况。这时，LLM 作为有范围的后备方案就有了用武之地：一个与实现代理没有共享上下文的独立验证代理，通过约束提示词，并输出带有文件引用和每个发现点推理的结构化报告。能确定性的就确定性，必须用 LLM 的地方才用 LLM。

## 先构建记录册，不要第一天就终结审查

取代代码审查的系统运行在两个层面上。

第一层捕捉意图：它观察会话，将决策转化为验收标准，并让人类确认。第二层进行语义分析：它获取这些标准，添加来自你的垃圾记录册的不变量，构建测试计划，并以确定性的方式运行它。这两个层面共同驱动了新型的“代码审查”——人类工程师现在审查的是意图、决策和结论，而不是代码差异。

这两个层面回答了传统代码审查试图回答的问题：这个实现是否完成了它应该做的事情，它是否达到了团队的标准？

请记住，转型不会在一夜之间发生。第一个月，团队很可能会觉得在做双倍的工作：既要审查代码，又要[构建不变量](https://docs.aviator.co/verify/concepts/invariants?utm_source=tns&utm_medium=content&utm_campaign=q3-2026-tns-verify&utm_term=net-new&utm_content=awareness)。要与团队设定明确的预期：构建记录册需要时间。到了第二个月，记录册捕获的内容足以让审查工作量缩减。到了第三个月，审查者读的是意图，而不是代码。

审查并没有被终结。它被改造了。消失的是那些本来就做得不好的部分：某人在下午 4 点潦草阅读 500 行差异，凭记忆对比代码库中的模式，写下六个月前写过的同样的评论。留下来的是自始至终都需要人类判断的部分。[用已验证的意图取代代码审查。](https://www.aviator.co/verify)
