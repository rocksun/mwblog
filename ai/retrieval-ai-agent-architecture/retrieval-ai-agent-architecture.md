本文指出 AI Agent 的诸多性能问题实则源于检索而非生成阶段。作者强调，构建高质量上下文是 Agent 架构的核心挑战，需通过链路追踪（Tracing）和精细化评估来优化多阶段检索流程，从而提升整体系统的准确性与可靠性。

> 译自：[Why retrieval quality is becoming the defining challenge in AI agent architecture](https://thenewstack.io/retrieval-ai-agent-architecture/)
> 
> 作者：Isaac Flath

AI Agent 系统通常有两个核心任务：构建上下文，然后利用这些上下文来生成答案或执行操作。

许多看起来像是 LLM 的问题，其实根源在于上下文构建阶段。LLM 给出的答案受限于它所获取的上下文，或是通过工具调用找到的内容。如果 Agent 模型无法找到正确的来源，那么无论如何优化生成模型，都无法改善整个系统的表现。

一家客户 [Specstory](https://specstory.com/) 希望让用户能够基于 Agent 的历史记录提问。例如，询问团队为何选择 Authlib 作为身份验证方案，以及他们曾考虑过哪些替代方案。聊天机器人需要从海量的编程会话中提取正确的过往对话、决策过程和权衡考量。模型和系统提示词（System Prompt）只有在这些聊天记录被检索并进入上下文后才能发挥作用。

> “许多看起来像是 LLM 的问题，其实根源在于上下文构建阶段。”

如果检索系统将实现代码片段的优先级排在团队讨论权衡过程的内容之前，Agent 依然能给出一个自信的答案。它可能会找到导入 Authlib 的代码和几行行内注释，然后基于实现细节来描述决策过程，而非依据实际的讨论内容。

我们在[私有社区](https://www.skool.com/ai-eng/about)的一份 AnkiHub 操作员审查中也发现了同样的模式。基于讲义进行学习辅导的请求，只有在 Agent 的工具调用能够检索到正确闪存卡时才有效。难点不在于找到任何相关卡片，关于心脏功能的讲座可能匹配到数百张卡片。排序（Ranking）决定了核心卡片能否进入上下文，否则系统就不得不提高 `top_k` 参数，导致提示词被无用信息淹没。

具体的设置因产品而异。上下文构建阶段可能会使用本地搜索、语义搜索、[网络或 API 调用](https://thenewstack.io/serpapi-google-search-api/)，或是数据库查询。它可能由 Agent、固定的工作流或应用程序代码来处理。但流程始终一致：收集正确的上下文，然后据此生成内容。

例如，一个[编程 Agent](https://thenewstack.io/github-mcp-security-scanning/) 在编写补丁之前会运行 `rg`、打开文件、读取日志并检查测试。研究类 Agent 在撰写答案前会搜索网络和内部笔记。学习助手在建议下一步学习内容前会搜索卡组事实和用户上下文。

当上下文构建失败时，其症状看起来很像是生成失败。

## 检索失败会伪装成生成错误

| **症状** | **检索原因** |
| --- | --- |
| 幻觉 | 答案的来源未能进入上下文。 |
| 上下文退化 | 低召回率迫使系统提高 `top_k`，导致嘈杂的结果填满了上下文窗口。 |
| 延迟 | 检索效率低下导致更多的工具调用、更大的候选集和更大的上下文窗口。 |

更好的模型有助于推理和写作，但[如果没有正确的上下文](https://thenewstack.io/better-context-will-always-beat-a-better-model/)，它也无法给出更好的答案。

> “更好的模型有助于推理和写作，但如果没有正确的上下文，它也无法给出更好的答案。”

[Mixedbread OfficeQA-Pro Eval](https://www.mixedbread.com/evals#officeqa-pro) 在基准测试规模上也展示了相同的模式。OfficeQA-Pro 使用了 89,000 页财务文档、密集表格、扫描版 PDF 以及需要跨文档推理的问题。为 Codex 提供更好的搜索工具减少了工具调用次数，并提升了答案质量。

![显示三种 AI 配置下准确率 (%) 与工具调用次数的散点图。](https://cdn.thenewstack.io/media/2026/07/2f86bc04-image1-1024x586.png)

纯文本工具（如 `grep` 和 `rg`）在扁平的代码文件中（勉强）有效。但当上下文存在于 PDF、表格、聊天记录、多模态输入、搜索结果和权限受限的数据中时，它们效果并不好。在这种情况下，Agent 需要一种能够结合精确术语、语义、元数据、权限和排名质量的检索方式。

## 检索需要追踪和评估（Traces and Evals）

一旦检索进入架构，下一个问题就是它是否找到了正确的信息。

为此，你需要追踪（Traces）和评估（Evals）。对于每个检索步骤，最基础的追踪包括输入、输出，以及一种标记每个输出是否相关的方法。

![流程图，展示了包含四个步骤的数据工作流：输入、工具调用、输出和标签。](https://cdn.thenewstack.io/media/2026/07/cc4b9d38-image3.png)

对于使用 `rg` 的编程 Agent，输入是命令，输出是返回的片段，标签则说明哪些片段有帮助、哪些是噪音，以及遗漏了哪些相关文件。

对于产品检索，该步骤可能是 BM25、语义搜索、带重排的混合搜索、生成的 SQL 查询或其他。捕获查询或参数、返回的文档或块，以及这些结果是否有用。

单独追踪每个检索步骤，然后评估整个上下文构建过程。局部追踪回答：“这个查询是否返回了有用的材料？”完整追踪回答：“系统在生成前是否收集了模型所需的一切？”如果是，则失败是生成问题。如果不是，则是检索问题。

没有追踪，你无法知道失败始于何处或该修复什么。

## 不同的失败需要不同的修复

“改进检索”这个说法太宽泛而无用，因为不同的问题需要不同的解决方案。如果缺少相关文档，追踪应该显示它在何处丢失：查询构建、检索、过滤、排序或最终上下文组合。

![顺序流程图，映射了五阶段流水线——查询构建器、检索器、过滤器、排序和上下文——每一阶段都指向各自的失败模式。](https://cdn.thenewstack.io/media/2026/07/2c677d9c-image2.png)

失败的步骤加上追踪显示的结果，会告诉你需要做出什么改变。

| **失败步骤** | **追踪显示内容** | **修复建议** |
| --- | --- | --- |
| `rg` / `grep` | 概念查询返回了字面匹配，但遗漏了相关文件。 | 对文件或块添加语义搜索，或在调用 `rg` 前生成更好的关键词查询。 |
| BM25 | 查询使用了正确的概念，但与源材料的词汇不同。 | 添加语义搜索、同义词或查询扩展。 |
| 语义搜索 | 结果中缺少确切名称、错误字符串、文档 ID 或领域术语。 | 添加关键词或 BM25 路径，或增强确切术语匹配的权重。 |
| 混合检索 | 相关段落排名第 7，但上下文只取前 5。 | 添加或调整重排器（Reranker），或在重排前提高候选集 `top_k`。 |

正确的修复取决于系统试图检索什么。关于决策历史的问题需要决策过程、替代方案以及团队讨论这些方案的聊天记录。学习问题则取决于讲义材料、卡组元数据、语义匹配和用户的学习上下文。

## 架构

一旦你追踪了单独的检索调用，整个架构就呈现出一个简单的形状：向外扩展到上下文构建工具，然后向内收敛以生成最终输出。

![展示 RAG 流水线的系统架构图。](https://cdn.thenewstack.io/media/2026/07/58b3af71-image4-1024x734.png)

检索层可以是一个搜索引擎、向量数据库、SQL 查询、本地文件工具、网络搜索 API 或自定义服务。模式保持不变：构建候选上下文、缩小范围、排序、组装，然后据此生成。

## 为 Agent 提供人类视角的搜索控制

语义搜索比较的是嵌入（Embedding，即意义的数值表示）。它在措辞不同时很有用，但大多数检索意图也依赖于结构化约束。会议搜索框可以对转录稿和笔记进行语义搜索，但一个好用的接口还应该允许用户按人员、日期、项目和来源进行过滤。

金融搜索除了最接近的语义匹配外，可能还需要最新的申报文件、特定的季度或官方来源。在电子商务中，“32×30 工装裤”的最佳语义匹配可能是一款缺货产品。系统必须决定是隐藏它、带缺货注释返回，还是将其显示出来以便用户稍后查看。那个产品决策就是一个检索决策，因为它改变了进入 Agent 的候选集。

在聊天界面中，这些控制项存在于工具架构、查询规划器或应用逻辑中。如果 Agent 运行搜索，它需要与人类通过过滤器、滑块、标签和排序菜单设置的参数相同的约束条件。

检索系统通常需要多种控制手段协同工作：

| **控制项** | **功能** | **示例** |
| --- | --- | --- |
| 精确匹配 | 匹配名称、ID、错误字符串、引用短语、代码或产品编码。 | 查找 `EADDRINUSE`、`Authlib` 或特定 SEC 编号。 |
| 语义匹配 | 在措辞不同时查找相关内容。 | 查找团队讨论认证权衡的会议记录。 |
| 硬过滤 | 在排序前移除无效结果。 | 按租户、权限、人员、日期范围、大小或库存状态限制。 |
| 排序 | 按结构化字段对候选集排序。 | 优先显示最新、最新申报、最低价格、最高评分或近期内容。 |
| 排名 | 根据对本次请求的潜在有用性对候选集评分。 | 结合语义匹配、精确匹配、时效性、来源质量和用途。 |
| 重排 | 在较小的候选集上使用较慢的模型或评分器。 | 在返回前将前 100 个候选结果与查询进行对比。 |

这里，“块”（Chunk）意味着源内容的一小部分，“候选集”（Candidate）是第一次搜索步骤返回的块。排名是为这些候选集排序的评分步骤。上下文组装随后选择哪些块和结构化字段包含在模型提示词中。

更好的排名能提高精度，这意味着返回的块中有更大比例是有用的。如果相关的块排在前面，系统可以将更少的块传递给模型，使用更少的 Token，降低延迟，并减少模型接触到的噪音。如果正确的块排在第 40 位，而上下文只包含前 10 位，系统的行为就像检索遗漏了它一样。

人类和 Agent 使用相同的基本搜索路径：请求结果、检查返回的内容、决定使用什么。人类可以浏览十个搜索结果，比较标题、摘要、日期、域名和 URL，判断结果集是否正确。他们可以打开第三个结果，忽略其余部分，并用更好的查询再次搜索。Agent 通常接收一组有界限的返回文档，并根据上下文进行推理。如果正确的来源低于截止线，Agent 可能会基于不完整的上下文回答。为了避免这种情况，系统必须检索更多的候选集、进行更多的搜索，或向模型传递更多的证据。

一个丢失的文档可能会改变 Agent 下一步搜索的内容。假设有人询问团队为何选择 Authlib。如果第一次搜索错过了团队比较 Authlib 与其他方案的转录内容，Agent 可能会转而搜索代码库。它找到了导入、回调处理器、测试，甚至是一条注释。然后，它会询问关于 OAuth 配置的后续问题。上下文看起来很完整，但它支持的是错误的答案。它解释了 Authlib 是如何被使用的以及为什么在代码库中很有意义，而不是团队为何选择它。

> “上下文看起来很完整，但它支持的是错误的答案。”

但排名无法修复所有搜索问题。如果 Agent 未能请求最新的申报文件，重排器可能会忠实地选择一份词汇匹配度更高但已过期的旧文档。如果工具没有 `date_range`、`person`、`source_type`、大小或 `in_stock` 参数，模型必须在搜索文本中强制加上硬约束，并希望检索系统能推断出它们。硬过滤器让模型需要推断的内容更少，从而使语义搜索更可靠。

## 规模改变了检索问题

搜索系统已经具备了解决这些问题的工具：索引、过滤器、分面（Facets）、排序、缓存、有界重排和时效性作业。Agent 系统需要同样的规范。

人类可能会搜索、调整日期过滤器、扫描第一页，然后再次搜索。一个 Agent 请求可以在几秒钟内多次执行此操作：重写查询、运行关键词和语义搜索、检查稀疏结果、发出后续搜索、获取引文来源，并在回答前请求更多上下文。随着并发用户或 Agent 的增加，检索层可能成为瓶颈。

人类通常愿意为了好结果而等待缓慢的搜索。而 Agent 系统通常将缓慢且不确定的搜索转化成更多的工作量。当排名较弱时，团队会通过提高 `top_k`、并行运行关键词和语义搜索、添加重排、获取更多源文档以及向模型传递更大的证据包来进行补偿。这可以改善答案，但成本转移到了 Token、延迟和检索负载上。更好的排名允许系统返回更少、更好的候选集，而不是让每个请求都携带一大堆可能的证据。

在小型语料库中，即使是完全的 Agent 化方法，也可以快速且廉价地进行全面搜索。这就是我在你刚开始且数据不多时所建议的做法。在你需要之前不要增加复杂性。但当有数百万或数十亿个块时，每一次额外的检索调用、候选集、排名通过和返回的 Token 都会迅速累积。

## 多阶段检索是生产环境的形态

大多数生产系统应该将检索分成多个阶段，即使 UI 是一个聊天框。

| **阶段** | **内容** | **追踪问题** |
| --- | --- | --- |
| 搜索参数构建 | 应用或 Agent 将请求和状态转化为查询、过滤器和排序。 | 它是否使用正确的约束请求了正确的内容？ |
| 候选生成 | 系统从文本、向量或结构化数据中找到可能的块。 | 正确的来源是否进入了候选集？ |
| 过滤 | 权限和产品约束缩小了可返回的范围。 | 来源是否被正确排除，还是错误地丢失了？ |
| 排序 | 当顺序很重要时，结构化字段对结果进行排序。 | 最新、最便宜、最高评分或当前项目是否被呈现？ |
| 排名 | 系统根据对本次请求的有用性对候选集进行评分。 | 来源是否在列表中但排名太低？ |
| 返回摘要 | 系统仅返回 Agent 所需的字段。 | 应用是否收到了可用的证据和来源？ |
| 上下文组装 | 应用选择、格式化并为模型分配证据预算。 | 有用的证据是否在生成前被丢弃了？ |

每个阶段都会留下不同的修复路径。如果 Agent 选择错误的过滤器，更改嵌入模型是没有用的。如果最新的文档可用但工具从未按日期排序，修复方案应在于搜索参数或检索 API。如果正确的来源存在但低于截止线，修复方案应在于排名。如果正确的来源返回了但在生成前被丢弃，Bug 就出在上下文组装中。

随着检索成为 Agent 架构的核心部分，团队越来越需要能够在一个系统中结合语义搜索、精确匹配、过滤、排名和大规模检索的基础设施。根据需求，这可能涉及 Vespa、Elastic 或 Coveo 等搜索和检索平台，它们各自支持不同的排名、检索和操作规模方法。

重点不在于具体的技术选择，而在于认识到检索质量已成为首要的工程问题。随着 Agent 工作负载的增长，检索系统正日益决定整个应用的准确性、成本、延迟和可靠性。
