Netlify CTO指出，AI时代写代码不再是工程师的核心任务。工程师的价值将转向定义智能体体验（AX），管理复杂系统架构，并为AI协作制定安全防线。

> 译自：[Netlify CTO Dana Lawson: Writing code is no longer the job](https://thenewstack.io/netlify-agent-experience-engineers/)
> 
> 作者：Jennifer Riggins

“自上世纪90年代以来，我一直在做这件事——几十年来都在构建保护屏障，防止人类破坏生产环境。而现在，我们正在拆除这些屏障并说：嘿，来和我们一起玩吧，来创造吧，去构建世界的下一个体验，”Netlify CTO [Dana Lawson](https://www.linkedin.com/in/dglawson/)本周在伦敦举行的 AI Native DevCon 上对座无虚席的观众说道。

智能体AI（Agentic AI）已经建立了下一个抽象层，其中通过对话式语言表达的——意图——成为了下一个编程语言，让呈指数级增长的人群能够以此进行创作。到2029年，将有[10亿个新应用程序被编写](https://www.idc.com/resource-center/blog/agent-adoption-the-it-industrys-next-great-inflection-point/)，因为AI赋能了 Lawson 所称的*构建者（builder）*。

当然，这对于这些全民开发者（citizen developers）来说是件好事。但是，对于许多软件工程师来说，这是他们第一次面临如此程度的职业不确定性。整个行业——乃至整个世界——都在如此快速地变化，科技行业的裁员消息充斥着媒体头条。即使你的公司没有采取那种激进的做法，但在你很少写代码的当下，去质疑自己的角色定位也是很正常的。

那么，如果软件开发不再由编写代码来定义，是什么让工程师脱颖而出？在 Lawson 看来，他们是那些理解智能体体验（agent experience, AX）能够且应该是什么样的人。

![](https://cdn.thenewstack.io/media/2026/06/162b569d-dana-lawson-ainativedevcon-1-1024x576.jpg)

本周在伦敦 AI Native DevCon 上的 Dana Lawson（图片来源：*The New Stack*）

## 工程师的角色改变了吗？

在某些方面，并没有。因为编写代码在工程师的工作中向来只占不到四分之一，而且越来越成为最缺乏战略意义的部分。

“你是生产环境的牧羊人，”Lawson 在谈到 AX 时代的工程师体验时说道。“确保输入和输出都得到清晰的理解。如果我们正确地打造了智能体体验，这些智能体应该是由事件驱动的活动，其中信号是被推送到开发者面前，而不是由开发者去拉取。”

如今，智能体应用成功与否，比以往任何时候都更取决于对复杂系统、生产路径和业务上下文的全面理解。

“智能体体验是在软件交付生命周期中，思考所有这些新的系统上下文和意图，”Lawson 主张。“它是设计人类与智能体无缝协作的实践——而不仅仅是让 API 调用对智能体更友好。”

Netlify 成立于 2014 年，最初是一个面向前端网站和应用程序开发者的平台。而这波汹涌而来的全民开发者并不懂开发者的语言；他们不知道什么是 git。为此，该平台必须进行重构，不仅要与开发者沟通，还要与 AI 智能体以及这些新的构建者沟通。

在为这两个新受众重新改造平台的过程中，Netlify 发现它反而能够更好地服务其开发者群体。因为正如 Lawson 所定义的那样，智能体体验是开发者体验和用户体验的结合。通过同时解决这两者，她的团队终于能够帮助用户克服机构和领域知识的界限，这些界限过去一直充当着行业的看门人，将行业局限于拥有计算机科学学位的人。

“当我们让智能体错误信息更清晰、为机器结构化构建输出并消除不必要的摩擦时，我们的开发者也从中受益，”Lawson 反思道。“我们消除的每一个人类假设，都让这个平台对每个人来说都变得更好。”

## 对于这两类构建者来说，核心都在于构建什么。

正如[结果工程 (Outcome Engineering)](https://o16g.com/)所论证的那样，AI 消除了作为限制因素的人类带宽。这意味着工程师的工作将更多地在于决定*不*构建什么。

“残酷的事实是，你将会被困在构建一大堆可能在几个月（而不是几年）内就会过时的事物中。因此，找到正确的路径并在正确的领域投入意图，才是保持竞争力的关键，”Lawson 在接受 *The New Stack* 的后续采访中谈到这种“欲速则不达”的风险时说道。“这正是开发和工程实践成为核心的地方，因为现在世界上的任何人几乎都可以构建任何东西。”

工程师的职责就是确保构建者所构建的东西是正确的——对你的业务、你的客户、安全性以及这个世界都是正确的。

“我们肩负着责任，要确保我们正在实施紧凑化，我们在思考压缩，我们在构建正确的资源，以便让互联网继续造福所有人、保持开放和民主化，同时也要对环境友好，”她继续说道。

这意味着工程设计比以往任何时候都更必须专注于内部技术栈、人员和流程。

## 为你的智能体和智能体意图而重构系统

“在此过程中，你确实必须重新思考整个技术栈，从我们如何表达意图到系统如何通信，这样我们才能建立信任，”Lawson 说道。毕竟，企业系统原本是为人类操作员设计的。

“原本的意图是你会成为该循环的一部分，即使某些内容被设置为持续运行，智能体也难以应对这种情况。它们无法跨越界限。每个 API 都在说不同的方言，”正如 Lawson 所说。关键的工作流程只能靠口口相传，“在 2022 年的一个 Slack 线程中，或者在未记录文档的 Terraform 模块中。”

Lawson 提出了推动更好智能体体验的架构演进方案：

1. **从 API 到能力（capabilities）**：虽然软件架构一直以来都通过 API（例如 POST、GET、PATCH）暴露端点，但智能体原生系统通过表达你想完成什么（例如 `create_site`、`deploy-repository`）的能力，来暴露意图级别的操作。
2. **从请求-响应到事件驱动**：告别“请求并等待”模式，智能体订阅事件、观察系统行为，并在获得许可时自主行动。
3. **从机器可读到智能体易读**：告别对某些“神秘”架构片段的个人化理解，智能体化平台为智能体和人类创建了统一的架构复杂度蓝图，以便在他们进行更改之前理解系统。

“这种开发周期变成了一个持续的人机（人类-智能体）协作环，因此不再是向运维团队进行‘交接’，”Lawson 说道。“而是每个人同时参与，人类通过提供判断力、品味和方向保留在协作环中。这个协作环以机器般的速度运行。这并不是在取代工程师，而是在放大每个人成为构建者的能力。”

## 人类设定的智能体边界

“智能体不仅是在编写代码；它们还在参与企业基础设施生命周期；它们在生成测试、检测故障、分析构建块、提出修复建议、提交 PR、进行 CI/CD——这一切都变成了一个持续的反馈循环，”Lawson 谈道。“有些事情可能还没有达到那种精细化水平，但这正是你需要开始思考如何重新定义这些需求的地方。”

至少在目前，智能体体验应该以推动它的人类为基础。对于 Netlify 团队来说，这表现为：

* 每个智能体仅在其指定的沙箱内执行。
* 默认引入“人机协同”（Human-in-the-loop），专注于工程师工作中富有意义且意图明确的部分。
* 记录智能体的每一步操作，使其可审计并可随时回滚。

> “如果你无法解释智能体做了什么，你又凭什么在生产环境中信任它？”

“如果你不能解释智能体做了什么，你为什么要信任它在生产环境中运行？”Lawson 说道。

“工程设计正在从实现每一个功能，演变为真正确保系统和保护屏障架构紧密且牢固。”

好消息是，至少在目前，为智能体进行设计实际上一直就是在为人类进行设计。“这不仅仅是智能体体验，更是人类体验，而这正是 AX 的意义所在，”Lawson 这样总结道。

> “构建能够让智能体协同工作的系统的判断力，才是未来的工程技能……”

“AI 迫使我们厘清我们的架构、我们的结构和我们的信号。实际上，它让我们都成为了更好的开发者。构建出让智能体能够协同工作的系统的判断力，才是未来的工程技能，而智能体体验可确保智能体放大人类的创造力，而不是取代它。”
