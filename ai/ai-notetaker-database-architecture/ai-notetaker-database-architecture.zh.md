想象一下，你刚刚结束了一场两小时的会议。你佩戴着一款小型AI工作伴侣，它承诺会捕捉谈话内容、进行整理，并允许你事后进行查询。会议结束后两小时，你打开应用请求查看转录文本。你等待着，加载图标不停转动。你购买这款设备的初衷就是为了再也不用费心记录会议，而现在，它唯一承诺能做好的事——回溯谈话内容——却成了让你漫长等待的罪魁祸首。

这是AI硬件产品中令我着迷的失效模式，因为它并非模型问题。转录非常完美，总结也很出色。真正出问题的地方是没人营销的环节：在用户请求的那一刻，如何将正确的数据从磁盘读取并展示到屏幕上。这是一个数据问题。对于一个核心承诺是“记忆”的产品来说，数据问题就是产品问题。

> “对于一个核心承诺是“记忆”的产品来说，数据问题就是产品问题。”

我想剖析一个真实的案例，因为遭遇此问题的团队 Plaud 打造了市场上最受欢迎的AI笔记应用，而他们崩溃的架构正是该领域几乎每个团队起步时采用的架构。在产品发布初期，这显得很合理；但随着规模扩大，它就成了沉重的负担。在发布产品之前而非之后，理解其背后的原因至关重要。

## 首先，什么是做对了的

如果把这个故事简单归结为“他们犯了错”很容易，但事实并非如此。最初的架构是一系列合理的决策，在剖析哪里出了问题之前，我必须明确这一点，因为经验教训存在于“合理”与“大规模下正确”之间的差距里，而非谁的能力不足。

Plaud 的产品每次录音会生成两种截然不同的数据。一种是结构化的元数据：谁录制的、时间、时长、标签、处理流程的状态等。另一种是[非结构化载荷](https://thenewstack.io/from-unstructured-data-to-rag-ready-with-docling/)：音频文件及其转录文本，单次会话可达数十兆字节。团队采用了教科书式的做法：将结构化元数据放入 MySQL，因为关系型查询和事务的成本很低；将大对象放入 S3，因为那里的存储成本极低且几乎无限。

如果你构建过系统，一定做过完全相同的决策。用对象存储存二进制大对象（blob），用关系型数据库存所有需要查询的数据。这存在于每一个架构图中，是默认配置。在产品生命周期的很长一段时间里，这运行良好。

问题不在于这个决定是错的，而在于它包含一个隐含假设：元数据和内容可以生活在独立的系统中，因为它们永远不需要实时保持一致。对于大多数应用，这个假设成立。但对于核心交互是“立即给我返回我刚才录制的内容”的产品，它就不成立了。而这两个世界之间的鸿沟，正是所有问题开始恶化的地方。

## 转录文本的“边车”（Sidecar）模式

我在关于 [RAG 检索](https://thenewstack.io/rag-pipeline-hybrid-search/) 的文章中，指出了一个我称之为“向量边车”的反模式：习惯于在主存储旁部署一个独立的向量数据库，结果发现两个系统无法共同回答一个查询。Plaud 的架构形式完全一样，只是应用在了AI硬件上。可以称之为“转录边车”。

无论何时，当你将结构化元数据与非结构化内容拆分到两个没有共享事务保证的系统中时，就会得到这种“转录边车”。元数据声称录音存在、完整且已就绪。而内容存储在别处，需要通过单独的调用访问，有其自身的延迟和失效模式。没有任何东西将两者绑定。不存在横跨“显示转录就绪的那一行记录”和“包含转录内容的对象”之间的事务边界。

这产生了三个明显的复合问题：

* **检索延迟不是网络问题，而是数据局部性问题。** S3 是出色的对象存储，但它不是数据库。当你将对象存储作为并发负载下数十兆字节载荷的主要检索路径时，你其实是在用数据库的事务一致性和低延迟保证，去交换 blob 存储的简单性。在轻负载下，这种交换是无感的。但在高并发负载下，S3 的检索延迟及其波动就成了用户感受到的产品体验。
* **两个独立故障的系统之间会产生一致性鸿沟。** MySQL 中的行记录可能在 S3 对象可达之前就显示录音已就绪，或者副本延迟会导致用户看到的元数据与获取的内容不一致。由于没有共享事务，应用层必须承担起修补这些缺口的责任，重试、轮询和对账逻辑每季度都在变得越来越复杂。
* **Schema（模式）在最糟糕的时间点变得难以变动。** 下文会详细说明，简短来说是元数据存储触及了规模上限，导致变更 Schema 需要停机维护。对于一个仍在进化功能特性的产品而言，这意味着数据库在统治路线图，而非反之。

统一的洞察是我在整个系列文章中不断重复的观点：任何时候你将数据拆分到可以独立失效的系统中，你就把一致性问题扔给了应用层。在那里，问题并非得到解决，而是被“管理”，并随着时间推移不断加剧。

## 当数据库统治产品路线图

[Plaud 案例](https://www.pingcap.com/case-study/how-plaud-eliminated-s3-latency-limitless-scale/) 中最有趣的失效点并非延迟，而是 Schema 的冻结，因为这是团队最不预期且感受最强烈的。

在达到约 3 亿行数据时，团队的 MySQL 设置达到了瓶颈，像增加列或更改索引这类常规演进操作，在不冒风险的情况下已无法在线执行。在如此庞大的表上进行 DDL 操作，意味着锁表、复制压力以及停机的现实可能性。因此，变更必须集中在维护窗口期内完成。

仔细想想这意味着什么。Schema 变更的维护窗口就是数据库留给产品团队发布功能的时间。产品路线图现在必须围绕数据库的局限性进行调整。一个需要新列的功能只能等待下一个窗口。产品的迭代节奏由数据层的运营脆弱性设定。大多数团队没有注意到这种倒置，因为它发生得很缓慢，直到某天产品经理询问为什么一个小改动变成了两周的项目，答案却是“数据库原因”。

> “Schema 变更的维护窗口就是数据库留给产品团队发布功能的时间。产品路线图现在必须围绕数据库的局限性进行调整。”

3 亿行的上限是一个强制函数，团队触及它的时间比预期的要晚，而且*应该*在更晚的时候触及。说“预期”，是因为单机 MySQL 在遇到瓶颈前感觉无限大。说“应该”，是因为能支撑到 3 亿行的架构往往无法支撑过 3 亿行，而在 3 亿行时进行迁移，比在 3000 万行时要麻烦得多。这个上限是真实的、可预测的，而大多数团队只有在触及它之后才会去规划。

## 架构重构真正修复了什么

Plaud 的解决方案是将元数据层整合到一个分布式 SQL 数据库中（他们迁移到了 [TiDB](https://www.tidb.io)），这消除了单机瓶颈并恢复了在线 Schema 变更。我使用他们的数字并非为了推销产品，而是作为证据，说明在这种产品中架构债的去向。

| | |
| --- | --- |
| **之前（大规模下的双存储大脑）** | **之后（整合的元数据层）** |
| 并发负载下吞吐量受限 | QPS 容量提升约 10 倍 |
| 负载下尾部延迟不稳定 | P95 延迟低于 10 毫秒 |
| Schema 变更需维护窗口 | 在线 DDL，无需停机 |
| 单机上限在数亿行规模 | 多集群水平扩展 |

*数据来自 Plaud 的迁移报告。*

QPS 容量和尾部延迟固然重要，但我指出的第一个结果是“在线 DDL”。恢复了无需维护窗口即可变更 Schema 的能力，是将路线图交还给产品团队的关键。数据库不再左右发布节奏。这不是基准测试图表上能体现的性能提升，但却是产品组织每周都能切实感受到的改善。

还要注意，修复并没有做的事：它并没有试图将 30MB 的音频文件存入数据库。大对象仍然可以[存储在对象存储中](https://thenewstack.io/slatedb-bottomless-databases-built-on-cloud-object-stores/)。重点从来不是“为了整合而整合”，而是元数据层——即那些必须实时保持一致、可查询且可演进的部分——需要在规模化下真正表现得像个数据库，而不是成为“双大脑”架构中脆弱的一半。

## 架构债的去向

这就是 Plaud 的数据所证明的结论：当数据库即产品时，架构债不会堆积在只有平台团队感知的后台系统中，而是堆积在产品体验中，让每一位用户都有所感知。

这是 AI 笔记应用与内部数据分析工具之间的结构性差异。对于这类产品，每一次有意义的用户交互都是一次数据库操作。录音创建行和对象。转录更新状态。检索是读取。编辑是写入。搜索是查询。产品中没有哪一部分不是在底层运行数据库操作。架构问题会一一转化为产品问题。一致性鸿沟会表现为转录文本短暂丢失；检索缓慢表现为产品记忆速度变慢；Schema 冻结意味着功能上线推迟。

没有用户会提交一份写着“你们的元数据存储和对象存储缺乏共享事务边界”的错误报告。如果置之不理，这类债务会表现为产品在用户最依赖它的时候显得不可靠。对于一个核心价值主张是“信任我为你记忆”的产品来说，债务最终会压垮承诺本身。这就是 Plaud 选择在那个时间点进行重构的意义所在。团队在失效被用户察觉之前修复了数据层。

> “你可以拥有类别中最顶级的转录模型，但交付的产品依然感觉不可靠，因为对于此类产品，可靠性是数据架构的属性，而非模型属性。”

这就是我说“数据库即产品”的原因。这不是口号，而是对产品质量决定因素的直观描述。你可以拥有同类中最顶级的转录模型，但交付的产品依然感觉不可靠，因为对于此类产品，可靠性是数据架构的属性，而非模型属性。

## 整个行业都在等待陷阱

AI 硬件正处于风口。笔记工具、可穿戴设备、环境录音笔、挂件、胸牌——每一个都建立在“设备会记住，你就不必费心”的前提上。这个类别在增长，产品在大家关注的领域（模型、外形、续航）上做得越来越好。

但在底层，几乎所有产品都从 Plaud 最初采用的那种架构起步，因为它看起来是合理的默认配置：结构化元数据在单机关系数据库中，大内容在对象存储中，没有共享事务边界。在 1000 万行数据时容易变更的 Schema，在 3 亿行时就冻结了。陷阱是相同的，而且正等待在每个团队增长曲线上的同一个位置。

能够胜出的团队是那些尽早认识到这一点、理解对于“记忆产品”而言元数据层不是后台管道而是用户体验脊梁的团队，他们会为想要达到的规模而非当前规模去[规划数据架构](https://thenewstack.io/3-ways-enterprises-can-scale-ai-gains-in-2026/)。Plaud 的迁移是一个值得在达到 3 亿行规模之前而非之后学习的范式。从别人的 3 亿行数据中吸取教训，比从自己的 3 亿行数据中吸取要便宜得多。

当你的数据库和你的产品是同一个东西时，你不能把数据库当作别人的问题。它就是产品本身。请像打造产品一样打造它。
