在6月的第一周，三家供应商将编程智能体推向了超越单开发者循环的新阶段。

1. Cognition 于 6 月 2 日发布了 [Devin Desktop](https://cognition.ai/blog/introducing-devin-desktop)。
2. 同一天，Microsoft 利用 Build 2026 大会推出了 [Rayfin](https://thenewstack.io/microsoft-build-2026-rayfin-replit-vibe-coding/)（关联[氛围编程](https://thenewstack.io/microsoft-build-2026-rayfin-replit-vibe-coding/)）。
3. Augment Code 于 6 月 3 日为所有团队方案宣布了 [Cosmos](https://www.augmentcode.com/blog/cosmos-the-platform-for-ai-native-engineering-teams)。

这三次发布处于同一个新兴技术栈的不同层级：

1. Devin Desktop 为团队提供了一个管理它们的单一控制台。
2. Rayfin 治理哪些由智能体构建的应用可以部署到企业中。
3. Cosmos 协调一组智能体机队。

总的来看，这三者标志着一种转变：编程智能体正在成为团队基础设施，而不仅仅是个人工具。对于任何观察过源码控制成长历程的人来说，这种模式都非常熟悉。

版本控制起初是单台工作站上的个人便利工具，随后 Git 和持续集成将其转变为共享基础设施，拥有分支、评审和全员遵守的策略。编程智能体如今也正在经历同样的旅程。本文中的参考点是每个工程团队都已经熟悉的：拉取请求（PR）、CI 流水线、访问策略以及将它们连接在一起的控制平面。

## 从单开发者挂架到团队挂架

几周前我曾主张，[智能体化编程工具已经趋同](https://thenewstack.io/claude-code-vs-cursor-vs-codex-vs-antigravity-2026/)，竞争已从模型转移到了“挂架”（harness）上——即包裹在模型周围的工作流和审批层。在诸如 [SWE-bench Verified](https://www.swebench.com/verified.html) 等公开编程基准测试中，顶级系统的表现已经非常接近，因此产品差异化越来越多地来自挂架、规划步骤、工具使用、评审关卡以及通往生产环境的路径。

> 6月的发布打破了“一个开发者、一个挂架”的假设。

这一论点曾假设一名开发者坐在一套挂架前。而 6 月的发布通过将挂架交给整个团队打破了这一假设。团队挂架必须完成个人挂架从未需要做的事情。它必须记住跨人员和会话的决策，这样同一个命名规范就不会在每个周一都被重新议定。它必须协调多个并行工作的智能体而不产生冲突。并且，它必须在需要判断力时给人类一个立足之地，扮演类似于拉取请求中评审者的角色。

你可以将其类比为：开发者在笔记本电脑上运行脚本，与团队通过带有缓存状态和审计日志的共享 CI 系统运行相同任务之间的区别——即使工作内容看起来一样，问责制也发生了变化。

## Augment Cosmos：全生命周期的控制平面

来自 [Augment Code](https://www.augmentcode.com/) 的 Cosmos 试图凌驾于团队已运行的智能体之上，并在整个软件生命周期中协调它们。Augment 描述的智能体可以跨越分拣、规格说明、实现、评审、测试、部署和反馈等环节，彼此协作并在需要判断力时引入人类。专用智能体共享记忆，因此一个智能体学到的东西不会在其会话结束时随之消亡。

最贴切的类比是 CI/CD 控制平面。流水线本身不编写代码；它决定运行什么、按什么顺序运行，以及在变更发布前必须通过什么测试。Cosmos 为智能体机队扮演了这一角色，持有共享上下文和智能体运行的规则。

考虑到凌晨 2 点传呼值班工程师的事件。在 Augment 自己的事件管理案例中，Cosmos 智能体在工程师加入之前就已经获取了警报、收集了上下文并开始调查，因此人类评审的是一个领先的起点，而不是一张白纸。这里的收益在于共享记忆和协调，这允许下一个智能体从上一个智能体结束的地方开始。

所谓的“冷启动问题”——即智能体没有上下文可用——正是 Cosmos 旨在解决的问题。智能体在会话之间几乎是无状态的，这就是为什么一个在第一周写出尚可代码的工具，在第十二周还会重复同样的错误。共享记忆层将这些分散的修正转化为整个团队可以复用的资产。

## Cognition Devin Desktop：管理者的控制台

Devin Desktop 从开发者的视角切入同样的转变。[Cognition](https://cognition.ai/) 将其称为下一代 Windsurf，它将“智能体指挥中心”设为 IDE 的默认界面，使工程师能在一个地方管理本地和云端智能体、拉取请求及上下文。一个名为 [Spaces](https://docs.devin.ai/desktop/spaces) 的功能允许相关的智能体共享上下文并协作完成任务。

Cognition 称 Devin Desktop 是“智能体中立”的。通过支持智能体客户端协议（ACP，一种智能体互操作性的开放标准），任何兼容 ACP 的智能体都可以在 Devin Desktop 中与 Devin 并行运行。因此，团队原则上可以在同一个面板上，针对一个任务驱动 Codex CLI，针对另一个任务驱动基于 Claude 的智能体。这种中立性是通过 ACP 适配器实现的，而非对每个商业 Codex 或 Claude 界面的原生钩子。这依然很重要，因为大多数团队已经在混合运行工具，且没人想要一个只能管理单一供应商智能体的控制台。

想象一下，一位技术主管在周五将积压工作切分为八张票，分别交给不同的智能体，然后在周一花时间评审八个拉取请求，而不是亲手编写代码。Devin Desktop 正是为这种工作日而设计的，同时保留了完整的编辑器，以便人类进行“最后一公里”的修改。如果说 Cosmos 倾向于一个有主见的平台，那么 Devin Desktop 看起来更像一个尊重你既有智能体选择的仪表盘。

## Microsoft Rayfin：智能体发布产物的治理

Rayfin 则沿着不同的维度移动。它不直接管理智能体，而是收窄了智能体越来越多产出的产物——应用后端的路径，并将其引导至一个受管的家园。在 Build 2026 上宣布的 Rayfin 是一个开源 SDK 和 CLI，允许开发者和编程智能体在代码中定义完整的应用后端（数据模型、业务逻辑、身份验证和访问策略），然后部署到 Microsoft Fabric。

一旦应用进入 Fabric，它就会继承组织已经在运行的安全性、合规性和治理，其数据存在于 OneLake 中，而不是新的数据孤岛。Replit 是首发合作伙伴，因此 Replit 中的智能体可以定义后端并将其推送到受管租户中。Microsoft 提出的问题是真实的：智能体化编程工具创建应用的速度超过了任何人的治理能力，每一个不受管的应用都是审计员关注的控制措施之外的又一个数据孤岛。

平台工程中的类比是“铺就之路”（paved road），即让合规方式成为简便方式的支持路径。Rayfin 试图成为智能体构建后端的铺就之路，使应用在进入生产环境时就已经处于数据资产管理之内，而不是在安全审查后才生搬硬套。这是对输出结果的组织级控制，而非对工作的协调，这也是为什么它拓宽了故事维度而非重复旧调。Rayfin 目前处于预览版，与 Replit 的配对是第一个集成而非广泛集成，其治理深度将随着预览版的运行而变得更加清晰。

## 各平台的位置

这三个发布解决了不同的问题，因此选择取决于工程组织最迫切感受到的痛点。

> 一个淹没在智能体生成的拉取请求中的团队面临的是协调问题。一个运行着五种不同智能体的团队面临的是管理问题。一个智能体不断部署不受管应用的团队面临的是控制问题。

下表将每种需求映射到为其构建的平台，以及相关的权衡。

| 场景 | 最适合的平台 | 理由与权衡 |
| --- | --- | --- |
| 在全生命周期中通过共享记忆协调多个智能体 | Augment Cosmos | 构建为生命周期控制平面，较重且偏向企业级 |
| 从一个控制台管理混合的第三方智能体 | Cognition Devin Desktop | 通过 ACP 实现智能体中立，仍是以 IDE 为中心的界面 |
| 治理哪些智能体构建的应用部署到数据资产中 | Microsoft Rayfin | 继承 Fabric 治理，绑定 Fabric 且处于预览版 |
| 需要行内编辑和自动补全的独立开发者 | 现有的个人开发者工具 | 对于单人来说，这些团队层级的工具都过于笨重 |

在实践中，这些类别开始相互模糊。一个大型工程组织可能会运行 Cosmos 进行协调，通过 Devin Desktop 管理部分第三方智能体，并依然通过 Rayfin 这样的受管后端引导智能体构建的应用。这些是层级，而非替代品，大多数团队最终会同时采用多种工具。

## 记忆的代价

团队记忆既是生产力特性，也是安全和治理层面。

> 正是这一层让智能体在组织中变得有用，也正是这一层让锁定效应（lock-in）变得根深蒂固，因为公司积累的上下文比读取它的智能体更难迁移。

一旦智能体跨会话携带了规范、事件历史、客户上下文、凭证、架构决策和评审反馈，购买者就必须询问这些记忆存在哪里、谁可以审计它、如何撤销它，以及合同结束时它是否随团队一起迁移。正是这一层让智能体在组织中变得有用，也正是这一层让锁定效应变得根深蒂固，因为公司积累的上下文比读取它的智能体更难迁移。

## 未来展望

对于工程团队来说，参考点依然是熟悉的。Cosmos 充当智能体的 CI 控制平面，Devin Desktop 充当机队的运维控制台，而 Rayfin 则作为保持产出在护栏内的铺就之路。这三者背后的共同转变是：智能体不再仅仅是个人生产力工具，而是成为了组织为其定义策略的对象，就像对待源码控制和部署一样。

对于购买者来说，决定因素是哪个平台托管了团队的智能体、它们共享的记忆以及它们记录的审批。如果像智能体客户端协议（ACP）这样的标准能保持该管理层的可移植性，团队将跨供应商组建自己的智能体机队。如果不能，智能体化软件开发生命周期（SDLC）将硬化为相互竞争的控制平面，每个平面都有自己的记忆、权限和通往生产环境的路径。GitHub 和 Cursor 已经在形成自己的答案，接下来的悬念在于，这个团队层级是会像单人工具那样趋向于单一形态，还是走向分裂。
