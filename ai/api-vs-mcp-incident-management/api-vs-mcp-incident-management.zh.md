新兴技术的诱惑力不可否认，但采用新技术并不意味着要完全抛弃已有的有效方案。相反，新技术必须在现有基础设施中找到自己的位置，以补充团队日常依赖的系统。

[Model Context Protocol (MCP)](https://modelcontextprotocol.io/docs/getting-started/intro) 在过去一年半中引起了极大关注，一些人甚至将[这项技术的出现与传统API的兴起进行了对比](https://thenewstack.io/is-model-context-protocol-the-new-api/)。

MCP和API为工程师提供了两种创建互联生态系统的方式。但对于事件管理团队而言，这两个术语并不能互换；它们各自扮演着截然不同的角色。

[工具蔓延 (Tool sprawl)](https://thenewstack.io/developers-unhappy-with-tool-sprawl-lagging-data-long-waits/) 在事件管理中是一个持续存在的挑战。分散且碎片化的方法阻碍了AI投资发挥其潜力，导致事件响应者的体验不佳。

MCP为克服这种碎片化提供了一条一致的途径，而API则继续提供可重复工作流所需的确定性控制。

让我们深入探讨这两种技术，并权衡它们在事件管理中的优缺点。

## 理解基本原理

API提供结构化的端点，允许一个系统向另一个系统请求数据或触发特定操作。当团队的监控工具查询数据库获取指标，或者CI/CD流水线通知部署服务发布新代码时，这就是API在发挥作用。

MCP代表了一种不同的战略方法。它是一种专门为通过标准化接口将AI助手和代理连接到外部数据源和工具（包括其他AI工具）而设计的协议。

MCP并不会取代API。它创建了一个标准化层，通过该层，[AI代理可以获取来自多个工具和供应商的上下文](https://thenewstack.io/rag-isnt-dead-but-context-engineering-is-the-new-hotness/)。在事件管理中，跨工具访问至关重要。响应者需要在警报、变更、通信、服务所有权和客户影响等方面拥有连贯的视图。

> “MCP并不会取代API。它创建了一个标准化层，通过该层，AI代理可以获取来自多个工具和供应商所需的上下文。”

以下是这两个概念之间差异的总结以及各自的优缺点：

|  |  |  |
| --- | --- | --- |
| **维度** | **API** | **MCP** |
| **主要用例** | 直接的系统对系统集成 | AI对系统（以及具有上下文感知的AI对工具集成） |
| **优点** | • 经受过实战考验，拥有大量可用工具  • 适用于任何编程语言  • 对执行结果有精确控制  • 安全模型广为人知（API密钥、权限、控制、合规性） | • AI自动发现工具  • 连接各组件所需的代码显著减少  • 暴露功能能力的标准化方式  • 基于上下文而非僵化的脚本执行 |
| **缺点** | • 用户需要确切了解端点  • 手动编排变得复杂  • 不具备对意图的理解 | • 新兴生态系统，尚在成熟中  • 需要由MCP客户端/宿主程序调用  • 比直接调用更不可预测 |
| **最佳场景** | • 自动化工作流  • 微服务  • 移动应用  • Webhooks | • AI代理和聊天机器人  • 智能助手  • 上下文驱动的自动化  • 需要多个来源和功能的复杂场景 |

## 用于确定性工作流的API

对于事件管理，API最适用于需要快速且一致（即使在高并发下）的重复性操作。另一种思考方式是将其视为确定性工作流。事件响应团队希望在不留任何AI解释空间的情况下获得快速、确定的行动，特别是在缓解阶段。MCP可能会引入不必要的风险。

基于API的集成还通过显式的身份验证流程、详细的审计日志和精细的权限控制来帮助组织满足[安全要求](https://thenewstack.io/developer-proves-ai-agents-can-be-reprogrammed-via-new-exploit/)——这对于SOC2合规性至关重要。虽然MCP可以使用API已有的相同授权和权限，但由于AI代理会自主选择执行内容，团队需要通过“人在回路”增加一层额外的安全保障。

## 用于非确定性路径的MCP

当人类操作员通过自然语言与系统交互时，特别是在分类、诊断和调查期间，MCP的吸引力就会显现出来。在这些时刻，响应者需要从整个技术栈中收集上下文，但往往会被工具蔓延所拖累。MCP为AI代理提供了一种获取分布式上下文的标准化方法，使其相较于孤立运行的代理具有战略优势。

> “当人类操作员通过自然语言与系统交互时，特别是在分类、诊断和调查期间，MCP的吸引力就会显现出来。”

例如，团队可以问：“为什么欧盟地区的结账错误激增？”AI代理可以从事件管理平台获取当前的事件详情，包括来自监控工具的事件数据、近期的变更事件和事件历史，以及来自Slack或Microsoft Teams等协作工具的信息和利益相关者更新。当这些上下文被收集并集中在同一个地方时，找出可能的促成因素或假设，以及验证的下一步最佳行动方案就变得容易得多。

MCP在动态的、探索性的场景中表现出色，即步骤序列是由用户驱动和批准，而非预先定义的。请求可以包括“将结账问题的状态更新发布到我们的沟通频道，并在事件中添加关于受影响业务服务的简短说明。”代理可以在工程师无需离开其首选工具的情况下完成这两项操作。它会提交一份草稿，在发布前与人类确认，并在获得批准后完成操作。

## 世界将发生改变

在AI原生的未来，MCP很可能代表了技术系统互联的方式。随着组织越来越依赖AI助手进行运营工作，标准化和互操作性变得极其宝贵。

对于事件管理而言，MCP为团队提供了一条路径，通过让代理访问更丰富的上下文，使其在超越孤立行动时更强大，从而减少工具蔓延并改善响应者体验，进而帮助团队从AI投资中实现更大的战略价值。
