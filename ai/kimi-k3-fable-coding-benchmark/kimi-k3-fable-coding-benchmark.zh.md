Moonshot AI 在七月中旬[发布了 Kimi K3](https://thenewstack.io/kimi-k3-open-weight-coding/)，将其定位为一款足以与 Claude 和 GPT 正面竞争且价格更低的专业级编程工具。这是一款拥有 2.8 万亿参数的开放权重模型，也是中国有史以来最大的模型。该公司声称其性能可媲美 Anthropic 的 [Opus 4.8](https://thenewstack.io/claude-opus-48-release/)。

Kimi K3 API 的价格为每百万输入 token 3 美元，每百万输出 token 15 美元。Anthropic 的顶级模型 Fable 5 的价格为每百万输入 token 10 美元，每百万输出 token 50 美元。这意味着其输入和输出成本均高出三倍以上。

一个成本仅为三分之一的模型，只有在输出质量和用户体验达到预期时才算划算。为了探究 Kimi K3 与其追赶的专业工具相比处于什么水平，我给 Kimi K3 和 Fable 5 分配了三个相同的实际编程任务，并跟踪了每一个 token 的消耗。

***另请参阅：** [Kimi K3 登顶 Arena 编码排行榜——且采用开放权重](https://thenewstack.io/kimi-k3-open-weight-coding/)*

## 测试流程

我使用了 [fd](https://github.com/sharkdp/fd)，这是由 sharkdp 开发的 Rust 文件查找工具。fd 是 Unix find 命令的快速且更友好的替代品。我选择它是因为它是真实的 Rust 生产代码，且拥有深厚且有据可查的 Bug 历史记录可供参考。

由于 Kimi K3 尚未集成到 Cursor 等编程工具中，因此我使用原生 CLI 完成了这三项测试。我在 Moonshot 新的 Kimi Code 终端代理（版本 0.27.0）中运行 Kimi K3，并在 Claude Code（版本 2.1.212）中运行 Fable 5。我使用了完全相同的提示词（Prompts），因此模型和 CLI 是唯一的变量。我在每个模型上为每个测试创建了一个新文件夹（总共六个），以确保测试的独立性。

以下是我运行的三项测试（按顺序）：

* Bug 修复
* 多文件重构
* 功能构建

对于每一次测试，我都使用秒表记录了时间，并从 Cursor 的使用仪表板中记录了 token 消耗和成本。

*本周早些时候，我在同一个代码库中使用完全相同的提示词，针对这三个任务对 Grok 4.5 与 Claude Opus 4.8 进行了对比测试。由于所有四个模型都运行了相同的测试，你可以将这两篇文章视为 Fable 对阵 Kimi 对阵 Opus 对阵 Grok。*

### 设置 Kimi

设置非常简单。Kimi Code CLI 只需一条 curl 命令即可安装，账户设置也很简单。不过有一个小插曲：我的测试命令“say hello”挂起了几分钟，没有回复，也没有错误信息。原因后来表现为 429 错误。我新注册的 API 账户余额为零，在绑定银行卡并充值之前，Kimi 不会做出响应。令人惊喜的是，充值后，Moonshot 赠送了 5 美元的注册优惠券，因此新账户确实可以获得一些免费额度。

> Moonshot AI 的 Kimi K3 在三项实际编程任务中与 Anthropic 的 Fable 5 逐行对应，成本仅为后者的三分之一……但完成时间却是后者的四倍。

## 测试 1：Bug 修复

提示词：

*此代码库（fd 命令行工具）中存在一个 Bug。当你传递 --no-ignore-vcs 标志时，fd 也会停止遵循父目录中的忽略文件。它不应该这样做。找到根本原因并修复它，使 --no-ignore-vcs 不再禁用父目录的忽略文件。不要更改任何其他行为。*

我在该 Bug 的真实 2021 年修复方案（问题 #907）之前的提交处检出了 fd，并清除了 git 历史记录，这样两个模型都无法查阅维护者的答案。两个模型都找到了根本原因，并从 src/main.rs 中删除了相同的一行代码。Diff 完全一致。四个模型，四个完全相同的 Diff。两者的所有 70 个测试均通过。

差异在于数据。Fable 在约 347K token 下耗时 1 分 4 秒完成，成本为 0.85 美元。Kimi 在 238K token 下耗时 3 分 7 秒，成本为 6 美分。K3 的 Bug 测试是我在这次测试中见过的最低成本但最高时间的案例。

## 测试 2：重构

提示词：

*在 src/main.rs 中，construct_config 函数很大，大部分工作都是内联完成的。对其进行重构以提高可读性和结构：将其与所需的任何辅助逻辑一起移动到新的专用模块中（例如 src/config_builder.rs），并将工作拆分为更小、更集中的函数。不要更改任何行为。所有现有测试必须仍然通过，且 CLI 的行为必须完全相同。*

两个模型做了同样的事情：一个新的 `config_builder.rs` 模块，大函数被拆分为集中的辅助函数，所有 264 个测试均为绿色，Diff 大小相差无几。

结果相同，但工程过程不同。Kimi 是一位更彻底的工程师。在触碰任何东西之前，它对旧二进制文件进行了快照，然后在约 40 个 CLI 场景中对比了旧二进制文件和新二进制文件，以证明行为没有改变。

但这以牺牲效率为代价。它在 928K token 下耗时 14 分 50 秒，是我过去几个月运行的所有测试中最慢的一次。Fable 在约 639K token 下用 3 分 11 秒完成了同样的重构。Kimi 成本为 0.70 美元，Fable 为 2.32 美元。

这与上一个测试讲述了类似的故事。结果相同，成本更低，耗时更长。

## 测试 3：功能构建

提示词：

*为 fd（命令行文件查找器）添加一个新的 --count 标志。当传递 --count 时，fd 不应打印匹配的路径。相反，它打印一行：匹配条目的总数，并遵循所有常规过滤器（例如，“fd --count --extension rs”打印匹配了多少个 .rs 文件）。将该标志添加到 CLI，通过搜索和输出路径进行连接，并确保所有现有行为和测试仍然通过。*

两者都交付了一个正确且可用的 --count 标志。Kimi 修改了六个文件并更新了帮助手册。Fable 修改了七个文件，并且更进一步，还更新了 `zsh` shell 补全功能。Fable 在编写自己的测试时两次计算错误，猜测是 13，然后是 10，最后定格在 11，但它在没有我帮助的情况下两次都自行发现并修复了错误。

Kimi 交付 --count 标志的时间几乎是 Fable 的 5 倍。它们的时间、token 和成本分别为 10 分 21 秒、210 万 token、1.38 美元，以及 2 分 34 秒、约 146 万 token、2.81 美元。

## 测试结果

| 指标 | Kimi K3 | Fable 5 |
| --- | --- | --- |
| Bug 修复 | 修复相同，通过 70/70 | 修复相同，通过 70/70 |
| 重构 | 通过 264/264 | 通过 264/264 |
| 功能构建 | 正确，修改 6 个文件 | 正确，修改 7 个文件 |
| 总时间 | ~28 分 18 秒 | ~6 分 49 秒 |
| 总 token | ~3.3M | ~2.4M |
| 总成本 | $2.13 | $5.98 |

在三项任务中，Kimi 的成本为 2.13 美元，而 Fable 为 5.98 美元，几乎正好是 Moonshot 定价承诺的三分之一。但折扣仅来自单价。Kimi 使用的 token 比 Fable 多，为 330 万对 240 万。此外还有时间成本。Fable 在 7 分钟内完成了所有三项工作，而 Kimi 需要超过 28 分钟。我每周都会在模型上运行不同的测试，这是我见过的最长时间。

那么，我对 Kimi 的真实看法是什么？结果很扎实，但这是一个拥挤的市场，且有既定的领跑者。我们仍处于 AI 免费增值时代，因此更低的成本很重要，但速度同样重要。Kimi 需要匹配其他模型的能力才能竞争，这意味着在某些情况下需要将速度提升 5 倍。

还需要记住，我们处于“完成胜于完美”的时代。这是 K3 的第一个公开迭代版本，它将会改变。

在目前的格局中，我没看到它的位置，但我很期待看到它在未来几个月内的改进。它能否通过提速赢得一席之地？我期待 Moonshot 的下一批更新，以便我们再次进行基准测试并找到答案。
