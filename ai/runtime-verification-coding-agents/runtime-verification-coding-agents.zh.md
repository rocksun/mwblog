*行业已经认识到，大规模交付由代理编写的代码需要运行时验证，并且正迅速朝这个方向发展。然而，对于大多数团队而言，这种验证的范围仍然有限。*

在代理式开发中，一种共识正在形成：要大规模交付[来自代理的代码](https://thenewstack.io/ai-agents-software-engineering/)，必须使它们能够在运行时验证自己的代码。静态验证（即阅读差异并针对模拟运行单元测试）是不够的。

瓶颈在于如何得知代理生成的代码是否真正有效，而这个答案只有在代码运行后才能揭晓。因此，行业正在将验证从人工审查队列中移出，转入代理自身的循环中，让代理在人类看到变更之前就有机会运行并检查自己的工作。

> “行业已经认识到，大规模交付由代理编写的代码需要运行时验证，并且正迅速朝这个方向发展。然而，对于大多数团队而言，这种验证的范围仍然有限。”

Greptile、Cursor、Codex 和 Devin 现在都倾向于运行代理编写的代码，而不仅仅是阅读它。

这是正确的方向。一旦[代理创建拉取请求（PR）](https://thenewstack.io/ai-generated-code-crisis/)的速度超过了人类的审查速度，审查就成了瓶颈。Stripe 的内部代理[每周交付超过 1000 个经过审查的 PR](https://stripe.dev/blog/minions-stripes-one-shot-end-to-end-coding-agents)。一个能够运行自己代码、读取故障并修复后再提交的代理，是防止该瓶颈完全压在高级工程师身上的关键。

对于云原生团队来说，这并非“可有可无”的功能。而且这也是大多数公司采用的方法所触及的上限。常见的模式是给每个代理提供一个沙箱环境，即一个可以复现开发者本地运行效果的快照。对于独立的应用程序，这已经足够了。

但对于在分布式系统中工作的代理，它只能针对模拟进行验证，从而错过了只有当变更与系统其余部分交互时才会出现的行为。

这些行为、集成以及其他系统级测试正是高成本 Bug 的藏身之处。趋势是正确的，问题在于你的代理所验证的运行时环境究竟是真实系统，还是它的替代品。

## 行业正在向运行时收敛

最明确的信号是，代理工具已经不再仅仅是读取差异，而是开始执行代码了。静态分析有其上限：它只能推理代码说了什么，而不能推理代码做了什么。需要真实请求触发的竞态条件，或者仅在页面渲染后才出现的回归错误——这些在文本中都无法体现。

这种趋势随处可见。Greptile [最近宣布推出 TREX](https://www.greptile.com/blog/trex-code-execution)，这是一项新功能，可以在一次性的沙箱环境中执行每次变更，并返回日志、跟踪记录和发生情况的截图。[Cursor 的云端代理](https://cursor.com/docs/cloud-agent)将存储库克隆到它们自己的虚拟机中进行构建和测试。[OpenAI 的 Codex Cloud](https://openai.com/index/introducing-codex/) 也做了同样的事情。[Devin](https://devin.ai/) 在一个具有自己的 Shell 和测试运行器的完整环境中运行。产品不同，但目标一致：给代理一个运行其工作并在移交前验证它的地方。

运行时验证正是大多数代理式技术栈所缺失的一层，将其推入代理自身的循环中，是保持交付规模增长时审查速度依然快捷的方法。悬而未决的问题是，要将变更真正称为“已验证”，需要达到什么程度的运行时保真度。

## 云原生团队需要进一步的突破

所有这些工具采用的方法都是给代理提供开发者在桌面上大致拥有的东西：它自己的服务、可以在本地启动的依赖项，以及替代所有其他内容的模拟。这是一种真实且有用的能力，也是正确的方向。

但对于由许多服务构建的云原生应用程序而言，这覆盖的范围不到一半。它错过的其余部分才是最重要的部分：变更相对于系统其余部分的行为方式。模拟只能返回预先预期的答案。它可以确认一个假设，但永远无法捕捉错误的假设。而在这些系统中，高成本的 Bug 正是发生在两个服务交汇处破坏假设的情况：

* 服务之间的契约漂移
* 边界处的序列化不匹配
* 在两个跳数之外行为异常的重试或超时策略

这些在单元测试或针对模拟的测试中都不会显示。它们会在集成测试、端到端测试以及其他系统级检查中浮现，而这些检查只有在变更针对其周围的真实服务、真实数据和真实流量运行时才有意义。

对于团队最关心的非功能性行为也是如此：性能和负载回归、资源争用以及运行时安全问题，只有当变更在真实的系统中运行时才会显现。这是沙箱环境无法触及的层面。

直观的解决方法——将整个系统克隆到每个代理的环境中——是行不通的。在代理生成代码的频率和速度下，为每个代理的每次迭代新鲜地启动数十个带有数据和配置的有状态服务是不切实际的。而且，副本始终只是一个快照，而不是实时系统。

![显示沙箱模型中真实依赖项替代品范围的图表。](https://cdn.thenewstack.io/media/2026/06/030fe0c7-image2-1024x601.png)

## 一种能够触及整个系统的架构

解决之道是停止为每个代理提供系统的副本，而是让每个代理针对一个共享的、具有隔离性的类生产系统进行验证。

这种模式很简单。一个共享集群运行所有真实的依赖项，即在生产环境中表现出的全套服务。为了验证变更，你只需将修改后的服务部署到该基准中，并利用请求级隔离来保持每个代理的流量处于其自己的路径上。代理的请求会到达其版本的服务，然后流经底层的真实服务。其他人的流量则保持在基准线上，不受影响。

这为代理提供了一个真实的运行时环境，使其变更能够与必须协同工作的实时服务、数据和策略一起运行。集成和系统行为变得具有**可观测性**，因为交互是真实的，而不是模拟的。

它也扩展到了代理的工作方式上。由于每次变更只是叠加在运行系统上的一个服务，而不是所有内容的私有副本，因此许多代理可以同时、廉价地进行验证，而不会产生冲突。环境轻量且生命周期短，并且运行在团队已经操作的基础设施上。这就是我们在 [Signadot](https://www.signadot.com/) 构建的模型。目前，这意味着 Kubernetes，尽管随着工具的成熟，这一理念也将延续下去。

![触及整个系统的架构示意图。](https://cdn.thenewstack.io/media/2026/06/3b921db6-image1-1024x677.png)

它契合了代理工具已经在使用的循环。代理在拉取请求之前，进行编写、运行、读取故障并重试。改变的是通过运行的标准。不是针对代理为了匹配自身假设而编写的模拟呈现绿色，而是针对这些假设所描述的真实系统呈现正确。

## 趋势是正确的，覆盖范围才是问题所在

运行时验证正在成为软件交付的一流组成部分，这是正确的演进。最先进的团队已经将代理的工作视为编写、证明、调试、循环，而不仅仅是编写。

> “不是针对代理为了匹配自身假设而编写的模拟呈现绿色，而是针对这些假设所描述的真实系统呈现正确。”

对于任何运行[云原生系统](https://thenewstack.io/introduction-to-cloud-native-computing/)的人来说，问题在于这种证明能达到多远。隔离的按代理环境可以在其自身层面上验证变更，对于许多变更来说，这已经足够了。

在必须与周围一切协同工作的情况下证明变更正确，是一个不同的问题。它需要针对持有其余系统的运行时进行集成和系统级验证，而不是针对它的替代品。

那些能从后台代理中获得最大收益的团队，将是那些验证循环能够触及整个系统，而不仅仅是其面前服务的团队。
