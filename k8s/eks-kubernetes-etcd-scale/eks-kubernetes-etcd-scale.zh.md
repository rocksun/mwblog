# 在大规模环境下运行 Kubernetes：Amazon EKS 的运维实战与思考

Amazon EKS 在全球 30 多个 AWS 区域运行着数十万个 Kubernetes 集群。在这种规模下运行的经验，塑造了我们构建服务的方式。我们认为，对于任何在大规模环境下运行 Kubernetes 的人来说，以下经验都非常有用：大多数可用性问题并非源于组件故障，而是源于组件对问题的错误响应，反而导致情况恶化。例如：缓存过期并返回错误数据，或运行状况检查重启了维持集群存活的关键进程。

一个有弹性的控制平面与脆弱的控制平面之间的区别，不在于故障数量，而在于故障是保持为故障本身，还是演变成服务中断。本文讲述了我们在规模不断扩大的情况下，如何让 EKS 管理的 Kubernetes 控制平面始终保持在这一界限的安全一侧：包括我们所做的基础性变革及其原因，以及我们在舰队规模下运维的经验——即如何构建能够容忍故障而非传播故障的系统。这些正是我们最严苛的客户能够放心地在 EKS 上运行其关键业务负载的原因。

## AI 和分析工作负载如何重新定义“规模”

Kubernetes 最初是为一种特定的工作节奏而设计的。Pod 以可预测的速率启动和销毁，控制器有几秒钟或几分钟的时间进行调解（Reconcile）。系统的设计反映了这种节奏：强大的数据一致性、有序的监控（Watch）以及将正确性放在首位的共识复制存储。它在其设计目标范围内表现得非常出色，至今依然如此。

> “一个有弹性的控制平面与脆弱的控制平面之间的区别，不在于故障数量，而在于故障是保持为故障本身，还是演变成服务中断。”

但[工作负载的演进速度](https://thenewstack.io/why-ai-workloads-are-fueling-a-move-back-to-postgres/)超出了任何人的预期。基础模型训练在几分钟内就会在数千个 GPU 节点上扩展训练作业。实时推理服务在几小时内从平稳基准扩展到数千个副本，然后迅速回落。Apache Spark 分析流水线从零骤增到数万个执行 Pod，处理完数据集后随即消失。

新兴的智能体（Agentic）AI 工作负载又增加了一个维度：自主智能体在几秒钟或更短的时间内启动、发散、执行任务并销毁。这些工作负载与传统微服务的一个显著区别在于：它们在压缩的时间窗口内产生海量的状态转换，且对延迟极其不耐受。这种状态变化的速率迫使我们重新发明了一些机制，以支持此前无法实现的规模，并尽可能地将成果回馈给上游社区。

## EKS 如何重构 Kubernetes 存储基础

每个 Kubernetes 集群都依赖 etcd 作为其事实来源。每个应用程序、每个服务终结点、每个调度决策都存储在那里。如果 etcd 丢失数据，集群就会遗忘一切。保护这种状态是托管 Kubernetes 服务最重要的工作。

运维单个集群的 etcd 并不难。但在数百万规模的集群舰队中运维它则是完全不同的挑战。硬件会故障，网络会抖动，磁盘会退化，因此必须有机制在无人干预的情况下处理这些事件。而 etcd 最需要的操作，例如替换故障成员或在区域性事件后恢复，恰恰是人在压力下最容易犯错、导致永久性数据丢失的环节。

从一开始，我们就构建了一个与每个 etcd 实例并行运行的 Operator Agent，负责自动化其整个生命周期。该 Agent 有两个职责。首先是备份与恢复：它获取时间点快照并将其持久化存储在集群之外。如果同时丢失太多实例且存活者无法达成多数共识，Agent 会自动检测该情况并从最新快照重建。其次是成员管理：当实例故障时，Agent 会移除已终止的成员并添加其替换者，其顺序旨在保护法定人数（Quorum）并防止脑裂。

最近的一项更根本的变革是，我们用一个专用的日志（Journal）系统取代了 etcd 的 Raft 共识机制，该日志提供独立于 etcd 的持久化、有序存储。在传统的 etcd 中，在写入提交之前，大多数成员必须就每次写入达成一致。如果三个成员中有两个不健康，集群就会变得不可用。

通过将持久性负担卸载到日志系统中，etcd 对等节点不再需要相互协商法定人数。一旦日志确认了持久化，写入即告提交，从而消除了那一类 etcd 法定人数丢失故障。由于日志处理了持久化，etcd 不再需要将写入内容 fsync 到本地磁盘，因此其数据存储已迁移到内存文件系统中。系统从“磁盘限制型”转变为“计算限制型”，存储延迟从关键路径中完全移除。欲深入了解此架构，请阅读“[幕后揭秘：Amazon EKS 超大规模集群](https://aws.amazon.com/blogs/containers/under-the-hood-amazon-eks-ultra-scale-clusters/)”。

> “系统从‘磁盘限制型’转变为‘计算限制型’，存储延迟从关键路径中完全移除。”

对于超大规模集群，我们更进一步，将 etcd 划分为资源特定的分片（Shards）。每个分区独立运行，拥有自己的存储预算和吞吐能力。其核心价值在于故障隔离。在单体部署中，如果事件（Events）键空间因行为不当的控制器创建对象速度快于垃圾回收移除对象的速度而超出配额，它会阻塞所有写入，包括节点租约。

突然间，健康的节点会被显示为不健康，因为它们的租约续订请求被拒绝了。通过分区 etcd，即使事件分区达到配额，租约分区仍能正常工作，节点保持健康，调度器持续运行。

## 取代 etcd 共识机制带来的突破

取消对法定人数的要求，使我们能够实现长期以来的目标：在与 API 服务器相同的主机上运行 etcd。在传统布局中，每次读写都要跨越不同机器之间的网络。单次往返很快，但如果是每秒数千次，延迟累积效应就很明显。通过并置（Collocation），API 服务器可以通过回环接口与本地 etcd 通信，Pod 调度和控制器调解的速度明显加快。对于那些以作业控制器队列深度为瓶颈的工作负载，每次 API 调用节省的毫秒数意味着在队列增长之前，每秒可以处理更多的作业。

![展示 EKS Kubernetes 架构演进的图表](https://cdn.thenewstack.io/media/2026/06/0ebb7b12-image2-1024x779.png)

这就是 Operator Agent 的价值所在。当 etcd 与 API 服务器并置在同一主机上时，每当控制平面主机被替换（这是常态），etcd 成员就会随之变动。这只有在成员管理完全安全且自动化的前提下才能奏效，而这正是 Agent 一直在做的事情。我们不必从头构建并置功能；它是建立在从第一天起就一直在安全管理 etcd 成员资格的基础设施之上的。

并置也教给我们一个值得分享的教训：便捷路径必须有失效转移机制。本地 etcd 是快速路径，但如果它受损，API 服务器会切换到另一个正在为同一日志的其他 API 服务器提供服务的 etcd 成员。在为常规情况进行优化时，也要为优化失效的情况进行刻意设计。

## 修复整个技术栈中的瓶颈

在极大规模下，你必须解决[整个 Kubernetes 技术栈中的瓶颈](https://thenewstack.io/ai-workloads-kubernetes-infrastructure-drift/)，其中大多数并不是传统意义上的 bug。它们往往是设计选择，在 Kubernetes 最初设定的规模下是正确的，只有在数值增大时才会崩溃。我们没有选择内部规避这些问题，而是将它们修复在上游，以便整个社区都能受益。

其中一个例子涉及 Watch Cache，这是将状态更改从 etcd 分发给所有监视更新的控制器的内存层。当控制器启动时，它会通过 WatchList 机制请求当前状态的全量快照，而现有实现会在构建响应期间持有共享读锁。

当对象达到数十万个时，该工作会运行很长时间，从而导致需要独占访问权的写入者挨饿，因此缓存的资源版本无法推进。一致性读取会看到陈旧的缓存并退回到 etcd，而响应构建过程则在锁下执行数十万次分配。我们将其确定为 Watch Cache 锁定模型的一个局限性，并正在与社区合作重构底层数据结构和接口，以消除争用。

同样的情况也出现在其他地方。在水平 Pod 自动扩缩器（HPA）中，单个保护缩放状态的互斥锁在 HPA 数量较高时变成了序列化点，此时 worker 几乎所有时间都在阻塞，而无法进行有效工作。重新设计的数据存储（PR #[139142](https://github.com/kubernetes/kubernetes/pull/139142)）恢复了并行性，并将调解吞吐量提高了几个数量级。在调度器中，我们发现了一个瓶颈（issue #[138426](https://github.com/kubernetes/kubernetes/issues/138426)）：每个调度周期都会通过扫描集群中的每个节点来重新构建正在使用的持久卷集，即使对于根本不使用存储的 Pod 也是如此。修复方案是仅为实际需要它的 Pod 计算该信息，从而在高规模下恢复了吞吐量。

每一个案例都始于真实的生产工作负载遭遇性能瓶颈，我们正在上游进行修复，以便这些改进能够惠及每一位 Kubernetes 用户。

## 从工程化到保障：EKS 预置控制平面

上述工程实践使 EKS 控制平面更具弹性和高性能。但客户面临另一个问题：他们可以观察到控制平面目前运行良好，却无法像预留计算资源或 GPU 容量那样预留其控制平面的容量。

一个规划千节点训练任务的团队可以提前几周锁定实例，却无法为协调这些实例的编排层预留同等规模的容量。[EKS 预置控制平面（EKS Provisioned Control Plane）](https://docs.aws.amazon.com/eks/latest/userguide/eks-provisioned-control-plane.html)填补了这一空白。它将控制平面的性能指标作为你可以明确调整的维度公开，并由你对其他基础设施所期望的那种承诺作为支持。

你可以选择一个对应具体、可衡量能力的扩展层级：API 请求并发性、Pod 调度速率和集群数据库大小。层级范围从 XL 到 8XL。在最高端，Kubernetes 1.34 上的 8XL 提供 16,000 个并发 API 请求席位、每秒 400 个 Pod 的调度速率和 16 GB 的集群数据库存储，所有这些都由以一分钟为间隔测量的 99.99% 可用性 SLA 提供支持。

层级不是静态的。你可以在 GPU 训练任务或大型促销活动前调高层级，在静默期调回，或者随着平台成熟永久增长。配置可以通过控制台、CLI、eksctl、CloudFormation 或 Terraform 在任何集群上进行，无需重建，也不会产生停机时间。

对于 AI 工作负载，编排容量与 GPU 容量同步规划，在计算资源上线时即可使用。对于每分钟提交数百个作业的分析平台，控制平面在突发流量到达之前就已准备就绪。对于需要在预发、生产和灾难恢复环境中保持一致性的组织，相同的层级可确保各地具有一致的性能特征。

## 将相同的基础带到边缘

![AWS Outposts 上 Amazon EKS 的架构图](https://cdn.thenewstack.io/media/2026/06/b641c1d2-image1-1024x753.png)

由于数据主权要求、延迟限制或与区域的连接不稳定，某些工作负载无法迁移到云端。在这些断开连接的环境中运行 Kubernetes 会带来独特的挑战：etcd 必须在只有几台机器的硬件上保持持久性，集群必须在无法连接到云端的情况下实现自愈，可观测性必须在持续数天的网络分区中存活。

通过实例存储 Outposts 上 EKS 本地集群的更新架构，我们将边缘集群纳入了与云端 EKS 集群相同的管理平面和软件栈中。

控制平面驻留在 Outpost 上的 EKS 管理账户中，而不是客户账户中，因此客户无需自行管理控制平面实例、etcd 备份或日志 Agent，也无法意外破坏维持集群存活的关键组件。相同的机器镜像、容器镜像和 Operator Agent 在两地运行，并根据配置选择边缘特定的行为。

因为采用相同的技术栈，新的 Kubernetes 和 EKS 平台版本与云端发布保持同步，并且 EKS 插件、Pod Identity 和访问条目等功能的工作方式与区域内完全一致。

最困难的部分是在可能断开与云端连接长达数天的几台机器硬件上保持 etcd 的健康。我们通过扩展相同的 Agent 解决了这个问题。它持续保持一份数据副本，并在机器发生故障的瞬间自动提升它，因此集群可以在无人干预且无需连接云端的情况下实现自愈。

[可观测性](https://thenewstack.io/taking-your-observability-strategy-to-the-next-level/)在断开连接时也能存活：指标 Agent 继续收集并写入本地磁盘，在空间不足时优先丢弃非关键数据，确保最重要的信号保留到最后。当连接恢复时，缓冲的数据会连同其原始时间戳一并回传。

这一切之所以能奏效，是因为系统从设计之初就是为了在无人登录的情况下运行。正是这种设计，使得在整个舰队中安全部署变更成为可能。

## 在舰队规模下安全运维

上述每一项变更都部署到了数十万个运行中的集群舰队中。日志迁移和并置要求对每个集群进行单独转换。每次迁移都遵循严格的顺序：验证先决条件、创建时间点快照、执行切换、验证后置条件。如果任何步骤失败，系统会自动回滚。

发布按单元（Cell）、区域（Zone）、区域（Region）分步进行，通过自动化监控对比更新后和未更新集群之间的延迟、错误率和吞吐量。任何统计学上的显著偏差都会触发自动停止。

这一切之所以成为可能，是因为 EKS 的构建原则就是无需人类在单个集群级别进行干预。通过“零操作员访问（Zero Operator Access）”，该架构从技术上防止了 AWS 人员访问托管控制平面中的客户内容。一个设计为无需人类访问即可工作的系统必须从一开始就是可观测、可恢复且自动化的，而正是这种纪律性使得在极大规模下进行运维成为可能。

三条运维经验塑造了我们对待这项工作的方式。

第一，健康的领导者（Leader）并不等同于一个在工作的领导者。控制平面的控制器以主动-被动配置运行，早期我们曾错误地将不健康的备用节点视为集群运维停滞的信号。实际上并非如此；真正重要的是是否存在领导者。但更深刻的经验是：领导者可以在通过所有健康检查并持续续订租约的同时，默默地停止处理任务。捕获此异常的信号是观察控制器的任务队列深度。如果队列在领导者看起来健康时依然堆积，系统就已经以活跃探测（Liveness Probe）无法察觉的方式落后了。

> “领导者可以在通过所有健康检查并持续续订租约的同时，默默地停止处理任务。捕获此异常的信号是观察控制器的任务队列深度。”

第二，维护的顺序与维护本身同等重要。etcd 的碎片整理（Defragmentation）是阻塞性的，且暂停时间随着数据库大小的增加而增长。当它发生在领导者身上时，所有写入都会停滞。我们教会了 Agent 在碎片整理前将领导权转移到健康的节点，这样破坏性的工作总是发生在追随者（Follower）身上，而写入操作可以持续进行。

第三，活跃（Liveness）不等于就绪（Readiness）。一个进程可能处于活跃状态，但在预热缓存时并不就绪，如果仅根据活跃度进行路由，就会将请求发送到无法处理它们的实例上。同样，在优雅停机期间就绪状态的抖动永远不应触发重启。我们将这两个信号严格分开：一个决定恢复，另一个决定路由。

所有这些工作在外部都是不可见的，而这正是重点所在。最大的集群教会了我们经验，使每个集群都变得更快。最冒险的迁移产生了保护每次升级的安全机制。我们为 Kubernetes 处理极限负载所贡献的上游修复，也流回到了项目的每一位用户手中。

> “所有这些工作在外部都是不可见的，而这正是重点所在。”

当你在 EKS 上部署应用，Pod 在几秒钟内启动，即使是在流量突增时，即使是在幕后出现问题时，这种速度也不是偶然的。这是多年来在小问题能迅速变成大问题的规模下进行运维，并对系统进行工程化改造以在问题爆发前遏制它们的积累结果。
