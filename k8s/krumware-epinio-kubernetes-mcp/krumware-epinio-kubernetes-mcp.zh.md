尽管过去十年在平台工程领域投入了大量资源并取得了显著技术进步，但大多数开发者仍然面临着与过去相同的问题：无法在没有阻碍的情况下交付产品。

企业仍在构建过程的不同阶段，跨不同团队使用多种工具。对于开发者而言，处理可观测性、合规性、部署和访问管理需要频繁切换工具，这在部署周期中造成了不必要的障碍和延迟。

在目前由 AI 技术和“全包式平台”解决方案主导的行业中，Krumware 采取了不同的策略。

[**Colin Griffin**](https://www.linkedin.com/in/colin-e-griffin/) 是 [**Krumware**](https://www.krum.io/) 的创始人兼 CEO，他是一位拥有关键基础设施和实时系统背景的软件工程师。作为 [CNCF 平台工程技术社区组](https://community.cncf.io/platform-engineering-technical-community-group/) 的联合组织者，他还共同撰写了 [平台工程成熟度模型](https://tag-app-delivery.cncf.io/whitepapers/platform-eng-maturity-model/) 和云原生成熟度模型，对开发者每天面临的真正痛点有着第一手的经验。

> “开发者每天必须打破的墙壁并没有真正改变。一切都分散在多个地方，没有简单的方法将它们整合在一起。”——Krumware CEO Colin Griffin

他的经验指导着 Krumware 构建的一切，包括 [Epinio](https://epinio.io/)（用于 Kubernetes 的开源应用程序开发引擎）以及 Krumware 的最新功能：[Epinio MCP](https://docs.epinio.io/getting-started/install-mcp) 服务器。该服务器促进了标准化，并实现了与企业现有防护栏的兼容。Epinio 并不是在现有层之上再加一层，而是底层的支撑系统。

## **改变对话：AI 就绪即平台工程就绪**

对于 Krumware 服务的大多数企业而言，阻力并非源于缺乏合适的工具。通常情况下，多个团队都在使用 Kubernetes，但整个企业缺乏能够充分发挥其潜力的平台。基础设施团队可能了解他们的集群，而开发者了解他们的代码。然而，两者之间存在鸿沟，组织在运营时缺乏监督和规范 Kubernetes 使用的统一方法。

团队领导试图通过添加更多工具来填补这一空白，但这不是仅靠更多的 AI 软件就能解决的问题。投入数十万美元购买 Backstage 实例的团队，往往在根本不了解开发者如何使用 Kubernetes 的情况下就进行了投资。事实上，在没有正确基础的情况下应用更先进的技术，只会加剧问题。AI 辅助开发带来的加速可能会拉大已做好平台准备的团队与未做好准备的团队之间的差距。

然而，AI 的“热潮”也可能成为一种财务和组织杠杆，为挣扎中的企业提供真正的解决方案。在 [平台工程成熟度模型](https://tag-app-delivery.cncf.io/whitepapers/platform-eng-maturity-model/) 中，Colin 评论说，平台工程和 AI 正在“合二为一”。对于董事、副总裁和首席信息官来说，“AI 就绪”的概念并不陌生，他们意识到自己在市场中的定位、相对于竞争对手的地位以及在开发前沿的位置。通往 AI 就绪的路径包括安全、可观测性、网络、互操作工具和工作负载的自助服务，所有这些都汇入平台工程。

从核心上看，“AI 就绪”与平台工程就绪是同一个话题，即把平台工程视为后续所有事物（包括代理层）的赋能层。

## **“像部署一样开发”**

Epinio 的核心主张很简单：它不会改变组织使用 Kubernetes 的方式。Epinio 提供了一个高度可适配的界面，使工程师能够让不一定是 Kubernetes 专家的开发者也能使用 Kubernetes。

对用户体验的关注，使得 Krumware 的设计理念与该领域的大多数工具不同。

> “你不需要知道 Heroku 是如何运行其应用程序的底层逻辑。我们采用了同样的理念，并使其与基础设施团队刻意构建的独特事物相兼容。”——Krumware CEO Colin Griffin

在实践中，这意味着一条单一的“Epinio push”命令就可以从本地文件获取源代码，并使用 Cloud Buildpacks 自动检测语言和运行时，构建容器镜像，将其部署到集群中，并提供一个实时 URL。Epinio 原生地运行在 Kubernetes 集群内，包括通过 Rancher Desktop 和 minikube 等工具构建的本地开发环境。因此，构建过程继承了监管生产环境的相同安全规则、合规性约束和网络配置。

Epinio 通过消除测试-迭代周期中的瓶颈，极大地提高了开发速度，并真正实现了快速原型设计。其结果是更好的测试和更高效的部署周期：故障发生得更早，处于正确的环境中，并被正确的防护栏捕获，而不是作为生产环境的意外惊喜出现。

Epinio 不仅适用于传统开发；随着氛围编程的爆发式增长及其被各类员工的采用，Epinio 正越来越多地被那些希望实施“黄金路径”（Golden Paths）并在全公司强制执行标准化的组织所使用。

![该图对比了充满等待步骤和反馈循环的“传统”代码-构建-部署-运行工作流程与直接从代码到运行的精简版“Epinio 快车道”。](https://cdn.thenewstack.io/media/2026/07/4e67fa32-unnamed-1024x561.png)

Krumware 将此称为“像部署一样开发”的方法。无论 Kubernetes 在哪里运行，环境都是一致的，从而消除了以前在本地开发机器使用与远程集群不同的 Kubernetes 配置时遇到的障碍。以前仅在 Kubernetes 设置中可用的安全和合规控制，现在也可以在本地机器上使用。Epinio 不是一个 [内部开发者平台](https://thenewstack.io/7-core-elements-of-an-internal-developer-platform/ "Internal Developer Platform") (IDP)，但对于想要构建该功能的用户来说，它可以胜任。

一旦该环境建立起来，它就成为符合组织政策的中央模板，确保了整个企业的标准化。Krumware 将这些模板作为可定制的工件交付，并根据每个团队的设置进行调整，使工程师能够在一致、受控的框架内构建和部署。

当团队开始使用 Epinio 时，那种“顿悟”时刻来自于第一个应用程序在不到一分钟的时间内从零到上线。团队突然拥有了更快的周期，此时他们就可以开始添加其他功能并创建用于控制的目录。

## **轻松实现访问和可操作性**

对于具有 Cloud Foundry 背景的企业来说，这种体验会感到熟悉。Epinio 最初由 SUSE 开发，灵感来源于 Cloud Foundry 的简洁性，其设计旨在隐藏 Kubernetes 的复杂性，就像 Cloud Foundry 为遗留基础设施所做的那样。虽然 Cloud Foundry 以开发者简洁性和基于 buildpack 的部署在市场上铺平了道路，但 Krumware 通过 Epinio 在现代基础设施上延续了这一理念。

Krumware 于 2024 年从 [SUSE](https://www.suse.com/) 手中接过了 Epinio 的管理权，作为其官方开源维护者，并花了一年时间更新了一些组件，将该项目推向了新的能力水平。Epinio 的一个基本方面是它与其他平台的互操作性，首先是 [Rancher](https://www.krum.io/products/rancher-accelerate)，即企业级 Kubernetes 管理平台。Epinio 继续作为 [Rancher](https://www.krum.io/products/rancher-accelerate) 的扩展提供，使其可以直接在许多企业已经使用的基础设施工具中访问，未来的版本也将将其互操作性扩展到其他企业平台。

在过去的几个版本中，Epinio 引入了几项重大改进，包括一个新的权限系统和由 SeaweedFS 提供支持的升级版文件存储系统。最新版本 Epinio 1.14 延续了这一势头，重新设计了应用程序创建流程，并全面刷新了 UI，包括 Trailhand 的元素——这是 Krumware 即将推出的专注于 UI 互操作性的平台组件系统。

## **Kubernetes 的代理套件**

Krumware 的 [Epinio MCP](https://docs.epinio.io/getting-started/install-mcp) 服务器提供了与其他 MCP 产品不同的解决方案。MCP（模型上下文协议）是 AI 代理和 LLM 与外部工具和数据源交互的新兴标准。现有的 Kubernetes MCP 服务器已经存在，并赋予 LLM 对集群的原始访问权限，但缺乏上下文。该模型需要消耗大量的 token 来确定哪些数据源可用、可以使用哪些服务以及哪些类型的应用程序适合此环境。如果没有足够的输入，它将部署看起来最合适的任何东西。

Epinio MCP 服务器通过从 Epinio 已经管理的应用程序层视图提供结构化的、预定范围的上下文来解决这个问题。它展示了批准的 buildpack、模板、服务目录和命名空间约束，并采用了用户的凭据，以便代理在与开发者相同的访问边界内运行。Epinio MCP 为构建提供了防护栏，并为增强部署提供了套件。人类仍然是导演；代理在定义的、被批准的空间内运行。

按照设计，Epinio MCP 服务器是 LLM 不受限的。它适用于任何支持 MCP 标准的模型，无论是云托管还是本地部署。虽然 Epinio 无疑将改善开发者的迭代周期、加快入职速度并提高标准化和合规性，但该套件的更深层目的是关于开发者的自由。

“如果我们能帮助开发者理解应用程序的‘谁、什么、哪里、为什么以及如何’，我们就越来越接近于让他们只思考自己的想法。这释放了认知负荷，让他们可以去创造、思考和设计新的解决方案，构建更好的结果。”——Krumware CEO Colin Griffin

有了 Epinio MCP，工程师可以将 Epinio 改造成他们需要的任何东西。为什么不利用该平台来构建它自己呢？应用程序可以意识到它们自己的部署环境，团队可以为特定的数据库构建自定义 MCP 服务器，将上下文数据源直接绑定到开发循环中。封闭的环境变成了一个创造的表面。

## **塑造行业**

在 1.14 版本之后，即将发布的版本将引入 Trailhand，这是一个开源的平台工程组件系统，具有框架无关的组件，允许 Epinio 的上下文被注入到其他工具中，而不仅仅是通过 Epinio 自己的界面来消费。一个新的基于包的 buildpack 生命周期也将被引入，为非 Paketo buildpacks 打开大门。

路线图反映了 Krumware 构建一切所基于的互操作性理念：每个 IT 生态系统都是不同的，平台团队需要一个从基层一直延伸到客户的价值交付管道。

Krumware 并没有要求团队彻底改变他们的工作方式。它提供的是在组织已经投资的 Kubernetes 环境中完成最佳工作的套件，并以此为基础，提供构建未来所需一切的表面。

对于 Krumware 而言，Epinio 正在成为平台互操作性的设计标准，它不仅仅是一个应用程序部署工具，而是连接层，使其他一切协同工作得更好。
