根据 [Cybersecurity and Infrastructure Security Agency (CISA)](https://www.cisa.gov/sites/default/files/2025-08/2025_CISA_SBOM_Minimum_Elements.pdf) 发布的 2025 年版 SBOM 指南：*“SBOM 应包含构成目标软件的所有组件信息，包括传递依赖项。没有最小深度限制。”* CISA 同时指出，配置文件和分支血统也必须包含在内。

没有最小深度限制。

一个加固后的镜像，只有在从上到下、每一个细节都通过“嗅探测试”后，才真正称得上是“加固”的。

“加固”容器镜像的承诺在于降低风险并提升安全性。

通过一点专业知识以及针对镜像内涵提出的一些基本问题，安全团队和平台团队可以对加固镜像进行“嗅探测试”，以检查其[透明度](https://www.docker.com/blog/100-transparency-and-five-pillars/)和准确性。对于任何重视安全的业务或组织而言，这些“嗅探测试”都应被视为极具价值的工具。

## SBOM：它们本应有的样子

为了实现供应链安全的承诺，加固镜像的 SBOM 必须提供[透明度](https://www.docker.com/blog/100-transparency-and-five-pillars/)，并包含镜像中每个组件的完整、机器可读的清单。

### SBOM 测试

SBOM 工作流通常包含以下主要组件：解析源代码、二进制文件或容器镜像；验证 SBOM 的完整性；将组件与数据库进行交叉比对；应用 VEX 建议以指出可被利用的漏洞；以及扫描 SBOM 以确保符合许可协议。

### 漏洞扫描与可利用性评估

漏洞可利用性交换 (VEX) 文档允许你记录漏洞在你的特定环境中是否真正可被利用。**持续监控**：新的 CVE 不断涌现，因此必须持续、自动地根据 SBOM 检查这些漏洞，而无需重新扫描工件本身。

### 合规性与来源

你的 SBOM 应列出每个组件的许可协议（MIT、Apache-2.0、GPL-3.0 等），以便法律团队验证你是否违反了 copyleft 要求或使用了不兼容的许可组合。现代 SBOM 可以包含加密签名，用于验证组件的真实性。

### 供应链风险管理与认证

当发生类似 Log4Shell 的供应链攻击时，结合适当 VEX 声明的 SBOM 让你可以立即回答“我们是否受到影响？”，而无需等待工程团队去 grep 代码库。当凌晨两点出现零日漏洞时，你需要在几分钟内知道哪些系统受到了影响。一个准确的 SBOM 可以将数天的调查变成一次简单快速的数据库查询。

**不完整和遗漏**：SBOM 有时无法列出镜像中存在的所有组件。这些遗漏可能是故意的，因为包含每一个依赖项或组件可能并不方便、成本高昂，或者会推迟加固镜像的发布。

**缺乏透明度**：完全透明的承诺与现实之间可能存在巨大差距。这里的问题在于，如果供应商无法解释每一个组件，就会损害安全所必需的信任基础。

**验证困难**：验证 SBOM 的准确性可能具有挑战性。当供应商仅提供“仅运行”或“仅构建”的 SBOM 而没有明确的文档时，开发和/或安全团队就很难知道他们是否对软件供应链有完整的了解。

**虚假的安全感**：SBOM 本身并不能保证安全。即使有一份冗长、详细的清单，如果清单不完整，漏洞扫描器也会因缺少组件而漏掉威胁。这会导致在审计甚至响应零日事件时产生虚假的安全感。

## 散发“异味”的加固镜像 SBOM：比你想象的更常见

有时，SBOM 无法通过嗅探测试，未能包含每一个组件和工件。这些组件被遗漏很可能是因为它们处理起来不方便，且会增加加固镜像提供商的安全要求。在某些情况下，额外的工作可能会推迟交付或增加成本。

这方面有哪些具体的例子？以下是三个接近现实发现的假设案例。

### “latest”标签陷阱

**问题**：latest 标签陷阱是指依赖浮动的容器标签（如 latest），这会导致虚假的安全感。例如，你可能会使用类似 FROM python:latest 或 FROM mariadb:latest 的标签。

**解决方案**：通过将所有容器镜像固定为不可变的 SHA256 摘要或特定的语义版本标签（例如 mysql:9-debian-dev）来避免“latest 标签”陷阱。

### 缺失包固定

**问题**：当开发者未能锁定包版本时，后续构建将自动从上游仓库拉取该包的最新可用版本。

**解决方案**：检查你的 SBOM 工具是否正确捕获了精确的包版本、哈希值和许可协议，或者是否默认使用模糊的、未版本化的依赖项，从而导致无法修复 CVE。

### 臃肿的基础镜像（“全包含”的异味）

**问题**：使用像 FROM ubuntu:latest 这样单体、臃肿的基础镜像，而不是像 Alpine 或 distroless 这样的小型环境。

**解决方案**：为避免包固定缺失的问题，请通过摘要固定基础镜像，锁定系统和语言包，自动化依赖项更新，并实施全面的镜像扫描。

需要注意的另一个问题是，AI 和 ML 镜像通常包含大量的依赖树，包括 CUDA、PyTorch 和 TensorFlow。由于这些镜像可能引入如此多的框架、库和其他依赖项，SBOM 必须完整。在这些镜像中忽略哪怕一个软件，也可能导致灾难。

### 只有数据库镜像……却没有 OpenSSL

**问题**：使用的数据库镜像不包含任何加密功能。

**解决方案**：确保你使用的数据库镜像包含某种形式的加密，以保护静态和传输中的敏感数据，例如 Chainguard FIPS 镜像。

### 没有 npm 包的 Node.js 镜像

**问题**：你的 Node 服务器的 SBOM 没有列出任何 npm 包。一个都没有。

**解决方案**：如果构建链中使用了 npm 包，或者它们存在于配套的构建镜像中，加固镜像供应商应该说明这一点并解释其流程。

### 只有一个孤立包的企业日志

**问题**：SBOM 仅显示了一个单一的日志库。

**解决方案**：运行深度扫描和多语言检查，以识别镜像中包含的每一个包。

## 快速 SBOM 嗅探测试！

幸运的是，每个平台和安全团队都可以通过视觉扫描对 SBOM 进行快速嗅探测试。这正如其名，且可以非常快地完成。第一步是询问你的供应商 SBOM 是仅运行、仅构建还是二者结合，以及它是如何生成的。

供应商的回答可能会引导你，但你不应完全信任它。你应该立即进行实际的嗅探测试。这个过程可以在五分钟内完成，并将提醒团队注意可能导致灾难性故障的潜在 SBOM 遗漏。

对于基础的嗅探测试，这里有一个有效的工作流示例：

**第一步：** 获取 SBOM。

**第二步：** 计算列出的组件数量。

**第三步：** 寻找包含“operating-system”、“alpine”、“ubuntu”或“debian”的条目。

**第四步：** 如果你知道镜像应该使用 Flask、Express 或 Spring Boot，请确保它在清单中。如果是，它包含哪个版本？

**第五步：** 如果 SBOM 中的包比你合理预期的要少，这可能表明某些包被遗漏了或将在稍后添加。

**第六步：** 搜索你确定必须存在的组件，例如 openssl、ca-certificates 和 libc。如果它们缺失，说明操作系统层未被捕获。

**第七步：** 使用容器镜像扫描工具（Docker Scout 是一个选择，也有 Syft 或 Trivy 等免费工具）。

## 向加固镜像供应商提出要求

加固镜像供应商出售给你的正是安全和信任。你为此支付了费用，因此应该要求获得一些基本的回报，例如基本的可信度、明确的范围、验证、透明度和响应能力。

## 开启嗅探体验

如果 SBOM “闻起来怪怪的”且无法通过简单的嗅探测试，请不要信任它。就是这样。如果供应商无法确切告诉你他们是如何构建 SBOM 的（涵盖每一个步骤），就应将他们的加固镜像视为可疑的。任何加固镜像最重要的组成部分就是对真实来源的信任，而 SBOM 必须是这一来源。
