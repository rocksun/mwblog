在周二举行的 Build 开发者大会上，微软为开发者推出了如此多的新功能，甚至让人有点目不暇接。

但其中最重要的一项其实非常简单直接：暗黑模式——而且它是默认开启的。

这是全新、专为开发者优化的 Windows 11 体验的一部分，它让操作系统变得更清静，并关闭了小组件、通知和产品内推荐，针对开发者的工作方式重新调整了 30 多个设置。其核心目的就是减少干扰（即便你*不是*开发者，关掉这些功能中的大部分也能让 Windows 11 变得更好用）。

微软产品管理合伙人总监 [Jatinder Mann](https://www.linkedin.com/in/jatinder-mann/) 在大会前接受《The New Stack》采访时表示，这一切的核心在于倾听开发者的心声。

他表示：“在谈论任何智能体（agentic）或 AI 之前，开发者最想要的只是一个干净、快速、无干扰的开发环境，让他们能够快速进入状态、保持专注流并更快地交付代码。”他说，开发者希望微软能让 Windows 11 变得“敏捷、平静、对资源敏感，并尊重我已有的肌肉记忆”。

## 微软显然在努力吸引开发者转投其阵营

这种肌肉记忆当然源于多年使用 Mac 和 Linux 系统的习惯，因为微软显然正在努力吸引开发者脱离这些操作系统。

> “我们知道开发者希望将环境打造成他们自己的专属空间。”—— Jatinder Mann，微软。

当我们问他这究竟是微软在防守还是在进攻，是在吸引新开发者，还是在挽回流失到其他操作系统的开发者时，Jatinder Mann 并没有上当。他说：“这真的是在倾听反馈——我们如何为客户提供最好的体验？”

## 开箱即用的功能

该配置打包了各种默认设置，而不是引入一个全新的产品。它预配置了 VS Code、GitHub Copilot、WSL 和 PowerShell 7，并预装了 PowerToys、[Oh My Posh](https://ohmyposh.dev/) 和 [Nerd Fonts](https://www.nerdfonts.com)。

文件扩展名默认可见，隐藏文件会显示，且 Git 版本控制直接融入了文件资源管理器。任务栏终于可以放置在屏幕的左侧、右侧或底部了（万岁！）。

Jatinder Mann 表示：“我们知道开发者希望将环境打造成他们自己的专属空间。”

开发者可以通过三种方式获得这种体验：在新的 OEM 开发者电脑上预装、作为 Windows 365 云电脑的开发者优化镜像，或者作为可在任何现有 Windows 11 电脑上运行的可下载配置脚本。它也被打包进了 PowerToys 中。这些设置按用户单独应用（类似于设置自己的个人资料），而不是应用到整台机器上。

## Mac 和 Linux 的印记

还有其他几项重大新功能，表明了微软如何努力劝说 Mac 和 Linux 开发者爱上 Windows。

微软正在添加 75 个原生运行在 PowerShell 中（而不仅仅是在 WSL 内部）的 Unix 核心实用程序。这些工具是基于 uutils 构建的，uutils 是 GNU Coreutils 在 Rust 中的跨平台重新实现。

Jatinder Mann 解释道：“如果你在 PowerShell 中输入 grep、ls、touch，它们现在可以直接运行了。再也不会有那种提醒你正处于不同操作系统中的刺眼突兀感。”

对于那些将大量时间花在[适用于 Linux 的 Windows 子系统](https://thenewstack.io/how-to-install-windows-subsystem-for-linux-on-windows-11/) (WSL) 上的开发者，现在还提供了一套全新的设置脚本，可将 starship、homebrew、zsh 等工具带到 Windows 中。

微软在 Build 2025 上开源了 WSL 本身，现在它也获得了一个容器功能：一个内置的命令行界面（CLI）和 API，用于在无需第三方工具的情况下在 Windows 上启动 Linux 容器。

Jatinder Mann 说：“无论你来自哪个平台，Windows 都应该让你感到熟悉，让你有宾至如归的感觉。”

## 保持专注流：智能终端

在 Windows 的全新开发者体验中，最有趣的或许是实验性的智能终端（Intelligent Terminal），它直接将一个编码智能体植入了 Shell 中。

他说：“你正在终端里，遇到了一个错误，你复制它，切换到包含你最喜欢的智能体的聊天窗口，粘贴，解释上下文，得到答案，然后再切换回来。这给人的体验割裂感很强。”

智能终端在现有 Windows Terminal 的基础上增加了一个智能体面板，可以跟踪 Shell 的实时状态。当命令失败时，它会提供上下文，并建议开发者可以运行的修复方案。开发者可以自行选择智能体，包括 Claude Code、OpenAI 的 Codex 或 Copilot（也可以关闭此功能）。

如果这听起来很熟悉，很可能是因为你一直在使用 Warp。虽然 Warp 本身可以免费使用，但它限制了用户的 AI 使用量，付费层级每月 20 美元起。微软则免费提供了其版本，尽管还算不上是最终完成版。Jatinder Mann 表示：“我们特意将其作为一个实验性分支进行发布。我们希望开发者能与我们共同设计它。”

## 另一个赌注：Windows 作为运行智能体的平台

如今，任何发布会如果不提及 AI 智能体都是不完整的。在 Build 大会上，微软正在扩展其现有工作，将 Windows 打造成一个供智能体运行的平台。

Jatinder Mann 表示：“我们希望确保 Windows 是构建和运行智能体的最佳场所，这样你就可以放心、安全、放心地运行它们。”

这里的主打功能是微软执行容器（Microsoft Execution Containers, MXC），这是一个由策略驱动的层，允许开发者声明智能体可以接触的内容（例如文件、网络、进程），并在运行时强制执行这些边界。Jatinder Mann 解释道，MXC 提供了一系列的隔离级别，从针对轻量级操作的进程级隔离，到完整的虚拟机，再到最顶端的、独立管理的云电脑。

Windows 还为每个智能体分配了一个本地 ID 或云端配置的 Microsoft Entra ID，并将其活动归因于该 ID，这样开发者就可以区分人类和智能体，并审计每个文件和网络调用。OpenClaw 可能是最需要安全限制的智能体工具，微软与该团队合作，使该智能体运行环境能够原生运行在 Windows 的 MXC 容器中。

对于构建 Windows 应用的开发者，“Windows 开发技能”为编码智能体提供了构建原生 Windows 应用（从 WinUI 3 到打包和身份识别）的结构化上下文。由于这些是标准的 skills.md 文件，它们可以与任何智能体配合使用，正如 Jatinder Mann 所指出的，这将使它们能够以更高 Token 效率的方式运行。

## Windows 上的“无限制智能”

当然，最便宜的 Token 是由本地模型生成的。这也是微软在 Windows 上关注的重点。Jatinder Mann 问道：“我们该如何帮助将每一台 Windows 电脑都变成你桌面上的 Token 工厂？”

为了让 Windows 成为能从本地 AI 功能中受益的应用的更好载体，微软正在将其 Windows AI API 从 Copilot+ PC 中的 NPU（这在以前看起来一直是个奇怪的限制）扩展到 CPU 和 GPU。它正在添加一个端侧语音识别 API，并将基于小语言模型的收件箱带到更多硬件上。然而，GPU 的具体要求目前尚不清楚。

此外还有两个新的端侧模型：Aion 1.0 Instruct 是当前 Windows SLM（小语言模型）体积更小、速度更快的后继者，将于 7 月在 [Hugging Face 上开源权重](https://thenewstack.io/how-hugging-face-positions-itself-in-the-open-llm-stack/)。

还有 Aion 1.0 Plan，这是一个拥有 140 亿参数的推理和工具调用模型，带有 32K 上下文窗口，将作为 Windows 的一部分一同出货。该模型旨在完全在设备上运行智能体工作流。目前，安装 Windows 11 需要 9 GB 的磁盘空间。一个 140 亿参数的模型可能也会占用同样多的空间（具体取决于量化程度）。鉴于很少有 Windows 用户会拥有显存足够大的 GPU，看到运行此模型的硬件要求将会非常有趣。

## 微软终于开始倾听了吗？

不过，对 Jatinder Mann 来说，这一切最终都归结为给用户他们想要的东西。他说：“这里的几乎每一项都是应客户要求而做出的。”

听到这个消息令人高兴，因为微软长期以来一直因反其道而行之而饱受批评。用户曾抱怨 Copilot AI 功能通过系统更新被强行推送，以及缺乏 UI 自定义选项等问题。

然而，倾听是一回事，根据你所听到的内容采取行动又是另一回事。微软现在似乎受够了冷落，终于开始对 Windows 11 做出实质性的改变了。
