2026年5月，OpenTelemetry (OTel) 正式从[云原生计算基金会 (CNCF)](https://www.cncf.io/projects/opentelemetry/)毕业。这一里程碑标志着云原生生态系统达到了生产就绪和成熟阶段，这得益于数百家公司和数千名开发者的共同努力，他们坚信可观测性的开放标准并共同构建了它。

## OpenTelemetry 曾已是标准，如今获得了官方认可

如果您在过去几年中一直在交付生产代码，那么 OpenTelemetry 几乎肯定已经触及了您的技术栈，无论是通过其 SDK 和收集器，还是通过追踪（traces）、指标（metrics）和日志（logs）。对于许多团队来说，OTel 已经悄然成为构建和运行现代应用程序的默认工具箱的一部分。它解决了一个真实存在的问题：行业需要一种通用的遥测数据语言，而 OpenTelemetry 成为了这种语言。

CNCF 的毕业反映了该生态系统的强大实力：多元化的贡献者群体、具有经过验证的生产就绪性的广泛厂商支持、全面的安全审计，以及一个由社区构建、服务于社区并保障未来可持续发展的治理模型。

## 为什么共享标准能改变一切

毕业使 OpenTelemetry 正式成为可观测性的通用协议和共享语言：

1. 标准协议允许不同的工具（无论是开源还是商业工具）协同工作。
2. 语义规范定义了遥测数据的命名和结构方式。当所有系统都使用相同的语言时，大规模的关联分析就成为了可能。
3. OpenTelemetry 将检测（instrumentation）与后端分析解耦，使团队能够将遥测数据导出到任何后端系统，并在不重写代码的情况下切换分析平台。
4. 标准化、高质量的遥测数据可实现自动化、异常检测和 AI 驱动的洞察。随着系统变得更加复杂和自主，一致的协议变得至关重要。

## 常见问题解答

### **CNCF 毕业后，OpenTelemetry 稳定吗？**

是的。毕业证实了核心规范、API 和数据模型已处于稳定状态，适合长期生产使用。团队可以充满信心在各种环境和用例中采用 OpenTelemetry。

### **OpenTelemetry 会将团队锁定在特定的厂商或后端吗？**

不会。OpenTelemetry 将检测与后端分析解耦。团队可以将遥测数据导出到任何后端，并在无需重写检测代码的情况下切换平台。

[无论您身在何处，Dynatrace 都能为您提供支持](https://www.dynatrace.com/info/whitepapers/streamline-dev-workflows-with-dynatrace-otel/)。您可以原生摄取纯粹的 OpenTelemetry 数据，无需任何专有代理。在此基础上，您可以在几秒钟内查询数十亿个 Span，在数 PB 的数据中自动关联指标、日志和追踪，并获得[完整的可观测性上下文。](https://www.dynatrace.com/news/blog/data-in-context-how-dynatrace-solves-the-opentelemetry-analytics-challenge/)
